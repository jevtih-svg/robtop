# RobTop Security Hardening — Spec and Implementation Plan

Date: 2026-06-09
Branch: `security-hardening` (never pushed to `main` by the agent; Jeff reviews and merges to deploy)
Source audit: `АУДИТ-2026-06-09-архитектура-и-безопасность.md`

## Goals (approved by Jeff)

1. **No unauthorized use, ever.** Hard-require a logged-in session for all data/API access. Remove the anonymous "everything belongs to Артём (id 1)" fallback entirely.
2. **Server-authoritative points.** The server becomes the only writer of the points ledger; a child can no longer mint points.
3. **Fix every finding** in the audit (SEC-1..14) plus the security-relevant headers.
4. **Rewrite all affected UI copy** in en/ru/lv, chiefly the `1234` default-password references, to match the new behavior.

Non-goals for this pass: offline service worker (BP-1), image-resize dedup (BP-2), schema foreign keys (ARCH-3) unless trivial, and other pure-quality items. These are tracked for later.

## Decisions

- Auth: **hard-require login everywhere** (no emergency toggle). Prod `config.php` will set `single_user => false`; the fallback is deleted from code so the value is moot, but the comment/docs are corrected.
- Points: **full server-authoritative**. Ledger stays in `module_data` (`module='bank'`, `collection='points'`) to avoid a data migration; a new server-only writer is the sole insert path. Amounts come from a server tariff, never from the client.
- Default password: **removed**. A random per-account temp password is generated server-side, returned to the parent/admin UI, and injected into invite emails. The forced-change-on-first-login flow stays.

## Part A — Hard-require login (SEC-2 identity, SEC-3)

### A1. `app/api/_bootstrap.php`
- `rt_user_id()`: remove the `single_user`/`RT_DEFAULT_USER_ID` fallback. Resolve session → uid, else `0`. Keep it returning `0` for anonymous so endpoints can detect "no user."
- Delete `RT_DEFAULT_USER_ID` define and the `single_user` branch.

### A2. Require login on data/action endpoints
Add `$me = rt_require_login($db);` (which 401s anonymous and 403s disabled accounts) at the top of, replacing direct `rt_user_id()` reliance:
- `data.php`, `state.php`, `stats.php`, `balance.php`, `sync.php`, `action.php`, `upload.php`, `image.php`, `tasks.php`, `registry.php`, `modules.php`, `notify.php` (the `send` path), `push.php`, and the new `points.php`.
- `_migrate.php` / migration runner: ensure it is admin-only or CLI-only, never anonymous.
- `mail_test.php`: gate behind `rt_require_admin` or delete.

### A3. Intentionally public paths (kept, narrowly)
- `accounts.php` auth ops: `login`, `register_parent`, `reset` (request + confirm), `accept`, `invite_info`. These must stay reachable pre-login; they get rate limiting (Part C) instead.
- `share.php`: the **public read of a shared wishlist** via share token stays public (the "share by link" feature, `w.html`). Every other op in `share.php` requires login. Verify the public read cannot enumerate or leak beyond the shared list.
- `notify.php` `peek`: stays, authorized by the 256-bit switch token (account-switcher badge), not by anonymity. Confirm it only returns unread counts.

### A4. Boot-flow verification (do FIRST, before gating registry)
- Trace `app/core/shell.js` and `app/index.html`: confirm `registry.php`/`modules.php` are called only after the lock screen / successful login. If any pre-login screen needs a tile list, carve a minimal unauthenticated read; otherwise gate fully.
- `landing.html` is a static showcase for logged-out visitors; confirm it does not call gated endpoints. Clicking through must land on the login/lock screen.

### A5. Config + docs
- `config.example.php`: change `single_user` comment to reflect that login is now mandatory; mark the key deprecated/no-op.
- `README.md`: update the "Пока один пользователь (Артём, id 1)" note to "login required; identity from session."

## Part B — Strong session tokens (SEC-2)

### `app/api/_accounts.php`
- `rt_start_session`: replace `$tok = rt_token();` with `$tok = bin2hex(random_bytes(32));` (64 hex). `rt_is_token` already accepts `[A-Za-z0-9]{4,64}`, so both old and new validate during transition.
- Keep `rt_code(6/8)` only for human-dictated invite/reset codes.
- Provide a documented one-liner for Jeff to force re-login if desired: `DELETE FROM sessions;` (kills weak legacy tokens immediately). Default: let them expire.

## Part C — Rate limiting + no default password (SEC-4, SEC-11)

### C1. Throttle (`app/api/_throttle.php`, new)
- A small helper backed by a new `auth_attempts` table (migration): columns `id, scope (account|ip), key_hash, window_start, count, locked_until`.
- `rt_throttle_check($db, $scope, $key)` → throws 429 if locked. `rt_throttle_fail()` increments; `rt_throttle_ok()` resets.
- Apply in `accounts.php` to: `login` (per account + per IP), `accept`/`invite_info` (per IP + per token prefix), `reset` confirm (per IP + per account). Exponential backoff: 5 fails → 1 min lock, doubling, cap 1 hour.
- Migration: `app/api/migrations/030_auth_throttle.sql`.

### C2. Remove the `1234` default password (server)
- New `rt_temp_password()` in `_accounts.php`: 6 chars from the readable no-confusable alphabet (reuse `rt_code` alphabet). Random per account.
- Replace every `password_hash('1234', ...)` and `rt_set_password($db, $id, '1234', 1)` with a generated `$temp`, in: `accounts.php` (add_child, reset_child, accept-create) and `admin.php` (user_reset_pass, create child, create parent).
- Each response already returns `temp_password`; set it to the generated value (not the literal `1234`).
- Drop the `=== '1234'` special-case validations (`accounts.php:139,242`, `shell.js:820`, `family.html`); keep the generic weak-password check (min 4 chars; optionally "must differ from the temp password").

### C3. Inject temp password into invite emails
- `app/api/mail/admin_invite_parent.{en,ru,lv}.html`: replace the hardcoded `1234` with a `{{temp_password}}` placeholder.
- `_mail.php` / `mail/registry.php`: pass the generated temp password as a template variable when sending the parent invite. Verify the templating supports variable substitution; if not, add a minimal `{{key}}` replace.

## Part C2 — Server-authoritative points (SEC-1)

### CP1. The single writer (`app/api/_points.php`, new)
- `rt_points_write($db, $uid, $n, $kind, $reason, $actorId=null)`: the ONLY function that inserts a `bank/points` row. Clamps `|n|` to a max (e.g. 100000). Records `kind`, `reason`, `by` server-side. Mirrors to `events` via `rt_log`.

### CP2. Block client minting (`app/api/data.php`)
- After the role check, hard-reject any write op (`create/update/move/favorite/delete/restore`) where `module === 'bank' && collection === 'points'` → `403 points_readonly`. Reads (`list/get`) still allowed (so the client can show history/balance).

### CP3. Points endpoint (`app/api/points.php`, new, login-required)
- `summary`: returns authoritative `{balance, earned, count}` (reuse `balance.php` logic; keep `balance.php` or fold it in).
- `award {source}`: looks up a **server tariff** amount for `source` and writes it via `rt_points_write`. Client never sends the amount. Per-source daily caps to prevent farming (e.g. a game can award at most N times/day). Validates caller role.
- `spend {item}`: validates the price against the stored shop catalog item (`module_data` shop/items), debits that exact price in one DB transaction, and creates the order row. Rejects if balance insufficient or price mismatched.

### CP4. Server tariff
- Enumerate exact amounts from the modules before coding: read award sites in `guess`, `snake`, `find`, `bank`, and `shop` `module.js`. Build `RT_POINTS_TARIFF = ['guess_win'=>X, 'guess_lose'=>Y, 'snake_apple'=>..., 'find_win'=>..., ...]`. Honest play yields identical amounts; the client just cannot change them.

### CP5. Tasks award points server-side (`app/api/tasks.php`)
- On `approve` transition to `done` (once / origin=child) or recur-complete, call `rt_points_write` with the task's stored, clamped `points`. Remove the expectation that the client awards them.

### CP6. Client SDK (`app/core/sdk.js`) and modules
- `sdk.points.add(...)` → replaced by `sdk.points.award(source)` calling `points.php`. Update call sites in `guess`, `snake`, `find`.
- Shop `buy` → `sdk.points.spend(itemId)` calling `points.php spend` (server validates price + creates order atomically).
- `sdk.points.summary` → reads `balance.php`/`points.php summary` (authoritative).
- Winstreak/pointstreak: keep as a **read-only client display** derived from the now-trustworthy ledger. Not a security boundary.

## Part D — Remaining findings (batched)

- **SEC-6 SSRF (`_push.php`):** allowlist push hosts (FCM `fcm.googleapis.com`/`*.google.com`, Mozilla `*.push.services.mozilla.com`, Apple `*.push.apple.com`, WNS `*.notify.windows.com`). Reject private/loopback. Set `CURLOPT_PROTOCOLS = CURLPROTO_HTTPS`, `CURLOPT_FOLLOWLOCATION = false`. Re-check at `subscribe` time in `push.php`.
- **SEC-7 headers (`app/.htaccess`):** add for HTML — CSP `default-src 'self'; object-src 'none'; frame-ancestors 'none'; base-uri 'self'` (allow `'unsafe-inline'` for now given inline scripts/styles; tighten later), `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: same-origin`.
- **SEC-8 color XSS:** escape `m.color` in `shell.js:589` and `:1389`; validate `color` against `/^#?[0-9a-fA-F]{3,8}$|^rgb/` in `store/install.php` (reject otherwise).
- **SEC-9 parent link:** run `w.link` through `normUrl` in `parent.js:607`; normalize on save in `wishlist/module.js` so stored links are always safe.
- **SEC-5 switch tokens:** reduce TTL; where feasible move the switch secret to an `HttpOnly` cookie. Do the safe subset; document anything deferred. (Most involved client change.)
- **SEC-12 upload:** login required (Part A) + per-user rate/size cap in `upload.php`.
- **SEC-13 CSRF:** add an `Origin`/`Referer` allowlist check on POST endpoints (helper in `_bootstrap.php`), on top of existing `SameSite=Lax`.
- **SEC-14 HTTPS:** enable the commented force-HTTPS redirect in `app/.htaccess`.

## UI copy rewrite (en/ru/lv) — driven by the `1234` removal

Files: `app/core/strings.js` (3 languages), `app/family.html`, `app/admin.html`, `app/api/mail/admin_invite_parent.{en,ru,lv}.html`, plus code comments in `accounts.php`, `admin.php`, `shell.js`.

Pattern of change:
- "First sign-in password: 1234" → "First sign-in password: {pass}" (value from server `temp_password`).
- "Reset {name}'s password to the one-time 1234?" → "Reset {name}'s password? A new one-time password will be shown." then "{name}: one-time password is {pass}".
- "New password (not 1234)" → "New password" (generic), validation message "At least 4 characters" (drop the 1234 mention).
- "You signed in with the one-time 1234." → "You signed in with a one-time password."
- Email templates: render the actual generated `{{temp_password}}`.

All three languages kept in lockstep. No long dashes in copy (Jeff's style).

## Rollout and prerequisites (critical)

Removing the anonymous fallback means **before this is deployed**:
1. Prod DB must have working login accounts (the parent owner + Артём's child account) with known passwords. If the family currently uses the app anonymously, create/confirm these accounts first (admin panel or a one-time SQL seed), or the family is locked out.
2. Prod `config.php`: set `single_user => false` (the code no longer honors `true`, but keep config consistent).
3. Run the new migrations on prod (`030_auth_throttle.sql`).
4. Optional: `DELETE FROM sessions;` to invalidate weak legacy session tokens immediately.

## Execution order (with verification gates)

Each phase ends with `node --check` on changed JS and `php -l` on changed PHP (PHP CLI being installed locally; if unavailable, lint on staging before merge).

1. **Boot-flow recon (A4)** — confirm registry/login ordering. No code yet.
2. **Auth core (A1, A2, A3, B)** — identity fallback removed, endpoints gated, strong tokens. Verify each endpoint 401s anonymous and works with a session.
3. **Throttle + temp password server (C1, C2, C3)** — rate limiting, generated temp password, email injection, migration.
4. **Server-authoritative points (CP1..CP6)** — enumerate tariff, new `_points.php`/`points.php`, block minting in `data.php`, tasks award, SDK + module rewrites. Verify a child cannot mint and honest play is unchanged.
5. **Remaining findings (Part D)** — SSRF, headers, color, parent link, switch tokens, upload cap, CSRF, HTTPS.
6. **UI copy rewrite** — all `1234` sites across en/ru/lv + emails + comments.
7. **Final verification** — full `node --check` sweep (matches CI), `php -l` sweep, manual logic review of the auth gate and points flow. Commit per phase on the branch. Summarize and hand to Jeff for review/merge.

## Verification approach (no live DB locally)
- Syntax: `php -l` (once PHP installs), `node --check` (CI parity).
- Logic: targeted reasoning + small PHP unit tests for pure helpers (tariff lookup, clamp, `rt_temp_password`, throttle math, push-host allowlist) that run without a DB.
- The auth gate and points endpoints get a documented manual test checklist for staging, since they need MySQL.
