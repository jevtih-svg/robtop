# RobTop — Аккаунты: пользовательские пути и план кликабельных прототипов

> Документ фазы проектирования. Сначала текст путей, потом два прохода ревью с улучшениями, потом план HTML-прототипов.
> Прототипы кликаются без ввода (видно, что происходит). Реальной разработки нет до одобрения.
> Парный контекст: `КОНТЕКСТ.md`, `ПЛАН-аккаунты-и-приглашения.md` (модель данных и API). Методология: `AGENTS.md`.
> Язык интерфейса прототипов: EN (по умолчанию), RU, LV (учитываем новую многоязычность).

---

## 0. Метод и порядок работы

1. Текст: полный перечень пользовательских путей (раздел 3).
2. Ревью №1, корректность и полнота, правки (раздел 4.1).
3. Ревью №2, безопасность, приватность, UX, многоязычность, доступность, правки (раздел 4.2).
4. План кликабельных HTML-прототипов (раздел 5).
5. Гейт: без реальной разработки до одобрения (раздел 6).

Принципы продукта (из ранее согласованного): семья (родитель и дети), приватность (никаких личных данных, только никнеймы; email только у родителей), одноразовый пароль `1234` со сменой при первом входе, сброс пароля ребёнку только родителем, дети могут звать детей с провизорным опекунством, передача прав настоящему родителю по email, логи сохраняются.

---

## 1. Актёры

1. Родитель-владелец (Owner): первый, кто создал семью. Полные права в своей семье.
2. Родитель (Parent): второй взрослый, приглашён в семью. Права как у владельца, кроме удаления семьи.
3. Ребёнок (Child): аккаунт без email и без личных данных, только никнейм и пароль.
4. Провизорный опекун (Provisional guardian): родитель, временно отвечающий за приглашённого чужого ребёнка, пока не появился его настоящий родитель.
5. Входящий родитель (Incoming parent): человек, которого пригласили по email стать настоящим родителем конкретного ребёнка.

Системные роли в данных: `family_members.role` = owner / parent / child; `guardianships.type` = primary / provisional.

---

## 2. Глоссарий состояний (понадобится для прототипов)

Состояния аккаунта: гость-без-пароля (только что по 1234, требуется смена), активный, отключён.
Состояния приглашения: ожидает (pending), принято, отклонено, отозвано, истекло.
Состояния опекунства: активное primary, активное provisional, разорвано (severed, остаётся в логе).
Состояния экрана (для UI): пусто, загрузка, успех, ошибка, требуется действие.

## 3. Пользовательские пути (полный перечень)

Формат пути: идентификатор, актёр, триггер, шаги (что делает пользователь и что делает система), задействованные экраны, итог, ветвления.

### A. Вход и онбординг

**A1. Родитель создаёт семью (саморегистрация).**
Актёр: новый родитель. Триггер: открыл приложение без входа.
Шаги: вводит никнейм, email, пароль -> система проверяет уникальность никнейма и email, хэширует пароль, создаёт аккаунт-родителя, создаёт семью и членство owner, открывает сессию.
Экраны: Выбор «Войти / Я родитель», Форма регистрации, Дашборд семьи.
Итог: родитель внутри, семья создана.
Ветвления: никнейм занят (A1a), email занят (A1b), слабый пароль (A1c).

**A2. Вход родителя.**
Актёр: родитель. Триггер: есть аккаунт. Шаги: вводит email и пароль -> сессия. Экраны: Вход, Дашборд. Ветвления: неверные данные (A2a), аккаунт отключён (A2b).

**A3. Вход ребёнка.**
Актёр: ребёнок. Шаги: вводит никнейм и пароль -> сессия. Экраны: Вход, Детский дашборд. Ветвления: первый вход по 1234 ведёт на обязательную смену пароля (A4), неверные данные (A3a).

**A4. Обязательная смена одноразового пароля.**
Актёр: любой, кто вошёл по 1234 (`must_change_password=1`). Триггер: вход по 1234. Шаги: система пускает только на экран смены пароля, остальное закрыто; пользователь задаёт новый пароль (не 1234) -> флаг снят, доступ открыт. Экраны: Смена пароля (форсированная). Итог: пароль сменён, полный доступ. Ветвления: пробует оставить 1234 (A4a, запрет), слишком короткий (A4b).

**A5. Выход.**
Актёр: любой. Шаги: «Выйти» -> сессия удалена. Доп. путь «Выйти со всех устройств» удаляет все сессии. Экраны: любой -> Вход.

**A6. Истечение сессии.**
Триггер: сессия просрочена. Система мягко возвращает на экран входа с пояснением. Экраны: любой -> Вход (с подсказкой «нужно войти снова»).

### B. Управление семьёй (родитель)

**B1. Родитель добавляет ребёнка.**
Актёр: родитель. Шаги: вводит никнейм ребёнка -> система создаёт детский аккаунт с одноразовым `1234` и флагом смены, добавляет в семью, заводит primary-опекунство. Родителю показывается, что пароль для первого входа `1234`. Экраны: Дашборд -> Добавить ребёнка -> Подтверждение с `1234`. Итог: ребёнок в семье. Ветвления: никнейм занят (B1a), подсказка выбрать другой.

**B2. Просмотр состава семьи.**
Актёр: родитель. Шаги: открывает «Семья» -> видит членов и роли, видит детей под опекой с пометкой «свой» или «гость». Экраны: Дашборд (список членов и детей).

**B3. Родительский дашборд по ребёнку (шаринг, только чтение).**
Актёр: родитель. Шаги: выбирает ребёнка -> видит его статистику и активность по модулям (чистка зубов, виш-лист и т.д.) только на чтение. Экраны: Карточка ребёнка, Статистика. Итог: видимость без права редактирования.

**B4. Родитель сбрасывает пароль ребёнку.**
Актёр: родитель-опекун. Шаги: на карточке ребёнка «Сбросить пароль» -> пароль ребёнка снова `1234`, флаг смены поднят. Экраны: Карточка ребёнка -> Подтверждение с `1234`. Итог: ребёнок войдёт по 1234 и сменит пароль.

**B5. Отключение или удаление члена (мягко).**
Актёр: родитель-владелец. Шаги: «Отключить» члена -> статус disabled, данные и логи сохраняются. Экраны: Карточка члена -> Подтверждение. Ветвления: нельзя отключить себя как единственного владельца (B5a).

**B6. Приглашение второго родителя в семью.**
См. C1. Результат: второй родитель получает права parent в семье.

### C. Приглашения

**C1. Родитель приглашает второго родителя (co-parent).**
Актёр: родитель. Шаги: вводит email второго родителя -> система создаёт приглашение, формирует ссылку, при настроенной почте шлёт письмо, иначе отдаёт ссылку для ручной пересылки. Экраны: Приглашения -> Форма email -> Экран со ссылкой (копировать). Итог: приглашение pending. Ветвления: см. жизненный цикл C5.

**C2. Ребёнок приглашает ребёнка (child to child).**
Актёр: ребёнок. Шаги: «Позвать друга» -> система создаёт приглашение без email, отдаёт ссылку или код для прямой передачи. Экраны: Детский дашборд -> Экран со ссылкой. Итог: приглашение pending. Решение заказчика (2026-06-05): свободно, без гейта родителя. Дети зовут друг друга сразу.

**C3. Родитель передаёт ребёнка настоящему родителю (transfer).**
Актёр: родитель-опекун (часто провизорный). Шаги: на карточке ребёнка «Привязать настоящего родителя» -> вводит email -> создаётся приглашение типа transfer с привязкой к этому ребёнку. Экраны: Карточка ребёнка -> Форма email -> Экран со ссылкой. Итог: приглашение pending, ждёт регистрации входящего родителя. Развязка в D2.

**C4. Посадочная страница приглашения (открытие ссылки).**
Актёр: приглашённый. Триггер: открыл ссылку с токеном. Шаги: система показывает, кто и куда зовёт, проверяет срок и статус. Дальше зависит от типа:
- child to child: спросить никнейм, создать ребёнка по 1234, привязать провизорно (D1).
- co-parent: спросить никнейм и пароль, создать родителя, добавить в семью.
- transfer: спросить никнейм и пароль, создать родителя, сформировать семью ребёнка, разорвать провизорную связь (D2).
Экраны: Посадочная (новый ребёнок), Посадочная (новый родитель). Ветвления: приглашение истекло или отозвано (C4a), уже принято (C4b).

**C5. Жизненный цикл приглашения (отзыв, повтор, отклонение, истечение).**
Актёр: приглашающий и приглашённый. Состояния: pending -> принято / отклонено / отозвано / истекло. Приглашающий может отозвать pending и переслать (старое гасится, выдаётся свежий токен). Экраны: Список отправленных приглашений со статусами и кнопками «Отозвать», «Переслать». Ветвления: принять после истечения нельзя (C5a).

### D. Провизорное опекунство и передача прав

**D1. Возникновение провизорного опекунства.**
Триггер: ребёнок B принял приглашение ребёнка A. Шаги системы: создаётся детский аккаунт B (1234, смена), определяется ближайший родитель A (его primary-опекун, владелец семьи A), создаётся провизорное опекунство B под этим родителем, B попадает в отдельную гостевую мини-семью с владельцем-провизорным родителем. Экраны: для B Посадочная и затем Смена пароля; для родителя A в дашборде появляется ребёнок-гость с пометкой «временно под вами».
Итог: родитель A имеет полные права опекуна над B. Ветвления: у A нет родителя (D1a, fallback на владельца семьи), двое детей зовут одного и того же нового ребёнка (D1b, первый создаёт, второй видит, что ник занят).

**D2. Передача прав настоящему родителю (разрыв провизорной связи).**
Триггер: входящий родитель принял transfer-приглашение по ребёнку B. Шаги системы: создаётся родитель, формируется нормальная семья B с владельцем-новым родителем, B переносится туда, создаётся primary-опекунство, провизорное опекунство переводится в severed с причиной (строка остаётся, это лог). Данные B остаются с ним. Экраны: для входящего родителя Посадочная и Дашборд; для прежнего провизорного родителя ребёнок-гость пропадает из активных, остаётся только в истории.
Итог: B в своей нормальной семье, прежняя связь разорвана навсегда, в логах всё сохранено.

**D3. Провизорное опекунство без передачи.**
Триггер: настоящего родителя так и не привязали. Состояние: B остаётся под провизорным опекуном неограниченно. Это допустимо. Экраны: ребёнок-гость в дашборде с пометкой «временно».

**D4. Провизорный родитель убирает гостя.**
Актёр: провизорный родитель. Шаги: «Убрать гостя» -> мягкое отключение членства и опекунства, лог сохраняется. Экраны: Карточка гостя -> Подтверждение.

### E. Пароли и сбросы

**E1. Обязательная смена 1234.** См. A4.

**E2. Сброс пароля ребёнку родителем.** См. B4. Единственный путь восстановления для ребёнка.

**E3. Родитель забыл пароль (по email).**
Актёр: родитель. Шаги: «Забыли пароль» -> вводит email -> система при наличии аккаунта создаёт одноразовый токен и шлёт письмо со ссылкой, ответ всегда одинаковый (не раскрываем существование email). По ссылке родитель задаёт новый пароль. Экраны: Забыли пароль, Письмо со ссылкой, Сброс по ссылке, Вход. Ветвления: токен истёк или использован (E3a), почта не настроена (E3b, запасной путь: сброс другим владельцем или админом).

**E4. Ребёнок забыл пароль.**
Актёр: ребёнок. Шаги: на экране входа подсказка «Попроси родителя сбросить пароль». Сам сбросить не может. Дальше путь B4. Экраны: Вход (подсказка), затем родитель делает B4.

### F. Краевые случаи и ошибки

F1. Никнейм занят: подсказка и предложение вариантов (никнейм глобально уникален).
F2. Email занят или уже зарегистрирован: при co-parent или transfer вход существующим аккаунтом и присоединение (вместо создания нового).
F3. Приглашение истекло или отозвано при открытии: понятный экран и предложение запросить новое.
F4. Аккаунт отключён при входе: понятная ошибка без деталей.
F5. Почта недоступна: приглашения работают как ссылка для ручной пересылки, родительский сброс по email временно недоступен (запасной путь через второго владельца или админа).
F6. Двое детей зовут одного нового: первый создаёт, второй получает «ник занят» или присоединение по правилам.
F7. Сетевые и серверные ошибки: единый стиль сообщения и кнопка повторить.

## 4. Журнал ревью (два прохода)

### 4.1. Ревью №1 (полнота и корректность). Что нашли и что исправили.

1. Не было пути восстановления для ребёнка. Добавлен E4 (ребёнок не может сам, только через родителя B4).
2. Не было обработки существующего аккаунта при приёме приглашения. Добавлен F2 (вход и присоединение вместо создания дубликата).
3. Не было полного жизненного цикла приглашения. Добавлен C5 (отзыв, повтор, отклонение, истечение) и экраны статусов.
4. Не было определения «ближайший родитель» при двух родителях у пригласившего. Зафиксировано: берётся primary-опекун пригласившего, при отсутствии владелец его семьи (D1a).
5. Не было состояния «провизорное опекунство навсегда без передачи». Добавлен D3 и удаление гостя D4.
6. Не было выхода со всех устройств и истечения сессии. Добавлены A5 и A6.
7. Уточнено, что данные ребёнка при передаче остаются с ним (D2), видимость переходит к новому родителю.

### 4.2. Ревью №2 (безопасность, приватность, UX, многоязычность, доступность). Что нашли и что исправили.

Безопасность и приватность.
1. Одноразовый `1234` известен и слаб. Зафиксировано жёсткое правило: до смены пароля доступ закрыт ко всему, кроме экрана смены (A4), плюс лимит попыток входа и нейтральные сообщения об ошибке.
2. Подтверждено отсутствие email у детей на всех путях. Email появляется только в A1, C1, C3, E3, и только у родителей.
3. Токены приглашений и сбросов одноразовые, с истечением, в базе только хэш. На запрос сброса единый ответ (E3), чтобы не раскрывать наличие email.
4. Детские приглашения (C2): добавлен открытый вопрос про опциональный гейт «требуется одобрение родителя», по умолчанию предлагается включать для младших. Решение за заказчиком.
5. Провизорный родитель несёт ответственность за гостя: добавляем явную пометку и короткое пояснение в UI (D1, D3).
6. **Картинки детей: приватны и нешарабельны (решение заказчика 2026-06-05).** Дети могут загружать картинки (виш-лист), но они нигде не публикуются. Видят их ТОЛЬКО сам ребёнок и его ПРЯМОЙ (primary) родитель; временный (provisional) опекун изображения НЕ видит никогда. В прототипе это правило показано на экранах childSafety и uploadGuard и в safebar. В UI и письмах формулируем аккуратно, без переобещаний типа «не храним данных».

UX.
6. Добавлены пустые состояния (нет детей, нет приглашений) и состояния успеха (ребёнок создан, приглашение отправлено, пароль изменён).
7. Ссылку-приглашение всегда показываем с кнопкой копирования и понятным текстом, что с ней делать.
8. Каждый ключевой экран получает короткую подсказку «что происходит» и тултип на главном действии (см. план прототипов).
9. Никнейм-конфликты решаем не молчаливой ошибкой, а подсказкой с вариантами.

Многоязычность.
10. Все строки и тултипы локализуем на EN, RU, LV. Язык переключается и запоминается. Псевдоперевод для проверки длинных строк (LV и RU длиннее EN).
11. Направление письма у EN, RU, LV слева направо, RTL не требуется. Учитываем, что RU и LV строки длиннее, верстка не должна ломаться.

Доступность и мобильность.
12. Цели нажатия не меньше 44 пикселей, видимый фокус, подписи для скринридера, контраст по WCAG AA на неон-теме (проверить тёмные тексты на ярких кнопках).
13. Мобайл-ферст: один столбец, нижние действия в зоне большого пальца.

Итог ревью: пути закрывают вход, семью, приглашения, провизорное опекунство, сбросы и краевые случаи. Открытый вопрос только один продуктовый: гейт на детские приглашения (C2). Остальное готово к прототипированию.

## 5. План кликабельных HTML-прототипов

### 5.1. Цели и принципы

1. Кликабельно без ввода: ни одного обязательного поля для набора. Поля показаны заранее заполненными демо-данными, кнопки сразу ведут на следующий экран. Видно, что происходит на каждом шаге.
2. Полнота путей: каждый путь из раздела 3 проходится кликами от начала до конца, включая ветвления (ошибки, истечение, провизорное опекунство, передача).
3. Outstanding UI/UX: единая неон-тема приложения, аккуратная иерархия, состояния (пусто, успех, ошибка), микро-анимации переходов, мобильный фрейм.
4. Инструкции и тултипы: на каждом экране короткая панель «что происходит» и значки-подсказки на ключевых элементах.
5. Многоязычность: переключатель EN, RU, LV, все надписи и подсказки переводятся, выбор запоминается.
6. Без бэкенда и без реального ввода: это визуальный прототип, не функциональная разработка.

### 5.2. Инвентарь экранов (полный список)

Вход и онбординг: Стартовый выбор (войти или я родитель), Вход (поле логина и пароля), Регистрация родителя, Смена пароля (форсированная), Сессия истекла.
Дашборды: Родительский дашборд (семья, дети, приглашения), Детский дашборд (мои занятия, позвать друга), Карточка ребёнка (статистика только чтение), Статистика по модулю.
Семья: Добавить ребёнка, Подтверждение создания ребёнка с 1234, Отключить или убрать члена, Подтверждение.
Приглашения: Пригласить второго родителя (email), Позвать друга (ребёнок), Экран со ссылкой (копировать), Список отправленных приглашений со статусами.
Посадочные: Посадочная для нового ребёнка (child to child), Посадочная для нового или входящего родителя (co-parent и transfer), Передача прав подтверждение, Провизорная связь разорвана подтверждение.
Пароли: Забыли пароль (родитель, email), Сброс по ссылке из письма, Подсказка ребёнку «попроси родителя».
Состояния и ошибки: Пусто (нет детей), Пусто (нет приглашений), Никнейм занят, Приглашение истекло или отозвано, Аккаунт отключён, Почта недоступна (ручная ссылка), Загрузка, Успех, Сетевая ошибка.

Ориентировочно от 24 до 28 экранов и состояний.

### 5.3. Карта навигации (потоки в экраны)

Стартовый выбор -> Вход -> (родитель) Родительский дашборд, либо (ребёнок по 1234) Смена пароля -> Детский дашборд.
Стартовый выбор -> Регистрация родителя -> Родительский дашборд.
Родительский дашборд -> Добавить ребёнка -> Подтверждение 1234 -> Родительский дашборд.
Родительский дашборд -> Пригласить второго родителя -> Экран со ссылкой -> Список приглашений.
Родительский дашборд -> Карточка ребёнка -> (Сбросить пароль -> Подтверждение 1234), (Привязать родителя -> Экран со ссылкой), (Статистика).
Детский дашборд -> Позвать друга -> Экран со ссылкой.
Посадочная (ссылка приглашения) -> (ребёнок) никнейм -> Смена пароля -> Детский дашборд; (родитель) никнейм и пароль -> Дашборд; (transfer) -> Передача прав подтверждение -> Провизорная связь разорвана.
Забыли пароль -> письмо -> Сброс по ссылке -> Вход.
Ветви ошибок доступны кнопками-сценариями: истечение, ник занят, отключён, почта недоступна.

Карту нарисуем как наглядную схему внутри прототипа (мини-диаграмма потоков).

### 5.4. Как реализуем «кликабельно без ввода»

1. Демо-данные зашиты: например семья с родителем «Alex», детьми «Robik» и гостем «Sunny», одно приглашение pending. Поля ввода показаны как заполненные, нередактируемые, с подписью «демо».
2. Любая кнопка действия просто переходит на следующий экран сценария, ничего не отправляя. Никакого набора текста не требуется.
3. Сверху или сбоку селектор сценариев: список всех путей из раздела 3 (A1, B1, C2, D2 и т.д.). Клик по сценарию запускает его пошагово.
4. Степпер: «шаг N из M» с кнопками назад и вперёд, чтобы пройти путь не только по экранным кнопкам, но и стрелками.
5. Бейдж «прототип, демо-данные, без реального входа» виден всегда.
6. Состояние хранится в памяти страницы, без localStorage и без сервера.

### 5.5. UI/UX-стандарты

1. Тема: неон приложения (фон-градиенты, cyan, pink, purple, тёмные карточки, шрифты Russo One для заголовков и Rubik для текста).
2. Мобильный фрейм: прототип показан в рамке телефона по центру, ширина около 390 пикселей, безопасные зоны учтены.
3. Состояния компонентов: обычное, наведение, нажатие, отключено, загрузка, ошибка, успех, пусто. Показать каждое там, где оно встречается.
4. Движение: мягкие переходы между экранами (fade и slide), без резкости, уважение к prefers-reduced-motion.
5. Иерархия: один главный CTA на экран, второстепенные действия визуально тише.
6. Обратная связь: тосты успеха, понятные блоки ошибок, подтверждения для разрушающих действий (отключить, убрать гостя).
7. Доступность: контраст AA, видимый фокус, цели от 44 пикселей, подписи для скринридера, тексты не только цветом.

### 5.6. Тултипы и инструкции

1. Панель «что происходит»: короткая строка вверху каждого экрана, объясняет шаг человеческим языком (например «Ребёнок входит по никнейму и одноразовому паролю»).
2. Значок-подсказка (i) на ключевых элементах: при наведении или тапе показывает 1-2 предложения (например на поле пароля ребёнка «У детей нет email, только никнейм»).
3. Поясняющие сноски на чувствительных шагах: провизорное опекунство, передача прав, почему 1234 надо сменить, почему сброс ребёнка только через родителя.
4. Аннотации-выноски на сложных экранах посадочной и передачи прав, чтобы было видно, что меняется в данных за кадром.
5. Все тексты подсказок переводятся (EN, RU, LV).

### 5.7. Многоязычность в прототипе (EN, RU, LV)

1. Все строки вынесены в таблицу переводов вида ключ -> { en, ru, lv }, ничего не зашито в разметку.
2. Переключатель языка вверху, мгновенно перерисовывает экран, выбор запоминается в памяти страницы.
3. Ключи именуем так же, как в реальном i18n приложения (`core/strings.js`), чтобы прототип сразу служил источником готовых переводов.
4. Проверяем длинные строки RU и LV, чтобы кнопки и карточки не ломались.
5. По умолчанию EN, как в приложении.

### 5.8. Структура файлов и что НЕ входит

Структура: один самодостаточный файл `family-prototype.html` (HTML, CSS, JS внутри), без внешних зависимостей и без сети. Внутри: селектор сценариев, степпер, рамка телефона, экраны, таблица переводов, тултипы, мини-карта потоков.
По желанию отдельный файл со схемой навигации, если будет полезно как картинка.
Не входит: бэкенд, база, реальная отправка писем, настоящая авторизация, реальный ввод данных. Это визуальный кликабельный прототип.
