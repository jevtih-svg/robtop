| `app/modules/<id>/module.css` | стили, скоуп `[data-mod="<id>"]` |

## Шаг 1 — папка модуля `app/modules/<id>/`

### module.json (шаблон)

```json
{
  "id": "mood",
  "name": "Настроение дня",
  "version": "1.0.0",
  "color": "#ffd23b",
  "status": "active",
  "wide": false,
  "entry": "module.js",
  "styles": "module.css",
  "roles": { "edit": ["child"], "read": ["child", "parent"] },
  "data": { "store": "generic", "collections": ["entries"] },
  "server": false,
  "permissions": [],
  "min_shell": "2.0.0",
  "sort": 30
}
```

`server:true` + `data.store:"dedicated"` — только для тяжёлых модулей с собственной таблицей и `api.php` (образец: `app/modules/wishlist/`; диспетчер `api/action.php` делегирует в `modules/<id>/api.php`). По умолчанию — generic: данные живут в `module_data` через `api/data.php`, миграция БД для данных НЕ нужна. `min_shell` пока зарезервировано (оболочка его не проверяет) — оставляй `"2.0.0"` для единообразия манифестов.

### module.js (скелет)

Эталоны в репо: `modules/reverse/module.js` (простой) и `modules/teeth/module.js` (очки, родительская панель, вкладки). Рамка:

```js
/* RobTop — модуль «<Название>». <одно предложение что делает>. */
(function(){
  "use strict";
  var MESSAGES={
    en:{ mood:{ title:"Mood of the Day", subtitle:"…", /* все строки UI */ }},
    ru:{ mood:{ title:"Настроение дня", subtitle:"…" }},
    lv:{ mood:{ title:"…", subtitle:"…" }}
  };
  var BACK_IC='<svg viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2.4" stroke-linecap="round" stroke-linejoin="round"><path d="M15 5l-7 7 7 7"/></svg>';
  var sdk=null, root=null, E={};
  function esc(s){ return String(s==null?"":s).replace(/[&<>"']/g,function(c){return {"&":"&amp;","<":"&lt;",">":"&gt;","\"":"&quot;","'":"&#39;"}[c];}); }
  function t(k,p){ return sdk.t(k,p); }

  function mount(rootEl, theSdk){
    sdk=theSdk; root=rootEl;
    root.innerHTML='…';                       // только внутрь root
    // шапка с кнопкой назад → sdk.ui.back(); HUD → sdk.ui.hud({...})
    // данные → sdk.data.list/create/update/move/favorite/remove/restore("<collection>", …)
  }
```

Для `title` переиспользуй готовые переводы плитки `tile.<id>` из `core/strings.js` (для плановых плиток они уже выверены — не переводи заново). Латышский проверяй внимательнее всего.

ОПЦИОНАЛЬНЫЙ хук `refresh` (живое обновление, v2026.06.07.47+): sync-поллер оболочки зовёт `def.refresh()` у АКТИВНОГО модуля, когда на сервере изменились данные в скоупе зрителя (другое устройство, родитель, член семьи). Если данные модуля может менять кто-то кроме самого ребёнка (задания, семейный пул, гранты) — объяви `refresh:function(){ if(<покой: нет busy/открытых шторок/мастера>) load(); }` и перечитай данные БЕЗ повторного навешивания листенеров (паттерн — `bank`/`walk`). Оболочка сама не дёргает refresh при фокусе в поле ввода или открытой `.overlay.show`, но внутренние состояния модуля (мастер, busy) — твоя забота. Без хука модуль просто живёт по-старому (обновление при mount).

Жёсткие правила контракта (нарушение = баг):
- Никакого DOM вне `root`, никакого `fetch` напрямую — только `sdk.*` (`sdk.api.*` для особых случаев). Оболочка сама инъектит css/js с `?v=` и вызывает `mount/unmount`.
- Ни одной захардкоженной пользовательской строки: всё через `sdk.t`/`sdk.plural`/`sdk.formatDate`, словари `messages` сразу на en+ru+lv (en — язык по умолчанию). Бэкенд-ошибки — коды `err.<code>`.
- События статистики: операции ЗАПИСИ `sdk.data.*` (create/update/move/favorite/delete/restore) сервер зеркалит в `events` generic-типами (`created`, `edited`, …) без доменного смысла; чтения не логируются, в демо-режиме события не пишутся вовсе. Поэтому значимые действия трекай явно: `sdk.events.track(type, payload)` с говорящим типом (напр. `mood_set`) — это источник будущего родительского дашборда.
- Очки: `sdk.points.add(n, reason)` (леджер `bank`), укажи `"points"` в `permissions`. Тарифы, виды транзакций, винстрик и подпись в истории Копилки (`bank.r_<reason>` в своих messages) — ОБЯЗАТЕЛЬНО по `ГАЙД-очки.md`.
- Родительская панель внутри модуля — гейт по роли сессии: `sdk.role==="parent" || sdk.isDemo()` (хелпер `parentAllowed()`, образцы `teeth`/`bank`); кнопку панели ребёнку НЕ рендерить. PIN и `sdk.admin.verify` упразднены 2026-06-07.
- UX ребёнка: тап-таргеты ≥ 44px, крупные кнопки, подтверждение перед удалением, `sdk.ui.undo` для отмен, приятные анимации; неон-стиль на токенах `core/ui.css`, акцент — цвет модуля.

SDK-поверхность (полностью — `app/core/sdk.js`): `data.list/get/create/update/move/favorite/remove/restore`, `events.track`, `media.pick/upload`, `ui.toast/confirm/sheet/undo/confetti/haptics/chime/hud/fab/back/enableDrag`, `points.*`, `storage.local`, `t/plural/formatDate/i18n`, `can(action)`, `user/role`, `isDemo` (`admin.verify` удалён 2026-06-07). Демо-режим (file:// или сервер недоступен) работает автоматически: `sdk.data` пишет в localStorage — модуль обязан работать и там.

### module.css

Все селекторы со скоупом модуля: либо префикс классов (`.mood-…`), либо `[data-mod="mood"] …`. Не переопределять глобальные токены/компоненты `core/ui.css` — использовать их.

## Шаг 2 — реестр БД (миграция)

Плитка на главном и строка в настройках берутся из таблицы `modules` (`api/registry.php` → только `enabled=1`; `api/modules.php?all=1` → все для родительского экрана). Авто-раннер `api/_migrate.php` применяет `app/api/migrations/*.sql` по порядку имён при первом обращении к БД — деплоя достаточно, phpMyAdmin не нужен.

1. Посмотри `ls app/api/migrations/` — возьми следующий номер `00N`.
2. Создай `app/api/migrations/00N_<id>.sql` (идемпотентно!):

```sql
INSERT INTO modules (id, name, version, manifest, source, trusted, server, enabled, sort_order) VALUES
 ('mood','Настроение дня','1.0.0','{"color":"#ffd23b","status":"active","wide":false,"roles":{"edit":["child"],"read":["child","parent"]}}','native',1,0,1,30)
ON DUPLICATE KEY UPDATE name=VALUES(name), version=VALUES(version), manifest=VALUES(manifest),
 server=VALUES(server), enabled=VALUES(enabled), sort_order=VALUES(sort_order), updated_at=NOW();
```

Нюансы:
- Для 12 плановых плиток (mood, guess, names, days, find, museum, rating, lost, bank…) строка УЖЕ есть в БД со `status:"soon"` (сид `002_subapps.sql`) — `ON DUPLICATE KEY UPDATE` переведёт её в active. Для нового id — вставится новая.
- `manifest` в БД — это то, что реально читает оболочка (цвет, status, wide, roles). Держи его побайтово согласованным с `module.json`.
- `enabled=1` — решение Джеффа по умолчанию (2026-06-07): модуль виден сразу, родитель может выключить в настройках. Если Джефф явно просит «пусть родитель сам включит» — ставь `enabled=0` и напиши об этом в журнале.
- Обнови тот же сид в корневом `schema.sql` (одноразовый импорт чистой БД) — там та же строка модуля.
- `roles.edit` проверяется сервером (`data.php` → `rt_role_can`): кто не в списке — 403 на запись.
- Раннер режет файл по `;` простым сплиттером (`rt_split_sql`) БЕЗ разбора строковых литералов: ни одного `;` внутри значений/JSON манифеста; одно выражение — один завершающий `;`.
- MySQL в песочнице нет — идемпотентность обеспечивай конструктивно: только `CREATE TABLE IF NOT EXISTS` / `INSERT … ON DUPLICATE KEY UPDATE`, никаких `ALTER`; мысленный двойной прогон: второй запуск ничего не меняет и не падает.

### Шаг 4а — родительский экран (Обзор / Экран приложения / Виш-лист / Журнал)

Состояние с 2026-06-07 (v2026.06.07.16): экран РЕАЛИЗОВАН в приложении — `core/parent.js` + `api/parent.php` (read-only дашборд вместо детского дома для родительской сессии). Прототип `parent-prototype.html` остаётся для дизайн-итераций (отстаёт от боевого, боевой первичен). Принципы после переработки по фидбеку заказчика: метрики считают только ОСМЫСЛЕННЫЕ действия ребёнка (`isDomain()` в `core/parent.js`), тап по карточке приложения открывает ЭКРАН ПРИЛОЖЕНИЯ с контентом ребёнка (события вторичны, внизу), журнал — человеческие формулировки с подгрузкой. Обязанности нового модуля:

1. **Писать доменные события** через `sdk.events.track` с осмысленным `type` и компактным `meta` (образцы: `day_rated {stars}`, `mood_set {mood}`). Модуль без событий невидим в статистике и журнале — это дефект роллаута.
2. **Шаблоны журнала ×3 языка**: ключ `parent.ev.<module>.<type>` (en/ru/lv). Доложить их можно из `module.js` через `messages` контракта register — `core/parent.js` сам найдёт ключ; без ключа событие получит generic-текст. Формулировки — в третьем лице о ребёнке. Особые случаи смотри в `evText()` (teeth done/skipped по `to_status`, guess по `meta.result`).
3. **`isDomain()` — фильтр осмысленных действий** (HUD, график, сводки, лента): у известных модулей считается ТОЛЬКО доменный тип (generic-дубли `created/edited` от `sdk.data.*` НЕ считаются — иначе двойной счёт); у неизвестных — generic-записи (кроме коллекции `meta`). Если твой модуль пишет И доменный track, И data-записи — добавь ветку в `isDomain()`, иначе его активность посчитается дважды. Шум никогда не считается: `opened_*`, `viewed_*`, `bank`, `accounts` (в ленте из accounts остаётся только «Вошёл в приложение»).
4. **Экран приложения (контент ребёнка)**: `api/parent.php` отдаёт свежие записи generic-стора (`content`, без коллекций `meta`, фото по правилу приватности), `moduleContentHtml()` в `core/parent.js` рендерит их. Известным модулям — своя ветка (оценки с текстами, раунды, слова, чистки); неизвестный модуль автоматически получает generic-список (первое строковое поле data + дата). Хочешь красивый экран — добавь ветку и строки `parent.m.*`/свои ключи ×3.
5. **Сводная карточка в Обзоре**: характерный агрегат (серия, среднее, «чаще всего») — ветка в `overviewHtml()`/`agg()` + ключ `parent.sum.<id>` ×3 языка. Без этого — generic-счётчик действий.
6. **Цвет и иконка модуля** берутся из реестра (манифест) — заполняй осмысленно ещё на Шаге 1.

При роллауте также добавь модуль в `parent-prototype.html` (MODS + словари + генератор демо-событий), чтобы прототип не отставал от состава приложения.

### Шаг 4б — мастер-админка (админ-контракт модуля)

Состояние с 2026-06-07: есть мастер-консоль `app/admin.html` + `api/admin.php` (гейт `rt_require_admin`, таблица `admins`, миграция 008). КАЖДОЕ новое субприложение обязано «показаться» админке. Что получается автоматически, без кода:

1. **Реестр**: строка модуля появляется в админке → «Приложения» (вкл/выкл, порядок) сразу после миграции Шага 2.
2. **Контент**: все коллекции модуля в `module_data` сами появляются в «Контент модулей» (generic-браузер: смотреть/добавить/править JSON/удалить). Модуль, хранящий данные НЕ в `module_data` (как wishlist в своей таблице), требует ветки в `api/admin.php` (`records`/`record_delete`) — избегай этого, используй generic-стор.
3. **Журнал**: события `sdk.events.track` видны в админке → «Журнал» с фильтром по модулю (ещё одна причина писать доменные события, см. Шаг 4а).
4. **Очки**: если модуль начисляет очки — ТОЛЬКО через леджер `bank/points` (`sdk.points.add`); тогда транзакции и балансы видны и правятся в админке → «Очки».

Опционально (для нестандартного контента вроде уравнений «Угадай число»): объяви в `module.json` блок `"admin"` — он уже отдаётся панели через op `modules` (поле `admin` в ответе) и помечается бейджем «админ-контракт»:

```json
"admin": { "collections": [ { "id": "equations", "label": "Уравнения", "fields": [
  { "k": "text", "label": "Уравнение", "type": "string" },
  { "k": "answer", "label": "Ответ", "type": "number" } ] } ] }
```

Сейчас панель редактирует такие коллекции generic-JSON-редактором; блок `admin` — контракт под типизированные формы (зарезервировано, не ломается). КАРТИНКИ: админка никогда не отдаёт содержимое/пути картинок — если модуль хранит фото, поле должно называться `photo`/`image`/`dataUrl` (эти ключи вырезаются из ответов админки автоматически).

Приёмка шага: после деплоя зайти в админку и проверить, что модуль виден в «Приложениях», его коллекции — в «Контенте», события — в «Журнале».

## Шаг 5 — проверки (до коммита; пропуск = нарушение AGENTS.md)

1. `node --check` на каждый изменённый/новый `.js` (это же гоняет CI на push).
2. jsdom-smoke (`npm install jsdom` в песочнице работает). Готового харнесса в репо нет — собери стаб: загрузи в jsdom `core/i18n.js`, `core/strings.js`, `core/sdk.js`, `core/loader.js`; подмени `RT._shell` минимальным объектом (`toast/hud/buzz/chime/confirm/sheet/moduleView/showModule/showHome/fabDestroy` + `user`, `demo:true`); затем зарегистрированный модуль: mount → действия → unmount. Минимум: монтируется ×3 языка, основные действия работают, рендер после `unmount` — no-op, роль parent не может писать.

## Шаг 7 — git (включение новых файлов и папок)

Перед коммитом: `git status` — убедись, что в списке только ожидаемые файлы. Папку модуля добавляй целиком — это и включает все новые файлы/папки субприложения:

```bash
cd "/Users/jefftilley/Cowork/30 Projects/RobTop/Приложение RobTop"
git status
git add app/modules/<id>/ \
        app/api/migrations/00N_<id>.sql \
        app/core/shell.js app/core/strings.js app/index.html \
        schema.sql КОНТЕКСТ.md
git commit -m "feat(<id>): модуль «<Название>» — <суть в одну строку>; v<версия приложения>"
git push origin main
```

Никогда не добавлять (gitignore — это решение заказчика, не случайность): `config.php`, `media/`, `uploads/`, `app/apps/*` (кроме уже версионированных `.htaccess` и `_catalog/`). Не использовать `git add -f` для этих путей. Если модуль dedicated — добавь также `app/modules/<id>/api.php`, `app/modules/<id>/schema.sql` и изменённый `app/api/action.php` (они покрываются `git add app/modules/<id>/` + явным путём диспетчера).

После push: вебхук Hostinger подтянет `main`; `rt_migrate` применит `00N_<id>.sql` при первом запросе к API. Smoke на бою: `#appver` слева внизу = новая версия (hard-refresh; iOS-PWA — пересоздать иконку), плитка открывается, данные сохраняются, в настройках модуль виден и тумблер работает, событие в `events` пишется.
