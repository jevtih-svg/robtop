## 0. Решения заказчика (зафиксировано)

- **Модель связей: семья.** Родитель приглашает ребёнка/детей и второго родителя. Образуется «семейный круг».
  - **Дети могут приглашать других детей.** Приглашённый ребёнок сначала **без своего родителя** — он временно (провизорно) привязывается к **родителю пригласившего**, и этот родитель получает над ним **все права опекуна**.
  - **Родитель может пригласить другого родителя, выбрав ребёнка** (передать/разделить права над ним) — по email.
  - **Как только приглашённый по email родитель создаёт аккаунт** — для ребёнка формируется **нормальная семья**, а **временная связь рвётся навсегда** (но в логах остаётся запись).
- **Никаких личных данных.** Приложение **не хранит имён** и любых PII. Только **никнеймы**.
- **Дети: только никнейм + пароль, без email.** Email есть **только у родителей** (нужен для входа, сброса пароля и приглашения других родителей).
- **Регистрация:** вводишь **никнейм** и **одноразовый пароль `1234`**, который **обязательно меняешь при первом входе**.
- **Сброс пароля:** ребёнок **сам не может**; пароль ребёнку **сбрасывает родитель**. Свой пароль родитель сбрасывает **по email**.
- **Логи сохраняются.** Система детская → всё важное пишется в журнал и не удаляется (подотчётность, кто кого пригласил, кто что сбросил).

## 1. Принцип приватности (главное ограничение дизайна)

Это фундамент, под него подстроено всё остальное:

- **Никакого PII.** Не храним настоящие имена, фамилии, телефоны, адреса, даты рождения. Только **никнейм** (придуманное имя, не личные данные) и, у детей, пароль.
- **У детей нет email.** Детский аккаунт = `никнейм + пароль`. Точка. Это убирает главный источник персональных данных и делает невозможным «достучаться» до ребёнка извне по почте.
- **Email только у родителей** и используется строго для трёх вещей: вход родителя, сброс пароля родителя, приглашение другого родителя. Дети к почте никак не привязаны.
- **Минимизация в логах.** В журнал пишем id и типы событий, а не содержимое и не секреты. Email родителя в логах — только там, где это сам факт (напр. «отправлено приглашение на email»), и по возможности в маскированном виде.
- **Картинки детей (виш-лист): приватны и нешарабельны.** Дети МОГУТ загружать картинки (например, в виш-лист) — значит формулировки «не храним никаких данных / каждая загрузка защищена» НЕЛЬЗЯ использовать, это неточно. Точное правило (решение заказчика 2026-06-05): загруженные картинки **нигде не публикуются и не пересылаются** (нет шаринга, ссылок, экспорта). Просматривать картинки виш-листа может **только сам ребёнок и его ПРЯМОЙ (primary) родитель**. **Временный (provisional) опекун НИКОГДА не видит картинки ребёнка**, даже пока опекает его (он может видеть базовую активность/статистику, но не изображения). Это жёсткий инвариант, см. §4.7 и §8.
- **Аккуратные формулировки.** В UI и письмах не переобещаем безопасность. Не пишем «не храним личные данные» (картинки — это данные) и «каждая загрузка гарантированно безопасна». Пишем фактически: вход по никнейму без email; картинки приватны и нигде не публикуются; их видит только ребёнок и прямой родитель, никогда временный опекун.

Следствие: даже при утечке базы по детям нельзя узнать ничего, кроме никнеймов и их активности; картинки виш-листа доступны строго ребёнку и его прямому родителю.

## 2. Понятия (словарь)

- **Аккаунт (`users`)** — личность для входа. Два вида (`kind`): **`child`** (никнейм + пароль, без email) и **`parent`** (email + пароль, есть никнейм).
- **Семья (`families`)** — круг, в центре которого дети; держится на одном-двух родителях.
- **Членство (`family_members`)** — связь «аккаунт ↔ семья» + роль `owner`/`parent`/`child` (это ВНУТРЕННИЕ значения данных).
- **Опекунство (`guardianships`)** — ребро «ребёнок ← опекун-родитель» с типом **`primary`** (настоящий родитель) или **`provisional`** (временный, пока настоящего нет). Именно здесь живёт логика «временной привязки к родителю пригласившего».
- **Роли в UI (решение заказчика 2026-06-05).** В интерфейсе показываем ТОЛЬКО три роли и НЕ показываем «владелец/owner»: **«родитель/опекун»** (parent/guardian — для owner и parent, т.е. любой настоящий родитель), **«временный опекун»** (temporary guardian — провизорный опекун гостя), **«ребёнок»** (child). Внутренняя модель данных (`owner`/`parent`/`child` + `primary`/`provisional`) не меняется, это лишь маппинг подписей: owner+parent → «родитель/опекун»; provisional-опекун → «временный опекун»; child → «ребёнок». Роль НЕ дублируем подписью под именем — она показана только «таблеткой» (pill) справа.
- **Приглашение (`invitations`)** — запись о зове нового участника; разные **типы** (ребёнок→ребёнок, родитель→родитель и т.д.). Хранит граф «кто кого пригласил».
- **Сессия (`sessions`)** — токен входа в HttpOnly-cookie (в БД только его хэш).
- **Сброс пароля (`password_resets`)** — одноразовые токены для сброса пароля **родителя** по email.
- **Журнал (`events`)** — общий append-only лог (уже существует); сюда пишем все значимые действия.

## 3. Модель данных (таблицы)

Принцип прежний — **аддитивно**: существующие таблицы не ломаем, применяем авто-миграцией `app/api/migrations/003_accounts.sql` (раннер выполняет файл один раз, поэтому одноразовый `ALTER TABLE` безопасен).

### 3.1. `users` — расширение (без личных данных)

Существующие `id`, `name`, `role`, `pin_hash`, `created_at` остаются; **`name` переосмысляем как `nickname`** (никнейм, не PII). Добавляем:

```
kind                  ENUM('parent','child') NOT NULL DEFAULT 'child'
email                 VARCHAR(190) NULL UNIQUE   -- ТОЛЬКО у parent; у child всегда NULL
password_hash         VARCHAR(255) NULL          -- password_hash() (bcrypt/argon2id)
must_change_password  TINYINT(1)   NOT NULL DEFAULT 0   -- 1 = вошёл по 1234, надо сменить
status                ENUM('active','disabled') NOT NULL DEFAULT 'active'
invited_by            INT UNSIGNED NULL          -- кто пригласил (ребро графа), FK users(id)
last_login_at         DATETIME     NULL
```

Правила-инварианты (проверяются в коде и, где можно, в БД):
- `kind='child'` ⇒ `email IS NULL`. Дети без почты — это инвариант приватности.
- `kind='parent'` ⇒ `email` обязателен и уникален (нормализованный: trim + нижний регистр).
- Никнейм **глобально уникален** (case-insensitive) — потому что ребёнок входит по никнейму (см. 8.1). Альтернатива (никнейм уникален в пределах семьи + выбор семьи на устройстве) описана как открытый вопрос 10.1.
- `must_change_password=1` блокирует доступ ко всему, кроме экрана смены пароля.

### 3.2. `families`

```
id          INT UNSIGNED PK AUTO_INCREMENT
owner_id    INT UNSIGNED NOT NULL          -- FK users(id), родитель-владелец
label       VARCHAR(40)  NULL              -- необязательная метка («Дом»), НЕ имя человека
created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
deleted_at  DATETIME NULL
```

### 3.3. `family_members`

```
id          BIGINT UNSIGNED PK AUTO_INCREMENT
family_id   INT UNSIGNED NOT NULL          -- FK families(id)
user_id     INT UNSIGNED NOT NULL          -- FK users(id)
role        ENUM('owner','parent','child') NOT NULL DEFAULT 'child'
joined_at   DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
status      ENUM('active','disabled') NOT NULL DEFAULT 'active'
UNIQUE KEY uq_family_user (family_id, user_id)
```

### 3.4. `guardianships` — опекунство (ядро провизорной логики)

```
id              BIGINT UNSIGNED PK AUTO_INCREMENT
child_user_id   INT UNSIGNED NOT NULL      -- ребёнок, FK users(id)
guardian_user_id INT UNSIGNED NOT NULL     -- родитель-опекун, FK users(id)
family_id       INT UNSIGNED NULL          -- семья, в рамках которой действует опекунство
type            ENUM('primary','provisional') NOT NULL
status          ENUM('active','severed') NOT NULL DEFAULT 'active'
source          ENUM('created','child_invite','transfer') NOT NULL
source_invitation_id BIGINT UNSIGNED NULL  -- из какого приглашения возникло
created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
severed_at      DATETIME NULL
sever_reason    VARCHAR(60) NULL           -- напр. 'real_parent_attached'
KEY idx_child (child_user_id, status), KEY idx_guardian (guardian_user_id, status)
```

Смысл: один ребёнок может иметь несколько опекунских рёбер во времени, но **активным** в каждый момент должно быть согласованное множество (минимум один). `provisional` — это «временно под родителем пригласившего», `primary` — настоящий родитель. Когда появляется настоящий родитель, провизорное ребро переводится в `severed` (с причиной), а не удаляется — **лог сохраняется навсегда**.

### 3.5. `invitations` — приглашения (с типами)

```
id              BIGINT UNSIGNED PK AUTO_INCREMENT
type            ENUM('add_child','co_parent','transfer_child','child_to_child') NOT NULL
inviter_id      INT UNSIGNED NOT NULL      -- КТО пригласил, FK users(id)
family_id       INT UNSIGNED NULL          -- в какую семью (если применимо)
target_child_id INT UNSIGNED NULL          -- для co_parent/transfer_child: над каким ребёнком даём права
email           VARCHAR(190) NULL          -- для приглашений РОДИТЕЛЕЙ (нормализованный); у child_to_child NULL
token_hash      CHAR(64)     NOT NULL UNIQUE  -- SHA-256 от случайного токена; сам токен в БД не храним
status          ENUM('pending','accepted','declined','revoked','expired') NOT NULL DEFAULT 'pending'
invited_user_id INT UNSIGNED NULL          -- КОГО пригласили (заполняется при принятии)
created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
expires_at      DATETIME NOT NULL
accepted_at     DATETIME NULL
KEY idx_inviter (inviter_id), KEY idx_type (type), KEY idx_email (email), KEY idx_status (status)
```

Типы приглашений:
- **`child_to_child`** — ребёнок зовёт ребёнка. Без email; токен раздаётся ссылкой/кодом напрямую. Принятие создаёт детский аккаунт + **провизорное опекунство** на родителя пригласившего.
- **`co_parent`** — родитель зовёт второго родителя в семью (полноправный взрослый).
- **`transfer_child`** — родитель зовёт другого родителя, **выбрав конкретного ребёнка** (`target_child_id`), чтобы передать/разделить права. Это же приглашение «прикрепляет настоящего родителя» к провизорному ребёнку.
- **`add_child`** — родитель сам создаёт детский аккаунт (по сути не «письмо», а провижининг; см. 4.2). Можно вести и без строки в `invitations`, но удобно логировать единообразно.

### 3.6. `sessions`

```
id          BIGINT UNSIGNED PK AUTO_INCREMENT
user_id     INT UNSIGNED NOT NULL          -- FK users(id)
token_hash  CHAR(64)     NOT NULL UNIQUE   -- SHA-256 от cookie-токена
created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
expires_at  DATETIME NOT NULL              -- напр. +30 дней, продлевается
last_seen   DATETIME NULL
user_agent  VARCHAR(200) NULL
```

### 3.7. `password_resets` — сброс пароля родителя по email

```
id          BIGINT UNSIGNED PK AUTO_INCREMENT
user_id     INT UNSIGNED NOT NULL          -- родитель, FK users(id)
token_hash  CHAR(64)     NOT NULL UNIQUE   -- SHA-256 от токена из письма
created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
expires_at  DATETIME NOT NULL              -- напр. +1 час
used_at     DATETIME NULL                  -- одноразовый
```

### 3.8. Журнал (`events`) — уже есть, используем как append-only

Пишем сюда: `account_created`, `first_password_set`, `login`, `logout`, `password_reset_by_parent`, `password_reset_requested`, `password_reset_done`, `invite_sent`, `invite_accepted`, `guardianship_created`, `guardianship_severed`, `family_created`, `member_disabled`. **Только мягкое удаление** где-либо в системе — историю не теряем.

## 4. Механика работы (потоки)

### 4.1. Родитель регистрируется (email + пароль) → создаётся семья

1. Открыл приложение без сессии → экран входа.
2. «Я родитель» → вводит **никнейм, email, пароль** (свой, не 1234). Сервер: проверяет уникальность email и никнейма, `password_hash()`, создаёт `users(kind=parent, must_change_password=0)`.
3. Сразу создаётся `families(owner=он)` и `family_members(role=owner)`.
4. Открывается сессия. Лог `account_created`, `family_created`.

### 4.2. Родитель создаёт детский аккаунт (никнейм + одноразовый 1234)

1. Родитель → экран **«Семья»** → «Добавить ребёнка».
2. Вводит **никнейм ребёнка** (email не просим — у детей его нет). Сервер: создаёт `users(kind=child, email=NULL, password_hash=hash('1234'), must_change_password=1, invited_by=parent)`, добавляет `family_members(role=child)`, создаёт `guardianships(type=primary, guardian=parent, source=created)`.
3. Родитель сообщает ребёнку его никнейм и что пароль — `1234`. Лог `account_created`.

### 4.3. Первый вход и обязательная смена пароля

1. Ребёнок входит: **никнейм + `1234`**.
2. Сервер видит `must_change_password=1` → пускает **только** на экран «Придумай новый пароль». Никакие данные/модули недоступны, пока пароль не сменён.
3. Ребёнок задаёт новый пароль → `must_change_password=0`. Лог `first_password_set`. Дальше обычная работа.

(Тот же механизм 1234 + смена используется после сброса пароля родителем — см. 4.6.)

### 4.4. Ребёнок приглашает ребёнка → провизорное опекунство

1. Ребёнок-A → «Позвать друга». Сервер (требует: A — активный ребёнок; по желанию — гейт родителя, см. 8.4) создаёт `invitations(type=child_to_child, inviter=A, family_id=семья A, email=NULL, token, pending, expires)` и отдаёт **ссылку/код** (email недоступен — делятся напрямую).
2. Ребёнок-B открывает ссылку → экран «Тебя позвал {никнейм A}. Придумай никнейм». Пароль — одноразовый `1234`.
3. Сервер при принятии:
   - создаёт `users(kind=child, email=NULL, password_hash=hash('1234'), must_change_password=1, invited_by=A)`;
   - определяет **ближайшего родителя A** = `primary`-опекун A (owner семьи A);
   - создаёт **провизорное** `guardianships(child=B, guardian=родитель_A, type=provisional, source=child_invite, source_invitation_id=...)` и членство B в семье A (или отдельной «гостевой» привязке — см. ниже);
   - помечает приглашение `accepted`, `invited_user_id=B`.
4. Теперь **родитель A видит B и имеет над ним все права опекуна** (B «временно без своего родителя»). B при первом входе меняет 1234. Логи `account_created`, `guardianship_created(provisional)`, `invite_accepted`.

Замечание о членстве B: чтобы не смешивать чужого ребёнка с родными данными семьи A, B можно держать как `family_member` семьи A с пометкой провизорности через `guardianships`, либо в отдельной мини-семье B, владелец которой временно — родитель A. Рекомендация: **отдельная семья B с owner = родитель A и provisional-опекунством** — тогда «разрыв» (4.5) чистый: меняем owner/опекуна, не трогая семью A. Финальный выбор — открытый вопрос 10.3.

### 4.5. Передача прав настоящему родителю (разрыв провизорной связи)

1. Родитель A (текущий провизорный опекун B) → карточка B → «Привязать родителя» → вводит **email** настоящего родителя B. Сервер создаёт `invitations(type=transfer_child, inviter=родитель_A, target_child_id=B, email=..., token, pending)` и отправляет письмо со ссылкой (или отдаёт ссылку вручную, если почта ещё не настроена).
2. Настоящий родитель открывает ссылку → регистрируется как `parent` (email + свой пароль).
3. В момент создания его аккаунта сервер:
   - создаёт **новую нормальную семью** для B (owner = новый родитель), переносит членство B туда;
   - создаёт `guardianships(child=B, guardian=новый_родитель, type=primary, source=transfer)`;
   - **рвёт провизорную связь навсегда**: старое `guardianships(provisional, guardian=родитель_A)` → `status=severed, severed_at, sever_reason='real_parent_attached'` (строка остаётся — это лог);
   - данные B (всё по `user_id=B`) остаются с ним, просто теперь их видит новый родитель, а не родитель A.
4. Логи `family_created`, `guardianship_created(primary)`, `guardianship_severed(provisional)`, `invite_accepted`. С этого момента родитель A ребёнка B **не видит** (кроме факта в истории).

Если родитель A так и не привязал настоящего родителя — B остаётся под провизорным опекунством A неограниченно (это допустимо по требованиям).

### 4.6. Сбросы паролей

- **Ребёнок забыл пароль** → сам сбросить **не может**. Его **опекун-родитель** на карточке ребёнка жмёт «Сбросить пароль» → пароль ребёнка снова `1234`, `must_change_password=1`. Ребёнок входит по 1234 и задаёт новый. Логи `password_reset_by_parent`, затем `first_password_set`.
- **Родитель забыл пароль** → «Забыли пароль?» → вводит email → сервер создаёт `password_resets` (одноразовый токен, +1 час) и **шлёт письмо** со ссылкой. По ссылке родитель задаёт новый пароль. Ответ всегда одинаковый («если такой email есть, мы отправили письмо») — не раскрываем существование аккаунта. Логи `password_reset_requested`, `password_reset_done`.
- Итог соответствует требованию: **дети сбрасываются только через родителя; родитель — только по email** (а email есть только у родителей).

### 4.7. Семья и шаринг (родитель видит детей)

- Родитель/owner видит своих детей (по активным `guardianships`) и **только на чтение** их данные и статистику.
- **Два уровня видимости (важно для картинок):**
  - **Статистика и активность** (счётчики, серии, события чистки зубов, списки и т.п.): видит любой активный опекун ребёнка и parent/owner той же семьи. Это `rt_can_read(target)`.
  - **Картинки виш-листа (изображения)**: видит **ТОЛЬКО сам ребёнок и его ПРЯМОЙ родитель** (активное `guardianships.type='primary'`). **Временный (provisional) опекун изображения НЕ видит никогда.** Для этого отдельная проверка `rt_can_view_images(target)` = (запрашивающий — сам ребёнок) ИЛИ (активный **primary**-опекун ребёнка). Provisional → всегда false для изображений.
- Картинки **нешарабельны**: нет публичных ссылок, шаринга, экспорта; раздаются только владельцу и прямому родителю через защищённый доступ.
- Реализация — `rt_can_read(target_user_id)` для статистики (любой опекун/родитель семьи), и более строгий `rt_can_view_images(target_user_id)` для изображений (только сам ребёнок + primary-родитель). Запись остаётся у владельца данных.
- Это и есть будущий **родительский дашборд** (чистка зубов, активность) из `events` и `module_data`; картинки виш-листа в дашборде показываются прямому родителю и скрыты у временного опекуна.

### 4.8. Управление и жизненный цикл приглашений

```
pending ──accept──► accepted
pending ──decline─► declined
pending ──revoke──► revoked
pending ──время──► expired
```

Переходы из `pending` терминальны (повторно — новое приглашение/«Переслать» со свежим токеном). «Отключить члена» — мягко (`status=disabled`), данные и логи сохраняются. «Выход» — удалить сессию; «со всех устройств» — все сессии пользователя.

### 4.9. Управление аккаунтом ребёнка и блокировка (решение заказчика 2026-06-05)

На экране «Управление аккаунтом» ребёнка доступны: сброс пароля (4.6), привязка настоящего родителя (4.5) и **блокировка/разблокировка**.

- **Кто может блокировать.** Заблокировать или разблокировать детский аккаунт может **ЛЮБОЙ родитель или опекун** этого ребёнка, включая **временного (provisional)** опекуна (он отвечает за гостя). Не требуется быть владельцем семьи.
- **Что делает блокировка.** Ставит `accounts.status = 'disabled'` у ребёнка. Заблокированный ребёнок **не может войти**: на экране входа он видит нейтральное «аккаунт отключён» (тот же экран, что 8.x/edge). **Данные не удаляются**, всё сохраняется; разблокировка возвращает `status = 'active'`.
- **Подтверждение.** Блокировка — действие в два шага (экран подтверждения), как разрушающее, но обратимое.
- **Логи.** `account_blocked` и `account_unblocked` пишутся в `events` с указанием, кто и когда (детская подотчётность).
- **Бэкенд (к реализации).** Новый эндпоинт, например `accounts.php op=set_child_status {child_id, status}`; гейт — `rt_can_manage_child(child_id)` = запрашивающий является активным опекуном ребёнка (primary ИЛИ provisional) или parent/owner его семьи. Это шире, чем доступ к изображениям (которые видит только сам ребёнок и primary-родитель, см. §4.7), но логично: за безопасность гостя отвечает и временный опекун, поэтому блокировать он может.
- Прототип: сценарии «Управление аккаунтом ребёнка», «Заблокировать ребёнка», «Разблокировать ребёнка» в группе «Управление аккаунтами» (`family-prototype.html`).

### 4.10. PIN администратора упраздняется (решение заказчика 2026-06-05)

- Старый механизм `admin_pin` (PIN в `config.php`, гейт «магазина приложений» и родительских панелей модулей) **уходит**. Его заменяет **родительский аккаунт**: управление приложениями и админ-действия доступны тому, кто вошёл как родитель/опекун. Без PIN.
- К реализации (после одобрения): эндпоинты `store/*` и прочие места с `rt_admin_ok($pin)` переводятся на `rt_require_parent()` (родительская сессия). `sdk.admin.verify(pin)` в модулях заменяется проверкой роли текущей сессии. `admin_pin` в `config.php` остаётся только как аварийный fallback на переходный период, затем удаляется.
- Родительские панели модулей (например, «Таймер чистки зубов») также перестают спрашивать PIN: панель видна, если текущая сессия — родитель.
- Страница настроек приложений (что включено, порядок, установка) проектируется отдельно: в прототипе приложения (`app-prototype.html`) пока заглушка, сама страница — следующая итерация после ревью заказчика.

## 5. Отправка писем (сброс пароля родителя и приглашения родителей)

Письма нужны **только для родительских сценариев** (сброс пароля родителя, `co_parent`/`transfer_child`). Дети писем не получают и почты не имеют — это держит детскую часть полностью офлайн по PII.

**Не использовать** голую `mail()` в PHP: нет DKIM, плохая доставляемость, почти гарантированный спам.

Рекомендуемый стек:
- **Библиотека отправки:** **PHPMailer** (простая, проверенная, идеально для PHP на Hostinger) или **Symfony Mailer** (современнее, удобные транспорты/провайдеры). Для прототипа — PHPMailer.
- **Транзакционный провайдер** (он берёт на себя DKIM-подпись, репутацию IP, очереди, повторы): на выбор **Amazon SES** (дёшево, надёжно), **Postmark** (отличная доставляемость транзакционных писем), **Brevo/Sendinblue** или **Mailgun**/**Resend**. Шлём через их **SMTP или API**, ключи держим в `config.php` (не в Git).
- **DNS-подписи домена** `tilley.live` (без них письма летят в спам):
  - **SPF** — TXT-запись, разрешающая провайдеру слать от имени домена.
  - **DKIM** — провайдер даёт CNAME/TXT-ключ(и); добавляем в DNS, провайдер подписывает каждое письмо.
  - **DMARC** — TXT `_dmarc.tilley.live` с политикой (начать с `p=none` для мониторинга, затем `quarantine`/`reject`) и адресом отчётов.
  - Отправитель — выделенный адрес, напр. `noreply@tilley.live`, и/или поддомен `mail.tilley.live` для изоляции репутации.
- **Содержимое письма** минимальное: ссылка с токеном и срок действия. Никаких данных детей. Текст на русском.
- **На переходный период** (пока DNS/провайдер не настроены): `co_parent`/`transfer_child` работают как **ссылка для пересылки вручную** (родитель копирует ссылку). Сброс пароля родителя по email требует рабочей почты — до неё запасной путь: сброс через второго родителя/owner или через админа (`admin_pin`). Это отражено в открытых вопросах.

Безопасность писем: токены сброса одноразовые и короткоживущие (≤1 ч), в БД только хэш; единый ответ «письмо отправлено» без раскрытия наличия аккаунта; лимит частоты запросов сброса.

## 6. Архитектура API

Стиль как сейчас: тонкие PHP-файлы поверх `_bootstrap.php`/`_db.php`, JSON, авто-миграции; новые папки `api/auth/`, `api/family/`, `api/account/` закрыть `.htaccess` как внутренние.

```
api/auth/register_parent.php  POST {nickname,email,password}        → создаёт родителя (+семью), сессия
api/auth/login.php            POST {login,password}                 → login = email(родитель) или nickname(ребёнок) + пароль
api/auth/logout.php           POST {}                               → удалить сессию
api/auth/me.php               GET                                   → текущий: user(kind,nickname,role), семья, дети, must_change
api/account/set_password.php  POST {new_password}                   → смена пароля (в т.ч. обязательная по must_change)
api/account/forgot.php        POST {email}                          → родителю: создать reset-токен, послать письмо (единый ответ)
api/account/reset.php         POST {token,new_password}             → сброс пароля родителя по токену из письма

api/family/add_child.php      POST {nickname}                       → родитель создаёт ребёнка (пароль=1234, must_change)
api/family/reset_child.php    POST {child_id}                       → родитель сбрасывает пароль ребёнка → 1234+must_change
api/family/invite.php         POST {type,email?,target_child_id?}   → co_parent | transfer_child | child_to_child → ссылка/письмо
api/family/invites.php        GET | POST {action:revoke|resend,id}  → список/отзыв/пересылка
api/family/accept.php         POST {token, nickname?,password?}     → принять (создать ребёнка / родителя / привязать)
api/family/members.php        GET                                   → члены семьи + опекунства (для дашборда)
```

Точка интеграции: **`rt_user_id()`** в `_bootstrap.php` перестаёт быть константой и читает сессию из cookie → `sessions` → `user_id` (то самое «единственное место» из комментария в файле). Добавляются помощники: `rt_current_user()`, `rt_require_login()`, `rt_require_parent()`, `rt_guardian_of($childId)`, `rt_can_read($targetUserId)` (статистика/активность) и `rt_can_view_images($targetUserId)` (строже: только сам ребёнок или primary-родитель; provisional-опекуну изображения недоступны).

## 7. Переход с однопользовательского режима (без поломок)

1. **Флаг** `single_user => true|false` в `config.php`. Пока `true` и нет сессии — `rt_user_id()` возвращает `1` (Артём), всё текущее работает как сейчас.
2. **Сид**: миграция создаёт семью, родителя (`users.id=2`, `kind=parent`) делает owner, Артёма (`id=1`, `kind=child`) — child + `guardianships(primary)`. Существующие данные уже на `user_id=1` — переносить нечего. Родителю при первом реальном входе зададим email/пароль.
3. Когда вход и «Семья» готовы и протестированы — `single_user => false`.
4. **Демо-режим** клиента (localStorage, офлайн) остаётся для file:// и не требует аккаунтов.

## 8. Безопасность (особенно детская)

### 8.1. Вход и идентификаторы
- Родитель входит по **email + пароль**, ребёнок — по **никнейм + пароль**. Поэтому никнейм глобально уникален (case-insensitive). Альтернатива — выбор семьи на устройстве + никнейм в её рамках (открытый вопрос 10.1).
- Единые сообщения об ошибке входа (не раскрываем, что именно неверно).

### 8.2. Пароли и одноразовый 1234
- Хэширование только `password_hash()` (bcrypt/argon2id), никогда не храним/не логируем пароль.
- **Одноразовый `1234`** — это сознательный компромисс ради простоты для детей. Митигации обязательны: аккаунт с `must_change_password=1` **не имеет доступа ни к чему**, кроме экрана смены пароля; войти по 1234 можно, только зная никнейм; лимит попыток; после смены 1234 больше не действует. Опция на будущее: вместо фиксированного `1234` — случайный 4-значный код, который видит родитель (сильнее, но менее предсказуемо для ребёнка). По умолчанию — `1234`, как просил заказчик.

### 8.3. Сессии и токены
- Cookie `HttpOnly`, `Secure`, `SameSite=Lax`; в БД только `token_hash`; срок + продление; «выйти со всех устройств».
- Токены приглашений и сбросов: CSPRNG ≥32 байта, в БД только SHA-256, одноразовые, истекающие; единый ответ без раскрытия наличия аккаунта; лимит частоты.
- **CSRF**: для изменяющих запросов — заголовок-токен или строгий `SameSite`.

### 8.4. Детская специфика
- **Приглашать детей могут дети, но это можно (и стоит) гейтить родителем.** Требование разрешает детям звать детей; для безопасности предлагаю опцию «приглашение ребёнка требует подтверждения опекуна» (включается родителем). Решение — открытый вопрос 10.4.
- **Провизорный опекун несёт полную ответственность** за приглашённого ребёнка, пока не привязан настоящий родитель. Родитель видит таких детей отдельной пометкой «гость, временно под вами». **НО изображения ребёнка временный опекун не видит никогда** (см. ниже).
- **Картинки ребёнка: приватны, нешарабельны, видны только ребёнку и ПРЯМОМУ родителю.** Дети могут загружать картинки (виш-лист), но они нигде не публикуются и не пересылаются. Доступ к изображениям — отдельная, более строгая проверка `rt_can_view_images(target)`: только сам ребёнок ИЛИ активный **primary**-опекун. **Provisional-опекун — всегда нет.** Это инвариант детской безопасности; в UI/письмах формулировать аккуратно, без переобещаний (раздел 1).
- **Только никнеймы, никакого PII у детей** (раздел 1). Email детям не присваивается ни при каких сценариях.
- **Логи сохраняются** (append-only): кто кого пригласил, кто сбросил пароль, когда разорвана провизорная связь. Это требование детской подотчётности и помогает при разборе инцидентов.
- Нормы о данных детей (COPPA/GDPR-K): родительское согласие, право удалить данные ребёнка. Это не юридическая консультация — перед публичным запуском свериться с юристом.

### 8.5. Авторизация
- На каждом эндпоинте проверяем права: создавать/сбрасывать ребёнка и приглашать родителей может только `parent`/`owner`; читать статистику/активность ребёнка — через `rt_can_read()`/активное опекунство; **отдавать изображения ребёнка — только через `rt_can_view_images()` (сам ребёнок или primary-родитель; provisional — запрещено)**. Эндпоинт выдачи картинок (например, `upload.php`/раздача файла) обязан вызывать `rt_can_view_images()` перед отдачей и не давать публичных/прямых ссылок.
- **Блокировка ребёнка** — через `rt_can_manage_child()` (любой активный опекун ребёнка, включая provisional, или parent/owner его семьи). Ставит `accounts.status='disabled'`; разблокировка возвращает `active`; оба действия в логах (`account_blocked`/`account_unblocked`). См. §4.9.

## 9. Граф и инварианты (самопроверка модели)

- Ребро «кто кого пригласил»: `users.invited_by` + `invitations(inviter_id → invited_user_id)`.
- Ребро «кто над кем опекун»: `guardianships(child ← guardian)`, тип `primary`/`provisional`, со статусом.
- Инварианты: у активного ребёнка ≥1 активного опекуна; `child ⇒ email IS NULL`; провизорное опекунство рвётся (не удаляется) при появлении `primary` от настоящего родителя; никнейм уникален; письма только родителям.
- Всё разрушающее — мягко (`severed`/`disabled`/`deleted_at`), чтобы **логи сохранялись**.

## 10. Открытые вопросы (решить до кода)

1. **Уникальность никнейма** — глобально (просто для входа ребёнка) или в пределах семьи + выбор семьи на устройстве (приватнее, но сложнее вход). Рекомендую глобально для прототипа.
2. **Почтовый провайдер** — какой берём (SES / Postmark / Brevo / Mailgun / Resend) и есть ли доступ к DNS домена `tilley.live` для SPF/DKIM/DMARC. До настройки родительские приглашения работают ссылкой вручную; сброс пароля родителя по email без почты невозможен (нужен запасной путь — второй owner/админ).
3. **Членство провизорного ребёнка** — отдельная мини-семья с owner=провизорный родитель (рекомендую, чистый разрыв) против пометки внутри семьи пригласившего.
4. **Гейт на детские приглашения** — разрешать детям звать детей свободно или требовать подтверждение опекуна (рекомендую опциональный гейт, по умолчанию включён для младших).
5. **Объём шаринга** — только чтение статистики (да) + нужен ли родителю доступ на запись в некоторые модули (сейчас это делает родительская панель по `admin_pin`).
