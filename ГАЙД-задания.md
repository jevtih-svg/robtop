## 1. Зачем это и главный инвариант

Задания от родителей — это **ресурс уровня приложения**, как очки и оповещения, а НЕ данные одного модуля. Их показывают ДВА места: модуль «Задания» (полноэкранный список) и блок «Задания» на вкладке «Родители» в Копилке. Требование Джеффа (2026-06-08): «унаследовать логику из Копилки, НЕ заменяя её; они зеркалят друг друга — поставил в Копилке, видно в Задачах».

Инвариант: **один источник правды**. Оба UI зовут ТОЛЬКО `sdk.tasks` и своей логики заданий не держат. Поэтому они синхронны автоматически: создал/выполнил/удалил в любом — другой показывает то же (вживую ≤4с через sync-поллинг).

История: до 2026-06-08 задания жили в generic-сторе как коллекция `bank/tasks` (`module_data`) и «принадлежали» Копилке — переиспользовать было нельзя. Перенесли в отдельную таблицу `tasks`; старые строки переносит ленивый бэкфилл (§5).

## 2. Где живёт (отдельная таблица `tasks`, миграция 024)

| Поле | Тип | Смысл |
|---|---|---|
| `id` | BIGINT | первичный ключ |
| `user_id` | INT | ребёнок-владелец (скоуп) |
| `title` | VARCHAR(120) | название задания |
| `points` | INT | очков за выполнение, 1..1000 (деф. 10) |
| `type` | `recur` \| `once` | повторяющееся / одноразовое |
| `status` | `active` \| `pending` \| `done` | состояние (см. §4) |
| `origin` | `parent` \| `child` | КТО создал: родитель назначил / ребёнок предложил (миграция 025) |
| `times_done` | INT | сколько раз подтверждено (для recur) |
| `last_done_at`, `claimed_at`, `done_at` | DATETIME | вехи |
| `created_by` | INT | кто создал (родитель или ребёнок-автор предложения) |
| `created_at`, `updated_at`, `deleted_at` | DATETIME | + мягкое удаление |

`pending` различается по `origin`: `origin=parent` — проверка выполнения задания, `origin=child` — ПРЕДЛОЖЕНИЕ ребёнка (он залогировал дело со своей оценкой очков, ждёт ревью). НЕ `module_data`: членство/скоуп проверяет сервер, переходы статусов условные (гонка устройств), и это «вызывается откуда угодно» (требование Джеффа). Аддитивность — жёсткий инвариант: старые строки `bank/tasks` в `module_data` НЕ трогаем, они становятся неживой историей.

## 3. Поверхность `sdk.tasks` (core/sdk.js)

```js
sdk.tasks.list()                 // → Promise<Task[]>  (живые задания скоупа, свежие сверху)
sdk.tasks.create({title,points,type})   // родитель: новое задание (origin=parent, active). Шлёт task_new
sdk.tasks.propose({title,points})        // РЕБЁНОК: залогировать дело (origin=child, once, pending). Шлёт task_proposed
sdk.tasks.update(id,{title?,points?,type?})  // родитель: правка (статусы двигают только потоки)
sdk.tasks.remove(id)             // родитель: мягкое удаление
sdk.tasks.claim(task)            // ребёнок «Сделал!» → {ok}  (ВСЕГДА на проверку родителю)
sdk.tasks.approve(task, points?) // родитель: подтвердить (+опц. поправка очков) → {ok, streak|null, bonus, points}
sdk.tasks.decline(task)          // родитель: вернуть проверку выполнения в active → {ok}
sdk.tasks.deny(task)             // родитель: отклонить предложение ребёнка (исчезает) → {ok}
```

Контракт `Task` (плоский, как отдаёт сервер): `{ id, title, points, type:"recur"|"once", status:"active"|"pending"|"done", origin:"parent"|"child", timesDone, lastDoneAt, claimedAt, doneAt, createdAt, updatedAt }` (таймстампы — ms или null).

Все методы — fire-and-forget-устойчивые: возвращают `{ok:false}` вместо reject. Очки (через `sdk.points`) и оповещения (`ntf.ev.tasks.*`) шлёт САМ движок внутри `claim`/`approve`/`propose`/`decline`/`deny` — UI их НЕ дублирует.

## 4. Потоки и статусы (рефактор 2026-06-08: универсальное подтверждение + предложения)

```
ЗАДАНИЕ РОДИТЕЛЯ (origin=parent):
     create ─► active ── ребёнок «Сделал!» (claim) ─► pending ──┬─ ✓ approve ─► recur: active (+очки, ×N)
                  ▲                                              │                once:  done  (+очки)
                  └──────────── ↩ decline (без очков) ───────────┘

ПРЕДЛОЖЕНИЕ РЕБЁНКА (origin=child):
     propose ─► pending ──┬─ ✓ approve (можно поправить очки) ─► done (+очки)
                          └─ ✕ deny ─► исчезает (мягкое удаление)
```

Правила (решения Джеффа 2026-06-08):
- **универсальное подтверждение**: «Сделал!» по ЛЮБОМУ заданию родителя (и `once`, и `recur`) → `pending`. Очки — только после `approve` (раньше `once` давал очки сразу; теперь нет — родитель всегда в контроле).
- **повторяющееся** (`recur`): approve → очки `kind=task_done` + назад в `active`, `times_done+1`. once → approve → очки + `done`.
- **↩ decline** (проверка выполнения): назад в `active`, без очков и без штрафа (оповещение `task_returned`).
- **предложение ребёнка** (`propose`): «залогировал дело» с предложенными очками → `pending`/`origin=child`/`once`. Родитель `approve(task, points?)` — может ПОПРАВИТЬ очки → `done` + очки; либо `deny` → исчезает. Тип предложения — всегда `once` (v1; повторяющиеся предложения — на будущее).
- **сумма** у каждого своя, дефолт +10 (канон ГАЙД-очки.md §4).
- **порядок «сначала очки, потом статус»** — защита награды: если `sdk.points.add` не прошёл, статус не двигаем.
- переходы на сервере условные (`WHERE status/origin=…`): гонка устройств → **409**, клиент перечитывает (живое обновление).

## 5. Сервер: `api/tasks.php` + `_tasks.php` (переиспользуемое)

`api/tasks.php` — операции `list/create/propose/update/delete/claim/approve/decline/deny`. `create` — только родитель (origin=parent); `propose` — только ребёнок (origin=child, type=once, pending). `approve` принимает опц. `points` (поправка для предложений). `decline` — только `origin=parent` (вернуть в active); `deny` — только `origin=child` (мягкое удаление). Скоуп как в `data.php`: ребёнок видит свои; родитель — выбранного на дашборде ребёнка (`child=<id>`, проверка `rt_can_manage_child`), без `child` — первый ребёнок семьи. События пишутся в `events` под РЕБЁНКОМ (`module='tasks'`). Очки сервис НЕ начисляет (канон — только `sdk.points` на клиенте).

`api/_tasks.php` — ПЕРЕИСПОЛЬЗУЕМАЯ часть (require_once из `tasks.php` И `parent.php`):
- `rt_task_clean_title/points/type` — валидация полей;
- `rt_tasks_backfill($db,$uid)` — ленивый перенос живых строк `bank/tasks` (module_data) → таблица `tasks`. Идемпотентно: если в `tasks` уже есть строка пользователя, выходим; старые строки `module_data` не трогаются. Срабатывает при первом `list` скоупа и в счётчике дашборда.
- `rt_tasks_pending_count($db,$uid)` — сколько заданий `pending` (для бейджа дашборда; внутри вызывает бэкфилл, чтобы счётчик не соврал, пока ребёнок не открыл задания после деплоя).

«Вызвать задания откуда угодно» = подключить `_tasks.php` и читать таблицу `tasks` / звать `rt_tasks_*`. Пример уже есть: `parent.php` отдаёт `tasksPending`.

## 6. Живое обновление (sync)

`api/sync.php` включает таблицу `tasks` в отпечаток `data` (COUNT + MAX(id) + MAX(updated_at) по скоупу uid зрителя, включая детей родителя). Любой оп `tasks.php` бьёт `updated_at` → оба UI и дашборд подхватывают изменение ≤4с. Оба модуля (tasks и bank) реализуют хук `refresh()` по контракту: занят (своя операция `busy` или открытая шторка) → вернуть `false`, shell повторит следующим тиком (изменение не теряется).

## 7. Оповещения (двусторонние)

Шлёт движок `sdk.tasks`, src `"tasks"`, link `{module:"tasks"}`. Ключи в `core/notify.js` (en/ru/lv):
- `task_new` — родитель → ребёнку: новое задание;
- `task_claim` — ребёнок → родителям: «сделал, проверь»;
- `task_approved` — родитель → ребёнку: выполнение подтверждено (+очки);
- `task_returned` — родитель → ребёнку: выполнение возвращено (попробуй ещё);
- `task_proposed` — ребёнок → родителям: предложил дело (+предложенные очки, проверь);
- `task_proposal_ok` — родитель → ребёнку: предложение одобрено (+очки, возможно поправленные);
- `task_proposal_no` — родитель → ребёнку: предложение отклонено.

Блок `ntf.ev.bank.*` оставлен для СТАРЫХ сохранённых строк (src `"bank"`) — не удалять.

## 8. Демо-режим

`file://` / без сервера: `sdk.tasks` работает через `demoData("tasks","items")` в localStorage (как `sdk.data`/`sdk.points`). Оба UI видят один список; оповещения — no-op. Песочница без гейта ролей (`isParent()` = true в демо): показывается РОДИТЕЛЬСКИЙ вид, открыты все действия — детский вид (propose/claim) проверяется только на не-демо пути.

## 9. Подключение заданий в НОВОМ месте (рецепт)

Хочешь показать или поставить задания из ещё одного модуля/экрана:
1. Клиент: зови `sdk.tasks.*` (НЕ `sdk.data`, НЕ свой fetch). Очки и оповещения получишь бесплатно.
2. Сервер (если нужен счётчик/чтение из PHP): `require_once __DIR__.'/_tasks.php'` и читай таблицу `tasks` или зови `rt_tasks_*`.
3. НЕ заводи второй источник правды и не пиши в `bank/tasks` (module_data) — он мёртв.
4. Реализуй хук `refresh()` в модуле, чтобы изменения с других устройств доезжали.

## 10. Чего НЕ делать

- Не писать задания в `module_data` (`bank/tasks`) — только таблица `tasks` через сервис.
- Не начислять очки в обход `sdk.points` (канон ГАЙД-очки.md §8).
- Не гейтить сервис заданий включённостью плитки: выключенная плитка «Задания» или «Копилка» не должна ломать задания в соседнем UI (поэтому `tasks.php` не проверяет `modules.enabled`).
- Не дублировать оповещения в UI — их шлёт движок.
- Не двигать статусы напрямую UPDATE'ом из UI — только потоки `claim/approve/decline`.
