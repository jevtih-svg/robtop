# АНАЛИЗ: перенос RobTop на iOS (нативный Swift)

> Аналитический документ по запросу Джеффа, 2026-06-07. Код приложения не меняет.
> Версия приложения на момент анализа: v2026.06.07.63 (16 модулей в реестре, 12 готовы).

---

## 1. Краткий ответ

Перенести можно. Технических блокеров нет: сервер (PHP + MySQL) переиспользуется целиком, у каждого веб-механизма есть нативный аналог. Цена вопроса: переписывание всего клиента заново (порядка 15 тыс. строк, 12 модулей), ориентировочно 3 месяца работы на паритет, затем постоянная двойная поддержка и потеря мгновенного деплоя. При этом приложение уже сегодня живёт на iPhone как PWA: иконка на домашнем экране, полный экран, push-уведомления, бейджи. Рекомендация: §9.

### Темп разработки (важный фактор для выводов)

87 коммитов за 4 дня; за один день 2026-06-07 выпущено 63 номера версии, параллельными сессиями. Такой темп возможен только потому, что деплой = git push, и через минуту новая версия у всех устройств без каких-либо действий.

## 3. Сервер остаётся: главная хорошая новость

API уже устроен как чистый JSON поверх HTTP, без серверного рендеринга HTML. Сессия живёт в cookie `rt_sess` (30 дней): URLSession на iOS поддерживает cookie из коробки (HTTPCookieStorage), значит вход по email/нику + паролю заработает без переделки сервера. Универсальный стор `data.php` (list/create/update/move/...) одинаково удобен и из Swift.

Единственная обязательная серверная доработка: рассылка пушей в APNs (Apple Push Notification service). Сейчас `api/_push.php` умеет только Web Push по VAPID. Для натива нужны: таблица device-токенов, подпись JWT ES256 для APNs (тот же openssl-приём, что уже использован для VAPID) и отправка на api.push.apple.com. По сложности сопоставимо с уже сделанным Web Push.

## 4. Карта соответствия технологий

| Сейчас (веб) | В нативном iOS | Оценка |
|---|---|---|
| SPA на чистом JS, плитки, шторки | SwiftUI (NavigationStack, sheets) | переписывание, прямых аналогов UI много |
| fetch + JSON | URLSession + Codable | просто |
| Cookie-сессия rt_sess | HTTPCookieStorage, работает из коробки | просто |
| localStorage (префы, демо) | UserDefaults | просто |
| canvas: Змейка, фон-плата | SpriteKit или SwiftUI Canvas | средне, игра простая |
| Web Speech TTS («Слова наоборот») | AVSpeechSynthesizer | просто, голоса станут лучше |
| AudioContext + mp3 (зубы, музыка) | AVFoundation | просто |
| FileReader + ресайз фото + upload.php | PHPicker / камера + тот же upload.php | просто |
| Web Push (VAPID, sw.js) | APNs + доработка сервера (§3) | главная серверная работа |
| Поллинг sync.php каждые 4 с | тот же поллинг + тихие пуши в фоне | просто |
| i18n словари en/ru/lv | String Catalog или перенос своего движка с теми же ключами | объёмно, механически |
| Темы (CSS-токены neon/tilley) | набор токенов в SwiftUI | средне |
| navigator.vibrate | Core Haptics | станет лучше |
| navigator.setAppBadge | нативный бейдж приложения | станет надёжнее |
| Реордер long-press + jiggle | нативный drag and drop | проще, чем было в вебе |
| Анти-зум, борьба с кэшем, Safari-причуды | не нужны вовсе | целый класс багов исчезает |

Вывод по таблице: непереносимых вещей нет.

## 5. Что переносится без потерь

Данные, аккаунты, семьи, роли, очки, оповещения: всё это серверное, клиент только отображает. История и статистика остаются в той же БД, ребёнок ничего не теряет. Ключи i18n уже отвязаны от движка и переносятся как есть. Фото-пайплайн (ресайз на клиенте, upload.php, пути uploads/) работает с нативным клиентом без изменений. КОНТЕКСТ.md и ГАЙДы фактически являются готовой спецификацией нативного клиента: это сильно удешевляет переписывание.

## 6. Что ломается или меняется концептуально

### 6.1. Динамическая загрузка модулей

Ключевой принцип архитектуры («подкинул папку модуля на сервер, появилась плитка») в нативе не существует: правила Apple (Review Guideline 2.5.2) запрещают загружать и исполнять нативный код по сети. Все модули компилируются внутрь приложения, «установка из магазина» превращается в тумблер показа уже встроенного модуля. На практике проект и так живёт на встроенных modules/ (в каталоге apps/_catalog один тестовый dice), поэтому потеря терпимая, но рантайм-установку придётся похоронить официально.

### 6.2. Темп доставки обновлений

Сегодня цикл «правка → у всех устройств» занимает минуты. В нативе: сборка Xcode, подпись, выгрузка в TestFlight, обработка Apple (минуты, иногда десятки минут), установка на каждом устройстве вручную. Реалистичный цикл 20-40 минут при настроенном CI. 63 итерации в день физически невозможны. Для стиля этого проекта (быстрые правки по фидбеку Артёма в тот же час) это самое болезненное последствие переноса.

### 6.3. Двойная поддержка навсегда

Веб не умирает: admin.html, w.html (бабушки открывают ссылку без установки чего-либо), family.html (посадочные из писем). Если хоть один член семьи окажется без устройства Apple, детский клиент тоже придётся держать в вебе. Тогда каждая новая функция пишется дважды (Swift и JS), тестируется дважды, и журнал изменений раздваивается. Опыт проекта: 12 модулей за 4 дня; при двойной разработке темп упадёт кратно.

### 6.4. Распространение и формальности

Нужен Apple Developer Program: $99 в год, обязателен для APNs-пушей и TestFlight. Для семьи правильный канал: TestFlight (сборка живёт 90 дней, затем требуется обновление). Публикация в App Store не нужна и нежелательна: детское приложение попадает в Kids Category с жёсткими требованиями (privacy policy, parental gate, ограничения сбора данных, ревью каждого обновления). Для личного приложения одной семьи это лишние обязательства.

### 6.5. Инфраструктура разработки

Понадобятся Mac + Xcode, сертификаты и provisioning-профили, CI на macOS-раннерах (в GitHub Actions они тарифицируются в 10 раз дороже Linux) либо Xcode Cloud. Песочница Cowork не собирает iOS-проекты: проверка каждой правки потребует ручных шагов на машине Джеффа (сборка, симулятор, прогон на устройстве). Автономность текущего процесса (агент сам проверил jsdom-смоуком и закоммитил) частично теряется.

### 6.6. Что станет лучше

Плавность анимаций и скролла, хаптика, качество TTS-голосов, надёжность пушей и бейджей, отсутствие борьбы с зумом/кэшем/Safari. Плюс открываются возможности, недоступные вебу: виджеты на домашний экран, Live Activities (таймер зубов на Dynamic Island), HealthKit-шагомер для Прогулки, геолокация в фоне, Game Center. SwiftUI почти бесплатно даёт версию под iPad.

## 8. Три стратегии

### А. Оставить как есть: PWA (усилия: ноль)

Уже сделано и работает: standalone-режим, иконка, safe-area, анти-зум, Web Push с iOS 16.4 из установленной PWA, бейджи на иконке, отключённый pull-to-refresh. По ощущениям на iPhone это уже почти нативное приложение, что и было целью многих фиксов в журнале.

### Б. Гибрид: тонкая Swift-обёртка WKWebView (усилия: 2-4 дня + $99/год)

Маленькое настоящее iOS-приложение: WKWebView на боевой URL, нативные APNs-пуши через мост JS↔Swift, сплеш-экран, иконка, при желании хаптика. 95%+ кода переиспользуется, обновления контента остаются мгновенными (для веб-контента внутри WKWebView это разрешено правилами). Минусы: в App Store чистую обёртку не пропустят (guideline 4.2 про минимальную функциональность), но для TestFlight это не имеет значения; Web Push внутри WKWebView не работает, пуши только через APNs-мост. Прирост относительно PWA скромный: надёжнее доставка пушей и «настоящая» установка.

### В. Полный натив SwiftUI (усилия: 3-4 месяца + двойная поддержка)

Максимальное качество и доступ к возможностям ОС (§6.6) по максимальной цене (§6.2-6.5, §7).

## 9. Рекомендация простым языком

Сейчас полный перенос не оправдан. Приложение уже стоит на iPhone Артёма, присылает пуши и выглядит как нативное, а главная сила проекта (десятки улучшений в день по живому фидбеку) при переносе погибнет первой. Сервер в любом случае остаётся прежним, значит решение обратимо: натив можно начать в любой момент позже, ничего не потеряв.

Если захочется шага вперёд без больших затрат: вариант Б (обёртка с нативными пушами) за несколько дней. Полный Swift (вариант В) стоит запускать только при появлении настоящей причины: полноценный офлайн, шагомер/HealthKit для Прогулки, виджеты, Live Activities, либо желание распространять приложение шире одной семьи.

## 10. Если натив однажды делать: что уже играет в плюс

Проект уже устроен «правильно» для будущего переноса, и это стоит сохранять: вся бизнес-логика на сервере (очки, права, оповещения считает PHP, клиент только рисует); единый стор data.php вместо россыпи эндпоинтов; ключи i18n отвязаны от движка; контракт модуля описан в ГАЙДе; КОНТЕКСТ.md фиксирует каждое поведение. Любая будущая нативная команда (или агент) получит готовое ТЗ. Единственное пожелание на будущее: не наращивать сложную логику в клиентском JS модулей, серверная логика переносится бесплатно, клиентская переписывается.
