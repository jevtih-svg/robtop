# Мастер-админка RobTop — спецификация и эксплуатация

Реализована 2026-06-07. Файлы: `app/admin.html` (консоль, RU), `app/api/admin.php` (все операции),
`app/api/migrations/008_admin.sql` (таблицы `admins`, `bans`), хелперы в `app/api/_accounts.php`
(`rt_is_admin`, `rt_require_admin`, `rt_email_banned`, `rt_family_banned`).
Прототип-референс дизайна: `admin-prototype.html` (в корне репо; реализация его заместила).

## 1. Доступ

- Адрес: `https://apps.tilley.live/robtop/app/admin.html` (закрыт от индексации `noindex`).
- Вход обычным РОДИТЕЛЬСКИМ аккаунтом (email + пароль, та же сессия `rt_sess`, что и приложение).
- Право даёт запись в таблице `admins` (status='active'). Проверка на КАЖДОЙ операции: `rt_require_admin` =
  активная родительская сессия + флаг (+ не в состоянии must_change_password).
- Выдать/снять права может только действующий админ: раздел «Пользователи», кнопка 👑 (только родителям).
  Последний активный админ не может снять права сам с себя (op admin_revoke вернёт `last admin`).
- Каждое действие пишется в `events` (module='admin', meta.admin = id исполнителя).

### Первый админ (бутстрап, один раз)

Когда родительский аккаунт уже зарегистрирован (в приложении или family.html), в phpMyAdmin:

```sql
INSERT INTO admins (user_id, status, granted_by)
SELECT a.user_id, 'active', a.user_id
FROM accounts a
WHERE a.email = 'ВАШ_EMAIL' AND a.kind = 'parent'
ON DUPLICATE KEY UPDATE status='active', revoked_at=NULL;
```

Таблица `admins` появляется автоматически при первом обращении к API после деплоя (авто-миграция 008).
Если выполняете SQL раньше первого запроса — сначала откройте приложение, чтобы миграция прошла.
Дальше все гранты — только из панели. (Раздел «Доступ и политика» из панели удалён по решению заказчика 2026-06-07 — этот файл единственный источник справки.)

## 3. Баны семей (полные, с запретом повторной регистрации)

- `family_ban {family_id, reason}`: все аккаунты семьи → disabled, сессии удалены, email КАЖДОГО родителя
  попадает в `bans` как kind='email' (хранится ТОЛЬКО sha256-хэш нормализованного адреса — лишних PII нет),
  действующие приглашения семьи отзываются, сам бан — строка kind='family'.
- Повторная регистрация: `register_parent` и `accept` (co_parent/transfer) проверяют `rt_email_banned` → 403 `banned`.
- `family_unban`: снимает family- и email-баны (lifted_at), аккаунты снова active.
- Себя забанить нельзя (страховка от самоотключения).
- Ограничение по честности: бан по email-хэшу не остановит регистрацию с ДРУГОГО адреса — это осознанный
  компромисс детского приложения без сбора лишних данных.

## 4. Политика картинок (неизменна, §4.7/§8 ПЛАН-аккаунты)

Админ видит факт картинки (имя/размер/дату/счётчик), может удалить (`image_delete`: файл с диска + uploaded_files.deleted_at + очистка пути в записи), но содержимое и ПУТИ никогда не отдаются: ключи `photo`/`image`/`dataUrl` вырезаются из данных в `records`/`record_add`/`record_update`.

Закреплено в `ГАЙД-новое-субприложение.md`, Шаг 4б: реестр и module_data-коллекции подхватываются админкой

## 6. Безопасность и эксплуатация

- Гейт на сервере на каждой операции; флаг в БД, не в конфиге; всё логируется (детская система — логи навсегда).
- Сброс пароля админом = одноразовый 1234 + обязательная смена (родителю при утере пароля быстрее, чем письмо; письмо-сброс тоже работает из family.html).
- `module_reorder` принимает массив id → sort_order 10/20/30…; те же строки `modules`, что видит родитель.
- Производительность (тысячи пользователей): списки users/logs — серверная пагинация по 50 с COUNT;
  агрегаты (записи/картинки) считаются одним GROUP BY-проходом на таблицу, без коррелированных подзапросов
  на строку; families — 2 запроса вместо N+1; индексы покрыты существующими (idx_scope, idx_user_*, idx_created).
- Картинки в счётчиках = фото живых желаний (wishlist_items.photo) + фото в данных модулей
  (module_data JSON $.photo — mood/rating), а НЕ uploaded_files: считается то, что реально видно у пользователя.
- Известные допущения v1: запись/правка контента — generic-JSON (типизированные формы — по контракту admin позже);
  отдельного снятия email-бана нет (снимается вместе с family_unban); points-агрегаты используют JSON_EXTRACT
  (MySQL 5.7+/MariaDB 10.2+ — на Hostinger есть); письмо admin_invite_parent содержит одноразовый 1234 текстом —
  осознанный компромисс простоты (пароль обязателен к смене при первом входе).
