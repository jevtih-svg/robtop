# RobTop — ГАЙД: система оповещений (+ Web Push)

> Канон системы оповещений: как она устроена и как ПЕРЕИСПОЛЬЗОВАТЬ её в любом субприложении.
> Появилась 2026-06-07 (v2026.06.07.59, миграции 020–021). Парных скиллов нет; при роллауте
> новых модулей этот гайд указан в `ГАЙД-новое-субприложение.md`.

## 1. Что это

Системный сервис ядра (как очки `sdk.points` или живое обновление sync): любое субприложение
или серверный эндпоинт может отправить пользователю оповещение, а оболочка единообразно
показывает его всеми поверхностями:

| Поверхность | Где | Что делает |
|---|---|---|
| Колокольчик 🔔 с бейджем | в ШАПКЕ страницы вверху справа на всех экранах (absolute с v.67 — уезжает со скроллом, не висит над контентом); на главном ребёнка и дашборде родителя — кластер [🔔][⚙] в одном хроме (v.65: ⚙ — глобальная кнопка в body) | число непрочитанных; тап → центр |
| Центр оповещений | шторка снизу (паттерн `sheet()`) | список (по 50), непрочитанные подсвечены, «Прочитать все», тумблер Web Push |
| Всплывающий баннер | верх экрана, стиль iOS, поверх всего | новое оповещение в реальном времени; тап = переход, свайп вверх = закрыть, автоскрытие 6с |
| Бейдж у аккаунтов устройства | переключатель аккаунтов (настройки и lock-экран) | «у Артёма 3 непрочитанных» — видно ДО переключения (op `peek`) |
| Web Push (PWA) | системные уведомления ОС, даже при закрытом приложении | «звонок» без содержимого: плашка с общим текстом, тап открывает приложение |
| Бейдж на иконке PWA | где поддерживается (`navigator.setAppBadge`) | число непрочитанных |

Правила:
- Оповещение всегда адресовано ОДНОМУ аккаунту (строка в БД на получателя). Себе — никогда.
- Тексты НЕ хранятся в БД: только `src` (модуль-источник), `type`, `params`, `link`.
  Локализует клиент (en/ru/lv) — язык получателя может отличаться от языка отправителя.
- В демо (file://) системы нет: колокольчик скрыт, `sdk.notify.send` — no-op.
- Колокольчик прячется вместе с HUD: `sdk.ui.hud({hidden:true})` (полноэкранные сцены, напр. змейка).

## 2. Архитектура и поток

```
отправитель                  БД                         получатель (другое устройство/аккаунт)
─────────────                ───────────────            ────────────────────────────────────
модуль: sdk.notify.send ──►  api/notify.php op=send ─►  notifications (кап 100/получателя)
сервер: rt_notify(...)  ──►  (_bootstrap.php)      └─►  rt_push_user() → Web Push «звонок»
                                                        ▲
оболочка получателя: поллинг api/sync.php (4с) ────────┘
  r.ntf = {n: непрочитанных, m: max id непрочитанного}
  → RT.Notify.sync(): бейдж = n; m вырос → list → баннер(ы)
```

- `app/api/migrations/020_notify.sql` — таблица `notifications` (id, user_id, actor_id, src,
  type, params JSON, link JSON, read_at, created_at). `021_push.sql` — `push_subs`.
- `app/api/notify.php` — list / read / read_all / send / peek (см. §5).
- `app/api/_bootstrap.php` → `rt_notify()`, `rt_child_parents()`; `api/_push.php` → отправка push.
- `app/core/notify.js` — ВСЯ клиентская часть + словари `ntf.*` (en/ru/lv).
- `app/api/sync.php` — поле `ntf` в ответе (сырые `{n,m}`, не хэш: n — готовый бейдж).
- Хуки оболочки: `shell.js` (boot → `RT.Notify.boot()`, sync-тик, hud, `openTicket`),
  `loader.js` (`RT._pendingLink` → хук `link` модуля), `sdk.js` (`sdk.notify`).

Почему поллинг, а не SSE/WebSocket — та же причина, что у sync: shared-хостинг.
Web Push добивает случай «приложение закрыто».

## 3. Как подключить оповещения в СВОЁ субприложение (чек-лист)

Пример «модуль days хочет оповещать: событие наступило».

1. **Отправка из модуля** — одна строка после успешного основного действия (fire-and-forget,
   промис никогда не reject, ошибки сервера глотаются):

```js
sdk.notify.send("parents", "day_arrived", {
  params: { title: ev.title },          // параметры шаблона
  link:   { module: "days", item: ev.id } // куда ведёт тап (необязательно)
});
```

   Адресаты `to`:
   - `"parents"` — родители отправителя-ребёнка (опекуны + родители семьи); для родителя — со-родители.
   - `"child"` — ТОЛЬКО роль parent; ребёнок, выбранный на дашборде (`child=` подставляется сам, как в sdk.data), иначе первый ребёнок семьи.
   - `"family"` — все активные члены семьи, кроме отправителя.

2. **Шаблон текста ×3 языка** — ключ `ntf.ev.<модуль>.<type>` в словарях `core/notify.js`
   (блоки en/ru/lv рядом с bank/tickets/wishlist):

```js
ev:{ days:{ day_arrived:"Наступил день: «{title}»" } }
```

   Шаблоны живут в ЯДРЕ, а не в messages модуля, потому что словари модуля грузятся только
   при его открытии, а центр оповещений должен показывать всё сразу. Без ключа клиент
   возьмёт `params.text` (нелокализуемый фолбэк), иначе — generic «Новость из {app}».
   `params.note` автоматически дописывается хвостом « · note». Имя приложения и иконку/цвет
   баннер берёт из реестра (`tile.<id>` + манифест) — отдельно ничего не надо.

3. **Переход к «предмету» (опционально)** — хук `link` в контракте register:

```js
RobTop.register({ id:"days", mount:mount, unmount:unmount, messages:MESSAGES,
  link:function(link, sdk){ openItem(link.item); } // вызовется ПОСЛЕ mount (или сразу, если модуль открыт)
});
```

   Без хука тап просто открывает модуль — для большинства случаев достаточно.
   Формы link: `{module, item?}` (модуль обязан быть active+enabled, иначе тост «недоступно»),
   `{view:"ticket", id}` — переписка тикета, `{view:"settings"|"shared"}` — настройки.

4. **Серверный источник** (если оповещение рождается в PHP — как ответ тикета):

```php
rt_notify($toUid, 'days', 'day_arrived', ['title' => $t], ['module' => 'days'], $actorId);
```

   Никогда не оборачивай в условия успеха самого rt_notify — он сам глотает ошибки
   и не должен ломать основную операцию.

5. **Проверь**: оповещение приходит на ДРУГОЙ аккаунт (не себе), текст на 3 языках,
   баннер ведёт куда надо, демо не падает.

## 5. API-справочник

### POST api/notify.php (сессия обязательна, кроме peek)

| op | Параметры | Ответ | Заметки |
|---|---|---|---|
| `list` | `before?` (id) | `{ok, items:[{id,src,type,params,link,read,createdAt}]}` | по 50, свежие сверху |
| `read` | `id` | `{ok}` | идемпотентно |
| `read_all` | — | `{ok}` | кнопка «Прочитать все» |
| `send` | `to, src, type, params?, link?, child?` | `{ok, sent}` | валидация src/type/link; 30 отправок/час → 429 |
| `peek` | `tokens:[switch-токены]` (≤10) | `{ok, accounts:[{user,unread}]}` | БЕЗ сессии (lock-экран); модель доверия = op switch |

### POST api/push.php (сессия обязательна)

| op | Параметры | Ответ |
|---|---|---|
| `key` | — | `{ok, key}` — публичный VAPID (null = push не настроен, тумблер скрыт) |
| `subscribe` | `endpoint, keys{p256dh,auth}?, lang?` | `{ok}` — upsert; endpoint глобально уникален: на семейном планшете пуши уходят ПОСЛЕДНЕМУ включившему аккаунту |
| `unsubscribe` | `endpoint` | `{ok}` |
| `status` | `endpoint` | `{ok, mine}` — записан ли endpoint за текущим аккаунтом |

### PHP-хелперы (_bootstrap.php / _push.php)

- `rt_notify($toUid, $src, $type, $params=null, $link=null, $actorId=null)` → bool.
  Пишет строку, держит кап 100/получателя, дёргает `rt_push_user()`. Никогда не кидает.
- `rt_child_parents($db, $cid)` → массив id родителей ребёнка (опекуны + семья).
- `rt_push_user($uid)` — web push «звонком» на все подписки получателя (no-op без VAPID).

## 6. Web Push (PWA) — как устроено

- **С РЕАЛЬНЫМ ТЕКСТОМ (зашифрованный payload, с v2026.06.07.68).** Плашка ОС несёт само
  сообщение («Копилка / Новое задание „Убрать комнату“ — +15 очков»), а не общий текст.
  - **Текст рендерит СЕРВЕР** (`api/_ntf_text.php`) на языке КОНКРЕТНОГО устройства
    (`push_subs.lang`), потому что пуш уходит при закрытом приложении — клиента нет.
    Это ЗЕРКАЛО словаря `ntf.ev.*` из `core/notify.js`: **при правке текстов меняй ОБА файла**
    (структура и набор типов идентичны; нет ключа на сервере → пуш покажет generic, в
    приложении текст всё равно правильный — клиент главный).
  - **Шифрование** — RFC 8291 `aes128gcm` (ECDH P-256 + HKDF-SHA256 + AES-128-GCM) на чистом
    openssl, без composer: `rt_push_encrypt()` в `api/_push.php`. Тело = `salt|rs|idlen|as_public|ciphertext`,
    заголовок `Content-Encoding: aes128gcm`. Алгоритм сверен с тест-вектором RFC 8291 Appendix A
    (round-trip-расшифровка вернула исходный текст, включая кириллицу).
  - Подписка без ключей `p256dh/auth` (редко) → фолбэк «звонок» без тела, sw.js покажет общий
    текст `TXT` (по языку из query регистрации).
- **sw.js** показывает `payload {title, body, url}`, БЕЗ кэширования (никакого перехвата fetch —
  деплой не воюет с SW). Регистрация: `sw.js?lang=<язык>&v=<RT_VER>`.
- **Включение** — тумблер «Уведомления на этом устройстве» в центре оповещений:
  разрешение → регистрация SW → `pushManager.subscribe(VAPID)` → op `subscribe` (с ключами).
- **iOS**: пуши работают ТОЛЬКО у приложения, добавленного на экран «Домой» (iOS 16.4+).
  В Safari-вкладке тумблера нет — вместо него подсказка «добавь на Домой».
- **Ключи VAPID** — один раз в `config.php` на сервере (НЕ в Git), без них push молча
  выключен, всё остальное работает. Настройка — РЕЛИЗ.md, раздел «Web Push».
- Мёртвые подписки (404/410 от push-сервиса) удаляются при рассылке автоматически.

## 7. Поведение и лимиты (чтобы не споткнуться)

- Кап хранения: 100 последних оповещений на получателя (rt_notify чистит старые).
- Антиспам отправки: 30 оповещений в час с аккаунта (op send) → 429.
- Баннеры: максимум 2 на экране, очередь до 6, новые из одного sync-тика — до 3.
- Баннеры показываются и поверх открытых форм (ничего под руками не трогают) — поэтому
  в shell.js вызов `RT.Notify.sync()` стоит ДО гейта `syncFormOpen()`.
- Прочитанность: тап по строке/баннеру или «Прочитать все». Бейдж обновляется на
  следующем sync-тике с сервера (и оптимистично сразу).
- `peek` отдаёт ТОЛЬКО числа непрочитанных по валидным switch-токенам — никакого содержимого.
- Версия приложения теперь в **11 местах** index.html (добавился `core/notify.js?v=`).

## 8. Тест-чеклист после изменений в системе

- [ ] тумблер push: вкл/выкл, плашка ОС приходит при закрытом приложении, тап открывает RobTop

- [ ] демо file://: колокольчика нет, модули не падают
