# RobTop — настройка и релиз

Цель: **один раз** настроить (подробно ниже), а дальше выпускать обновления **в одно действие** — просто отправить изменения в ветку `main`, и сайт сам обновится на Hostinger.

```
  ты вносишь правки
        │
        ▼
   git push в main ──────────►  вебхук Hostinger Git  ──pull──►  Hostinger
                                  (авто-деплой)                  apps.tilley.live/robtop
                                                                       │
                                              PHP API  ◄──►  MySQL (статистика)
```

Как это работает: деплой делает **вебхук Hostinger Git** — при push в `main` Hostinger сам тянет (`pull`) свежий код из репозитория. Клиент (`index.html`) общается с PHP-API (`api/*.php`), а тот пишет всё в MySQL. Пароли БД лежат в `config.php` **только на сервере** и в Git не попадают.

> **Деплой = вебхук Hostinger Git (см. раздел «Альтернатива: деплой кнопкой прямо в hPanel»).** GitHub Actions больше НЕ деплоит: FTPS-заливка убрана, workflow `.github/workflows/deploy.yml` оставлен только как проверка синтаксиса JS (`node --check`, без деплоя). Секреты `FTP_*` и шаги ниже про FTP — устаревшие, можно игнорировать или удалить. Музыку и графику (`media/`, вне Git) по-прежнему заливаем вручную по FTP один раз.

> **РАСКЛАДКА НА СЕРВЕРЕ (важно).** Вебхук кладёт в `robtop/` ВЕСЬ репозиторий, поэтому приложение живёт по адресу **`apps.tilley.live/robtop/app/`** (с корня `/robtop/` стоит редирект). Отсюда все «ручные» пути:
> - `config.php` → **`robtop/app/config.php`** (рядом с index.html приложения);
> - музыка → **`robtop/app/media/music/`**, графика → **`robtop/app/media/...`**;
> - загрузки → `robtop/app/uploads/...`;
> - тест почты → `https://apps.tilley.live/robtop/app/api/mail_test.php?...`.
> Документация (`*.md`), `schema.sql`, миграции и `.git` закрыты от публичного доступа корневым `robtop/.htaccess` (лежит в репозитории). Возможный переезд на чистый `/robtop/` через deploy-ветку отложен и описан в журнале КОНТЕКСТ.md.

## Часть 1. Разовая настройка (делается один раз)

### Шаг 1. Репозиторий на GitHub

1. Создай новый репозиторий на github.com (например `robtop`), приватный.
2. На своём компьютере открой папку проекта (`Приложение RobTop`) в терминале и выполни:

```bash
git init
git add .
git commit -m "RobTop: первый релиз"
git branch -M main
git remote add origin https://github.com/ТВОЙ_ЛОГИН/robtop.git
git push -u origin main
```

> `config.php` не закоммитится — он в `.gitignore`. Это правильно.

### Шаг 2. База данных на Hostinger

1. hPanel → **Базы данных → Управление базами данных MySQL**.
2. Создай новую базу и пользователя (пароль сохрани). Запомни:
   - имя базы (вид `u1234567_robtop`),
   - имя пользователя (вид `u1234567_robtop`),
   - пароль,
   - хост (обычно `localhost`).
3. **Импорт схемы НЕ обязателен.** Таблицы создаются и обновляются **автоматически из PHP** при первом обращении к API (см. ниже «Авто-миграции»). Достаточно создать пустую базу и заполнить `config.php` — приложение само построит таблицы (`users`, `wishlist_items`, `events`, `uploaded_files`, `modules`, `module_data`) и реестр модулей.

> Хочешь применить вручную (по желанию) — в phpMyAdmin → **Импорт** → `schema.sql`. Файлы в `app/api/migrations/` — тот же набор, который выполняет авто-миграция.

### Шаг 3. Папка сайта и `config.php` на сервере

1. hPanel → **Файлы → Файловый менеджер**.
2. Зайди в корень сайта `apps.tilley.live` (обычно `.../public_html`) и создай папку **`robtop`** (если её ещё нет).
3. Внутри `robtop` создай файл **`config.php`** со своими данными базы (за образец — `config.example.php`):

```php
<?php
return [
    'db_host'    => 'localhost',
    'db_name'    => 'u1234567_robtop',
    'db_user'    => 'u1234567_robtop',
    'db_pass'    => 'ТВОЙ_ПАРОЛЬ_БД',
    'db_charset' => 'utf8mb4',
    'api_token'  => '',
    'admin_pin'  => '',  // PIN родителя для магазина приложений; пусто = управление выключено
];
```

> Этот файл деплой не трогает и не перезаписывает. Пароли остаются только здесь.

4. hPanel → **PHP Configuration** → выбери **PHP 8.x**.

### Шаг 4. Данные FTP

1. hPanel → **Файлы → FTP-аккаунты**. Возьми:
   - **FTP host** (вид `ftp.tilley.live` или IP),
   - **FTP username**,
   - **FTP password** (можно задать/сбросить там же).
2. Узнай **полный путь к папке `robtop`**: в Файловом менеджере зайди в `robtop`, скопируй путь из адресной строки. Он понадобится с `/` в конце, например:
   `/domains/apps.tilley.live/public_html/robtop/`
   (у разных аккаунтов путь отличается — бери свой).

### Шаг 5. Секреты в GitHub

В репозитории: **Settings → Secrets and variables → Actions → New repository secret**. Добавь 4 секрета:

| Имя секрета       | Значение                                              |
|-------------------|-------------------------------------------------------|
| `FTP_SERVER`      | FTP host из шага 4 (без `ftp://`)                     |
| `FTP_USERNAME`    | FTP-логин                                              |
| `FTP_PASSWORD`    | FTP-пароль                                             |
| `FTP_REMOTE_DIR`  | путь к папке `robtop/` со слешем в конце (шаг 4)       |

### Шаг 6. Первый деплой

- Либо сделай любой коммит в `main` (`git push`),
- либо в репозитории: вкладка **Actions → Deploy RobTop to Hostinger → Run workflow**.

Открой `https://apps.tilley.live/robtop/` — приложение должно работать с базой. Готово.

## Часть 2. Как выпускать обновления (каждый раз)

Когда я (или ты) меняем файлы в папке проекта:

```bash
git add .
git commit -m "что изменили"
git push
```

Через ~1 минуту изменения уже на сайте. Прогресс виден во вкладке **Actions**.

**Кнопка вместо авто-деплоя.** Если хочешь деплоить вручную: вкладка **Actions → Deploy RobTop → Run workflow**. Это та самая «кнопка в один клик».

**Откат.** Нужно вернуть прошлую версию — на GitHub: **Actions** → открой удачный прошлый запуск → **Re-run jobs**. Либо `git revert <коммит>` и `git push`.

## Альтернатива: деплой кнопкой прямо в hPanel (без секретов)

Если не хочешь возиться с FTP-секретами, у Hostinger есть встроенный Git:

1. hPanel → **Advanced → GIT**.
2. **Create repository**: вставь URL репозитория, ветку `main`, путь установки `public_html/robtop`.
3. Появится кнопка **Deploy** — нажал, и свежая версия с GitHub раскатана. Можно включить **Auto-deployment** (webhook в GitHub), тогда деплой по push.

> Для приватного репозитория добавь SSH-ключ Hostinger в GitHub (Deploy keys). Для публичного — ничего не нужно. В этом варианте `config.php` так же создаётся на сервере и в Git не хранится.

Можно использовать любой из двух способов. GitHub Actions удобнее для «push = в проде», hPanel Git — для «нажал кнопку Deploy».

## Проверка и частые вопросы

- **Открывается, но без данных / ошибка БД** → проверь `config.php` (логин/пароль/имя базы), что `schema.sql` импортирован, и что выбран PHP 8.x.
- **Actions падает на FTP** → проверь `FTP_SERVER/USERNAME/PASSWORD` и что `FTP_REMOTE_DIR` оканчивается на `/` и существует.
- **Фото не сохраняется** → большие картинки. Клиент сам уменьшает их до ~900px, но при необходимости в hPanel → PHP Configuration можно поднять `post_max_size`/`upload_max_filesize`.
- **Сайт открыли локально (файл с компьютера)** → это демо-режим (данные в браузере). Боевые данные — только на сервере с базой.

## Безопасность

- `config.php` не в Git и не публикуется (`.gitignore` + правило в `.htaccess`).
- Внутренние файлы API (`_db.php`, `_bootstrap.php`) закрыты от прямого доступа `.htaccess`.
- При желании можно включить простой токен API: задать `api_token` в `config.php` (тогда добавим заголовок в клиент).

## Авто-миграции БД (без phpMyAdmin)

Структуру базы приложение чинит само. При первом обращении к API (`api/_db.php` → `rt_migrate`) выполняются все `.sql` из `app/api/migrations/`, которых ещё нет в таблице `schema_migrations`, по порядку имён (`001_…`, `002_…`). Файлы идемпотентны (`CREATE TABLE IF NOT EXISTS` + `INSERT … ON DUPLICATE KEY`), поэтому повторный запуск безопасен, а добавление новой функции = новый файл миграции.

- Тебе не нужно открывать phpMyAdmin: задал `config.php`, задеплоил — таблицы и реестр модулей создаются сами при первом открытии приложения.
- Новая миграция: положи файл вида `app/api/migrations/003_что_то.sql` и задеплой. Применится автоматически один раз.
- Если у пользователя БД нет прав `CREATE`, миграция тихо пропускается (API не падает) — тогда импортируй `schema.sql` вручную.
- Прогресс зафиксирован в таблице `schema_migrations` (имя файла + дата).

## Медиа, загрузки и пользователи

- **Графика приложения** (иконка, логотипы) лежит в `app/media/`, тоже **вне Git** (репозиторий = только код). Заливаешь эту папку на сервер по FTP в ту же папку, где `index.html` (получится `robtop/media/`). В коде путь `media/...`, например иконка `media/icon/icon-180.png`. Без этой загрузки сайт работает, но иконка и графика не покажутся.
- **Музыка-награда** (модуль «Таймер чистки зубов») лежит в `app/media/music/` — это лицензированные mp3, тоже **вне Git**. Залей папку `media/music/` по FTP в `robtop/app/media/music/` (раскладка вебхука, см. баннер вверху). Имена файлов в коде зашиты массивом `PLAYLIST` в `app/modules/teeth/module.js` — при добавлении/замене треков правь и этот массив (имя без `.mp3`), и заливай файлы по FTP. Без загрузки таймер работает, но после чистки музыка не заиграет.
- Загруженные фото лежат на сервере в `robtop/uploads/users/<id>/wishlist/`, а в базе хранится только путь. Файлы **не** в Git (`.gitignore`) и деплой их **не трогает и не удаляет** (исключения в workflow), поэтому обновления кода их не сбрасывают.
- Папки создаются автоматически при первой загрузке. Нужно, чтобы у PHP были права на запись в `robtop/` (обычно так и есть). Если фото не сохраняется — проверь права на папку `uploads`.
- На будущее: хранилище за абстракцией (`api/_storage.php`). Позже можно переключиться на S3 / Cloudflare R2 / Supabase / Spaces, поменяв `storage_driver` в `config.php` и добавив драйвер, без переписывания приложения.
- **Временный режим одного пользователя:** пока вход по PIN не сделан, все данные принадлежат Артёму (id 1). В коде это одна функция `rt_user_id()` (`app/api/_bootstrap.php`, константа `RT_DEFAULT_USER_ID`). Когда добавим вход — поменяем только её, остальное приложение не трогаем.

## Почта (Brevo): как включить отправку писем

Письма получают только родители (сброс пароля, приглашения). Код уже в приложении, тебе нужно только заполнить `config.php` на сервере. Без настройки работает режим `log`: письма не шлются, ссылки видны в журнале `events`.

1. Зарегистрируйся на [brevo.com](https://www.brevo.com) (бесплатный тариф, 300 писем в день).
2. В Brevo: **Senders & Domains → Domains → Authenticate your domain** для `tilley.live` (Brevo покажет DNS-записи DKIM/SPF, добавь их в hPanel → DNS). Это правильный путь, письма не будут падать в спам. Быстрая альтернатива на время теста: **Senders → Add a sender** с твоим существующим ящиком.
3. В Brevo: **SMTP & API → API Keys → Generate a new API key**, скопируй ключ (вид `xkeysib-...`).
4. На сервере в `robtop/config.php` поставь `'mail_driver' => 'brevo'` и вставь ключ в `'mail_brevo_key'`. Поле `'mail_from'` должно совпадать с подтверждённым отправителем/доменом из шага 2.
5. Проверь отправку (подставь свой PIN и адрес):
   `https://apps.tilley.live/robtop/app/api/mail_test.php?pin=ВАШ_PIN&to=ваш@адрес.com&lang=ru`
   Ответ `{"ok":true,...}` плюс письмо во входящих = всё работает.

Шаблоны писем лежат в `app/api/mail/` (простые HTML, языки en/ru/lv), их реестр с темами и плейсхолдерами — `app/api/mail/registry.php`. Язык письма равен языку приложения пользователя (клиент передаёт `lang`), фолбэк — `mail_default_lang` из конфига. Новое письмо = запись в реестре + 3 HTML-файла. Запасной драйвер без сервисов — `'phpmail'` (PHP `mail()` через Hostinger), но письма чаще попадают в спам.

## Магазин приложений (субприложения)

С версии 2.0 приложение — это **оболочка + модули**. Каждое субприложение живёт в своей папке: родные модули в `app/modules/<id>/` (могут иметь серверный `api.php`), установленные в рантайме — в `app/apps/<id>/` (только статика).

- **Включить управление.** Задай непустой `admin_pin` в `config.php`. На главном экране кнопка ⚙ открывает «Приложения». Введи этот PIN — появятся переключатели вкл/выкл, стрелки порядка и установка новых приложений. Без PIN экран только для просмотра, ребёнок ставить приложения не может.
- **Установка приложения.** Бандл — это файл `*.robtop.json` (`{ manifest, files }`). В магазине: «Выбрать бандл» → файл проверяется (запрет серверного кода, whitelist типов, лимиты) → распаковывается в `apps/<id>/` и добавляется в реестр (выключенным). Включи переключателем. Пример бандла — `app/apps/_catalog/dice.robtop.json` («Кубик»).
- **Деплой и рантайм.** Папка `app/apps/**` исключена из Git и из FTP-деплоя — установленные приложения **переживают** релизы и не удаляются деплоем. Файл `apps/.htaccess` запрещает выполнение PHP в этой папке (установленные модули — только статика); при первой установке он создаётся автоматически.
- **Безопасность.** Все операции магазина (`api/store/*`, `api/modules.php` для записи) требуют `admin_pin`. Установленные модули не могут содержать серверный код и работают только через клиентский SDK и универсальное хранилище (`api/data.php`).
