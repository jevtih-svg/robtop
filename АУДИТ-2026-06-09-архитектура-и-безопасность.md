# RobTop — Audit: Architecture, Best Practices, Security

Date: 2026-06-09
Scope: full app (PHP/MySQL backend in `app/api`, JS shell in `app/core`, 18 modules in `app/modules`, schema and migrations, PWA, `.htaccess`).
Method: direct review of the security-critical core files plus three parallel deep reviews of the front-end shell, the API endpoints, and the modules/schema. Every Critical and High finding below was confirmed against source (file:line cited).

---

## Executive summary

The codebase is disciplined for a hand-written, framework-free project. SQL is parameterized everywhere (no SQL injection found), HTML output goes through a single `esc()` helper consistently, file uploads validate real image bytes, ownership checks (`rt_can_manage_child`, `rt_can_read`) are applied on the parent and sharing endpoints, and session cookies are `HttpOnly` + `Secure` + `SameSite=Lax`. That is a solid baseline.

The serious problems are architectural, in two places:

1. **The points economy is client-authoritative.** A logged-in child can mint any balance with a single API call. This is the headline issue and it defeats the core promise of the app (parent-controlled rewards). Confirmed.
2. **Session tokens carry about 30 bits of entropy.** The post-login credential is a 6-character code, while the codebase already has a strong generator it uses elsewhere. One-line fix, high impact. Confirmed.

The app is also mid-migration from single-user (Артём, id 1) to a multi-account family model. The default config still resolves unauthenticated requests to user 1, so the security posture depends on production config values that this audit could not see. That ambiguity is itself a risk.

Top 5 to fix, in order:
1. Make points server-authoritative (SEC-1, Critical).
2. Generate session tokens with `random_bytes` (SEC-2, High, trivial).
3. Verify production `config.php`: set `single_user=false` and require login on data endpoints (SEC-3, High/config-dependent).
4. Add rate limiting + lockout on login, invite-accept, and password-reset (SEC-4, High).
5. Add a CSP and `X-Frame-Options`, and escape/validate the registry `color` (SEC-7/SEC-8, Medium).

## 1. Architecture

### 1.1 The design

The app is a "shell + modules" PWA. A thin `index.html` host loads `core/*` (the shell), which reads a module registry from the server (`registry.php`), renders home-screen tiles, and mounts one module at a time into a shared container. Each module is an IIFE that calls `RobTop.register({id, mount, unmount, ...})` and talks to the shell only through an injected `sdk` object (`core/sdk.js`). The SDK mediates data CRUD (`data.php`), the points ledger, the tasks engine (`tasks.php`), notifications, media upload, i18n, and theming. The backend is PHP with PDO over MySQL; a generic `module_data` table backs most modules so a new module needs no new table or endpoint.

This is a coherent, well-reasoned design. The seams are clear, the SDK boundary is mostly respected, and there is good inline rationale in the code explaining why decisions were made.

### 1.2 Strengths

- **Single identity resolver.** `rt_user_id()` in `_bootstrap.php:44` is the one place identity is decided. Centralizing it makes the auth model auditable.
- **Consistent ownership scoping.** Since v.44 the client sends `child=<id>` on every parent request, and the server re-checks with `rt_can_manage_child` (`data.php:62-65`, `tasks.php:51-54`, `balance.php:24-26`). Parent reads use `rt_can_read`. This closes the obvious IDOR class.
- **Generic data store.** `module_data` keyed by `(user_id, module, collection)` lets modules ship without schema changes. Good for velocity.
- **Soft deletes and an audit log.** `events` (via `rt_log`) records actions; most deletes are soft (`deleted_at`). Good for a family app where history matters.
- **Disciplined output encoding.** One `esc()` helper, used at essentially every render site including the highest-risk surfaces (chat bodies, nicknames, ticket text, notifications). The public wishlist viewer `w.html` even normalizes `javascript:` URLs with `normUrl`.

### 1.3 Weaknesses and risks

| ID | Severity | Where | Finding |
|----|----------|-------|---------|
| ARCH-1 | High (architectural) | `data.php`, `tasks.php:28`, `core/sdk.js` | The points economy lives client-side. The server stores ledger rows verbatim and `tasks.php` explicitly does not award points ("ОЧКИ сервис НЕ начисляет"). The trust boundary for currency is the child's own browser. See SEC-1. |
| ARCH-2 | Medium | `loader.js:26-34` | "Sandboxed modules" is aspirational. Modules load as plain `<script>` in the shell's own realm, so any module has full access to `window.RobTop`, `localStorage`, and the DOM. The SDK is a convention, not isolation. For runtime-installed apps this matters (see SEC-9). |
| ARCH-3 | Medium | `schema.sql`, migrations 007/023/024/028/029 | Most newer tables (`tasks`, `chat_*`, `notifications`, `push_subs`, `events`, `uploaded_files`, `sessions`) have no foreign keys and there are no CHECK constraints anywhere. Orphaned rows and out-of-range values are silently possible. |
| ARCH-4 | Low | `core/parent.js`, `core/shell.js` | Duplication between the parent dashboard and the shell: `tileHtml`, `esc`, day/time helpers, and color logic exist in both and have already drifted (one escapes `color`, the other does not, see SEC-8). |
| ARCH-5 | Low | modules reaching `window.RobTop._shell` | `wishlist/module.js` and others call `window.RobTop._shell.openSettings()` / `setDemo()` directly, bypassing the SDK they otherwise respect. Expose these via the SDK and forbid `_shell` access. |

## 2. Security

Findings are ordered by severity. Confirmed = traced to source in this audit. Config-dependent = depends on production `config.php` values not visible here.

### CRITICAL

**SEC-1 — A child can mint unlimited points (broken economy / access control). Confirmed.**
Files: `app/api/data.php:112-127`, `app/modules/bank/module.json:10`, `app/api/balance.php:36-45`, `app/api/tasks.php:28`, `app/core/sdk.js` (bank path).

The points balance is computed as `SUM(data.n)` over rows where `module='bank'` and `collection='points'` (`balance.php:36-44`). Those rows are ordinary `module_data` records written through the generic `create` op, which stores the client-supplied `data` JSON with no validation of the amount (`data.php:113-119`). The only gate is `rt_role_can($mod, 'edit', $role)` (`data.php:33`), and the bank manifest grants `edit` to `child` (`bank/module.json:10`). Worse, the default for any non-read action is `['child']` (`_bootstrap.php:215`), so even modules that omit a `roles.edit` are child-writable. The tasks engine deliberately does not award points server-side (`tasks.php:28`); the client does it.

Attack (from a logged-in child's browser console):
```js
fetch('api/data.php', {method:'POST', headers:{'Content-Type':'application/json'},
  body: JSON.stringify({op:'create', module:'bank', collection:'points',
                        data:{n:1000000, kind:'win', reason:'x'}})})
```
The next balance read (client `bankSummary` or `balance.php`) returns the inflated total. The same path forges game wins (`guess`, `snake`, `find`) and zero/negative shop prices.

Fix: make points server-authoritative. Reject writes to `bank/points` (and any currency collection) through `data.php`. Introduce a single server writer that derives `n` from the action it authorizes (an approved task awards its stored point value; a game outcome is validated against a server-held tariff; a shop purchase debits the stored item price inside one transaction). Then switch `sdk.points.summary` to `balance.php`. Clamp and record `by`/`kind` server-side, never trusting client `data.n`.

### HIGH

**SEC-2 — Session tokens have ~30 bits of entropy. Confirmed.**
Files: `app/api/_accounts.php:29-35`, `:203-212`.

`rt_start_session()` issues the session cookie from `rt_token()` = `rt_code(6)`: 6 characters from a 31-symbol alphabet, about 8.9x10^8 combinations (~30 bits). This is the sole post-login credential and `rt_session_user_id()` validates it with no per-token throttling or lockout, against a `sessions` table that holds many live 30-day tokens at once. With many valid tokens present, guessing any one of them online is feasible, yielding account takeover including parents and admins. Note the codebase already generates strong switch tokens with `bin2hex(random_bytes(32))` (`_accounts.php:287`); only the session token is weak.

Fix: change `rt_start_session` to use `bin2hex(random_bytes(32))`. Keep `rt_code(6/8)` only for human-dictated invite and reset codes. Add throttling on session lookups.

**SEC-3 — Default config resolves unauthenticated requests to user 1 (Артём). Confirmed in code; impact config-dependent.**
Files: `app/api/_bootstrap.php:42-55`, `app/api/_db.php` / `config.example.php` (`single_user => true`, `api_token => ''`), README line 47.

With the shipped defaults, `rt_guard()` is a no-op (empty `api_token`) and `rt_user_id()` falls back to user 1 whenever there is no valid session. Endpoints that call `rt_user_id()` directly without `rt_require_login` (`data.php`, `state.php`, `stats.php`, `balance.php`, `sync.php`, `action.php`, `upload.php`, `image.php`) then read and write user 1's data anonymously. The README still describes the app as single-user with `rt_user_id()` pinned to id 1, which suggests this fallback is live in production. If so, anyone on the internet can POST to `api/data.php` and read, tamper, mint points for, or upload files as Артём with no credentials.

Fix: verify production `config.php`. Set `single_user=false`, and have the data endpoints call `rt_require_login()` rather than relying on the silent fallback. Fail closed if `single_user` is true while more than one account exists.

**SEC-4 — No rate limiting on login, invite-accept, or password-reset. Confirmed (no throttling primitives exist).**
Files: `app/api/accounts.php` (`login`, `invite_info`/`accept`, `reset`), `app/api/_accounts.php` (no throttle helpers defined).

`login` runs `password_verify` with no attempt counter, delay, or lockout, and the default child password is the well-known `1234` (minimum length 4, `_accounts.php:22`). Invite tokens are 6-char codes that live 7 days; reset codes are 8-char and live 1 hour. None of these flows is throttled, so all three are online-guessable, especially given short codes and known default passwords.

Fix: per-account and per-IP throttling with exponential backoff and temporary lockout on `login`, `invite_info`/`accept`, and `reset`. Stop seeding the universal `1234` (use a random temp password shown once to the parent). Where codes must stay short because they are read aloud, throttling is mandatory.

### MEDIUM

**SEC-5 — Switch tokens persisted in plaintext `localStorage`. Confirmed.**
Files: `app/core/shell.js:406-417, 441`.

`rt_accounts = [{id, nick, kind, tok}]` stores long-lived (180-day) switch tokens in `localStorage`. They are exchanged via `op:"switch"` for a full session of another family account, including a parent's. Because modules run in the shell's own realm (ARCH-2), a malicious installed module, any XSS, or shared-device access can read every stored token and impersonate the parent, defeating parental controls. The tokens themselves are strong; the storage location is the risk.

Fix: prefer an `HttpOnly` cookie for the switch secret, shorten TTL, and bind tokens to the device. Treat a leaked switch token as account takeover. At minimum, document the threat and scope tightly.

**SEC-6 — Authenticated SSRF via the Web Push endpoint. Confirmed.**
Files: `app/api/push.php:41-54` (`subscribe`), `app/api/_push.php:140-154` (`curl_init($endpoint)`).

`subscribe` only checks that the endpoint starts with `https://` and is at most 500 chars. The server later POSTs to that stored endpoint. An authenticated user can register an internal `https` URL and have the server make outbound requests to it (blind SSRF).

Fix: allowlist push endpoints to known FCM, Mozilla, Apple, and WNS hosts (or block private and loopback ranges), set `CURLOPT_PROTOCOLS` to HTTPS only, and disable redirects.

**SEC-7 — No CSP, `X-Frame-Options`, or `frame-ancestors`. Confirmed.**
Files: `index.html` (inline `<script>`), `app/.htaccess` (cache headers only). `X-Content-Type-Options: nosniff` is set in `_bootstrap.php:10` for API responses but not for HTML.

Given heavy `innerHTML` use, there is no defense-in-depth backstop if an output-encoding bug slips through, and the app can be framed (clickjacking).

Fix: add a CSP (at minimum `frame-ancestors 'none'`, `object-src 'none'`), `X-Frame-Options: DENY`, and `X-Content-Type-Options: nosniff` for HTML via `.htaccess`.

**SEC-8 — Attribute-injection XSS via registry `color`. Confirmed.**
Files: `app/core/shell.js:589`, `app/api/store/install.php` (no `color` validation).

`tileHtml` writes `style="--c:'+(m.color||"#19e3ff")+'"` with no escaping, while `parent.js:451` escapes the same value (the drift in ARCH-4). `store/install.php` stores the manifest as-is, so a runtime-installed app can supply `color` like `x" onmouseover="..."` and break out of the attribute. Any parent can install an app (SEC-9), and the tile renders for children too.

Fix: escape `m.color` in `shell.js:589` (and the banner `--c` at `shell.js:1389`), or validate it against a hex/rgb regex at the registry boundary in `install.php`.

**SEC-9 — Parent wishlist link allows `javascript:` execution in the parent session. Confirmed.**
Files: `app/core/parent.js:607` versus `app/modules/wishlist/module.js:250-251` and `w.html`.

The child view and the public viewer normalize links with `normUrl` (forces an `https://` prefix, neutralizing `javascript:`). The parent dashboard does not: it renders the raw stored link as `href="'+esc(w.link)+'"`, and `esc()` does not neutralize the `javascript:` scheme. A child can store a `javascript:...` link; when the parent taps "Open link" from the read-only dashboard, it runs in the parent's origin.

Fix: pass `w.link` through `normUrl`/`safeUrl` in `parent.js:607`, rejecting any non-`http(s)` scheme. Better, normalize on save so the stored value is always safe.

### LOW / by design / informational

- **SEC-10 — Any parent can install arbitrary client-side JS for the whole family.** `store/install.php:18` gates on `rt_admin_gate()` (any active parent). Server code is blocked (extension denylist, `server=0`, filename validated against traversal at `install.php:42`), but the uploaded `module.js` runs in the app origin for every family member including children. This is the intended "install an app" feature, but a compromised parent account becomes stored JS execution against the children. Worth documenting; consider signing or reviewing installed bundles.
- **SEC-11 — Default child password `1234`.** `strings.js`, `accounts.php:262`. A window exists between account creation and first sign-in where the known password is valid. Use a random temp password shown once to the parent.
- **SEC-12 — Unauthenticated, unthrottled upload.** `upload.php` relies on the `rt_user_id()` fallback and has no rate limit (4 MB base64 per call). Content validation is good (`getimagesizefromstring`, server-generated filename, `.htaccess` denies script execution in `uploads/`), so the risk is storage abuse, not code execution. Require a live session and cap per-user storage/count.
- **SEC-13 — CSRF: acceptable baseline, add defense-in-depth.** The session cookie is `SameSite=Lax` + `HttpOnly` + `Secure` (`_accounts.php:189-202`) and all state changes are POST, so cross-site POSTs do not carry the cookie. There is no explicit CSRF token. Adding an `Origin` allowlist check on POST endpoints would harden older-browser and edge cases. (This corrects an earlier "High CSRF" read: Lax already mitigates the main vector.)
- **SEC-14 — HTTPS redirect commented out.** `app/.htaccess` leaves the force-HTTPS block disabled. Enable it once SSL is confirmed on the domain (cookies are already `Secure`, so a plain-HTTP load would simply fail to send the session, but the redirect avoids confusion).

### Verified as handled well (so they are not re-flagged)
- No SQL injection found. PDO with `ATTR_EMULATE_PREPARES=false`; every dynamic SQL fragment interpolates only `(int)`-cast or `array_map('intval', ...)` values.
- IDOR checks present on `parent.php`, `share.php` (single 404 to avoid enumeration), `tasks.php`, `data.php`, `balance.php`, `tickets.php` (`WHERE user_id = me`), and `image.php` (traversal-safe + guardian check).
- Admin endpoints run after `rt_require_admin`; `points_grant` clamps to +/-10000; admin file ops are constrained within `uploads/`.
- DB error responses are generic (`db_failed`/`config_missing`), no stack traces or secrets.
- `tasks.php` is the model to copy: real server-side role gates, conditional state transitions returning 409 on races, and server-side point clamping.

## 3. Best practices and code quality

Overall quality is high for a framework-free codebase: thorough i18n (en/ru/lv complete across sampled modules including plurals), consistent `.catch` + toast error handling, `alive`/`busy` race guards, and useful rationale comments.

| ID | Severity | Where | Finding | Recommendation |

## 5. Notes on scope

This audit read all of `app/core` and `app/api` shared includes directly, plus the security-sensitive endpoints and modules, the schema, and migrations. It did not execute the app or test against a live server, and it could not see the production `config.php` (which determines `single_user` and `api_token`, central to SEC-3). The one cross-review disagreement (CSRF severity) was resolved against source in favor of the lower rating because `SameSite=Lax` is set (SEC-13). The strong items above are cited to specific lines and were each confirmed.
