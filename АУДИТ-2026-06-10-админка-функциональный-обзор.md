# АУДИТ 2026-06-10 — Мастер-админка: функциональный обзор

> Объект: `app/admin.html` (863 строки, SPA) + `app/api/admin.php` (936 строк, 45 операций), состояние v2026.06.10.11.
> Фокус заказчика: **функциональность и покрытие всех сценариев**; визуальный дизайн — не приоритет.
> Связанные документы: `ПЛАН-мастер-админка.md` (спецификация v1), `АУДИТ-2026-06-09-архитектура-и-безопасность.md`.
> Статус: обзор и рекомендации. Ничего из этого ещё НЕ реализовано.

## 1. Вердикт

Крепкая v1: 11 разделов закрывают базовый CRUD по всем сущностям, гейт на сервере на каждой операции,
последовательные паттерны (свои диалоги вместо native, pmGuard от менеджеров паролей, серверная пагинация
users/logs, политика приватности картинок). Для одного админа и десятка семей — работает.

Пять системных пробелов, из которых вытекает почти всё остальное:

1. **Разделы — силосы.** Ни одна сущность не ссылается на другую: из тикета нельзя попасть к автору,
   из пользователя — к его записям/очкам/событиям/семье. Любое расследование «что случилось у этого ребёнка» —
   это 5 разделов с ручным сопоставлением по нику.
2. **Тихое усечение списков.** invites (200), tickets (200), records (300), points (300), mail_log (100) —
   без пагинации и без индикатора «показано не всё». Админ уверен, что видит всё, но это не так.
3. **Нет проактивных сигналов.** Новый тикет виден, только если зайти в панель и в раздел. Ни бейджей в навигации,
   ни авто-обновления, ни письма/пуша админу. Время реакции на проблему ребёнка = «когда админ случайно зашёл».
4. **Невидимые данные.** Контент-браузер покрывает только `module_data` + `wishlist_items`. Dedicated-таблицы —
   `chat_*`, `tasks`, расписания ухода (walk care), `wishlist_share_*`, `notifications`, `push_subs`,
   `uploaded_files`, `auth_attempts`, `sessions`, `password_resets` — админу не видны вообще.
5. **Нет путей восстановления.** Мягко удалённые пользователи и записи исчезают из UI навсегда (восстановление —
   только SQL), одноразовые пароли живут 2,4 секунды в тосте, перед необратимым удалением семьи нет экспорта.

## 2. Функциональные баги (чинить до новых фич)

| # | Где | Что не так | Как чинить |
|---|---|---|---|
| Б1 | Письма → получатель | `mail`-вью берёт `api('users')` — это **первая страница (50)**. При >50 пользователях родителей со 2-й страницы в выпадайке нет: написать им письмо невозможно | Отдельный op `mail_recipients` (все родители с email, без пагинации) или поиск-селект |
| Б2 | Вся панель | Истёкшая сессия: каждый клик — тост «⚠ 401», панель «мёртвая», редиректа на вход нет | В `api()` ловить статус 401/403 → `boot()` с сообщением «сессия истекла» |
| Б3 | Создание/сброс пароля, привязка родителя | Одноразовый пароль показывается **только в тосте 2,4 с**. Не успел записать — сбрасывай заново | Модалка с паролем и кнопкой «Скопировать» (секрет показывается до явного закрытия) |
| Б4 | Все кнопки действий | На время запроса кнопки не блокируются → дабл-клик = двойное начисление очков, двойной ответ в тикете, двойное создание пользователя | Хелпер: disable + спиннер на время fetch (одно место — обёртка вокруг `api()`) |
| Б5 | Мобильный | Таблицы по 7–8 колонок без горизонтального скролла — на телефоне сжимаются в кашу. Ночная модерация с телефона невозможна | Обернуть все `<table>` в `.twrap{overflow-x:auto}` (одна строка CSS + правка V()) |
| Б6 | Семьи | **Сменить владельца семьи нельзя**: `parent_detach` владельца запрещён, а op `transfer_owner` (упомянут в комментарии admin.php:450) не реализован. Владелец ушёл → только SQL | Реализовать `transfer_owner {family_id, new_owner_id}` + кнопка в карточке семьи |
| Б7 | Invites / Tickets / Records / Points / Mail-log | Лимиты 100–300 без пагинации и без «показано N из M» | Серверная пагинация по образцу `users`/`logs` (паттерн уже есть) |
| Б8 | Контент | API `records` принимает `user_id`-фильтр, UI его **не выводит** | Добавить фильтр-селект владельца над таблицей |
| Б9 | Журнал | Фильтр по пользователю — голый числовой ID руками (паттерн пикера уже есть: `eligible_parents`, `askPickFamily`) | Пикер пользователя с поиском |
| Б10 | Диалоги | `askConfirm`: Enter не подтверждает (Esc работает) | Биндинг Enter → OK; в textarea-формах Ctrl+Enter → отправить |

## 3. Пропущенные фичи по разделам

### 3.1 Дашборд
- **Счётчики некликабельны** — «Тикеты открыты: 3» должен открывать раздел тикетов с фильтром.
- **Нет панели здоровья системы**: драйвер почты (виден только в «Письмах»), статус веб-пуша (VAPID настроен?),
  размер БД и крупнейших таблиц (`events` растёт вечно — by design), занятое место в `uploads/`,
  версия приложения + уровень миграций, время последнего `care_due`-прогона.
- **Нет ленты «требует внимания»**: новые тикеты, ошибки почты, «зависшие» проверки в «Найти предмет»
  (ребёнок ждёт, родитель не проверяет N дней), спящие дети (нет событий 14 дней), аномально большие
  строки `module_data` (детектор inline-base64 — прецедент 2026-06-10 с фото >64 КБ был бы виден заранее).
- Спарклайн активности: нет разбивки по модулям, нет выбора периода, значения только в title.
- Нет авто-обновления (см. §5: лёгкий op `badges` раз в 60 с).

### 3.2 Пользователи
- Фильтры: роль / статус / админ / семья (сейчас только текстовый поиск).
- Сортировка по колонкам: последний вход (поиск спящих), записи, дата создания (колонки «создан» вообще нет).
- **Карточка пользователя** — главный недостающий экран, хаб всего раздела (см. §5.1).
- **Сессии**: список активных сессий пользователя + «разлогинить везде» без блокировки (сейчас сессии сносятся только блоком).
- **История входов**: таблица `auth_attempts` (миграция 030) уже пишется — показать неудачные попытки по аккаунту.
- **Смена email родителя**: опечатка при регистрации → сейчас тупик (user_update меняет только ник).
- **Показ удалённых + восстановление**: user_delete мягкий, но удалённые исчезают из UI навсегда.
- Просмотр пользовательских настроек плиток (user_prefs / tile_hidden) — диагностика «у ребёнка пропала плитка».
- Экспорт CSV текущей выборки.

### 3.3 Семьи
- Поиск/фильтр (забаненные, пустые, без родителя) и пагинация — сейчас грузятся все.
- **Передача владения** (Б6).
- Создать семью напрямую; слить две семьи (родители поженились / дубль).
- Статистика семьи: период фиксирован (7 дней + 15 событий) — нужен выбор диапазона; члены семьи не кликабельны;
  нет ссылки «все события семьи» (в `logs` нет фильтра по семье — добавить).
- **Экспорт архива семьи (JSON)** — и отдельной кнопкой, и **автоматически перед `family_delete`**
  (сейчас необратимое удаление не оставляет ничего, кроме сводного лога).

### 3.4 Приглашения
- Пагинация/фильтры/поиск (Б7).
- **Скопировать ссылку приглашения** и **переотправить письмо** — если письмо не дошло, сейчас приглашение
  можно только отозвать и создать пользователя заново.
- Создать приглашение из админки (сейчас только через создание пользователя).

### 3.5 Тикеты
- **Оповещение админу о новом тикете** (email через mail-реестр + веб-пуш) — самая важная фича раздела:
  сейчас ребёнок с проблемой ждёт, пока админ случайно откроет панель.
- Пагинация (Б7), поиск по тексту/автору, фильтр по источнику (module:X / настройки), фильтр «только непрочитанные».
- Ссылка автор → карточка пользователя/семьи.
- Канонические ответы (шаблоны на en/ru/lv — ребёнок может писать на любом из 3 языков).
- Внутренние заметки админа на тикете (автору не видны).
- Ctrl+Enter в поле ответа; авто-обновление открытого треда; массовое закрытие.
- (Политика) вложения-фото в тикетах — ребёнку проще показать проблему, чем описать.

### 3.6 Контент модулей
- Пагинация + фильтры: владелец (Б8), статус, дата.
- Серверный поиск по JSON (`data LIKE ?`) — найти запись по слову.
- **Редактирование `status`** отдельным контролом: сейчас правится только `data`, хотя API умеет status
  (статусы wishlist want/think/bought, friends open/secret из админки не сменить).
- У wishlist — правка title/note (сейчас только удаление записи/картинки).
- **Браузер удалённых + восстановление** (record_delete мягкий — UI восстановления нет).
- Жёсткое удаление одной записи (приватность: стереть бесследно без удаления семьи).
- **Dedicated-таблицы не видны** (§1.4): chat, tasks, care-расписания, шары виш-листа — расширить
  админ-контракт модулей: модуль декларирует свои датасеты и отдаёт их админке через свой api.php
  (факты, без приватного содержимого — по той же политике, что картинки).
- Типизированные редакторы по admin-контракту — известный осознанный долг v1, подтверждаю приоритет ниже остального.

### 3.7 Очки
- Фильтр по ребёнку + период + пагинация (API уже умеет user_id, UI — нет).
- Разбивка «откуда очки» по модулям (guess/teeth/walk/snake/tasks/admin) за период — сейчас только общий леджер.
- **Ремонт стриков**: винстрик/пунктстрик живут в meta банка — при сбое админ не может починить (и родитель не может).
- Ссылка из транзакции на источник (запись/событие), где она есть.
- Детектор дублей claim'ов (идемпотентность walk_done и пр.) — отчёт «подозрительные двойные начисления».

### 3.8 Письма
- **Б1 (получатели > 50) — критично.**
- Рассылка всем родителям (или выбранным семьям) с учётом языка получателя — анонс «вышел новый модуль».
  Сейчас канал «всем» отсутствует.
- Предпросмотр стандартных шаблонов (en/ru/lv, образцовые данные) — сейчас только список placeholder'ов.
- Счётчик ошибок отправки на дашборде + «переотправить» из журнала.
- Фильтры журнала почты (получатель/тип/статус) + пагинация (Б7).
- Кнопка «тестовое письмо себе» (mail_test.php существует, но из панели не доступен — связать).

### 3.9 Приложения (реестр)
- **Диагностика эффективной видимости**: «почему у Артёма нет плитки X?» = глобальный enabled ×
  родительские скрытия (parent_hidden) × пользовательские (tile_hidden/prefs). Сейчас админ видит только глобальный слой.
- Здоровье модулей: последнее событие по каждому (детектор «мёртвых» модулей), дрейф версии реестра vs module.json.
- Сверка реестр ↔ файловая система (папка модуля есть, в реестре нет — и наоборот).

### 3.10 Журнал
- Фильтры: тип события, семья, диапазон дат, свободный текст; пресеты «только действия админов», «без шума» (sync/heartbeat).
- Живой хвост (авто-добавление новых событий при открытом разделе).
- Деталка события (модалка с полным meta, а не усечённый JSON в строке).
- Экспорт выборки (CSV/JSON).
- Метрика размера: счётчик строк/МБ таблицы events на дашборде (логи «навсегда» — рост надо видеть).

### 3.11 Админы и баны
- **Просмотр auth_attempts**: неудачные входы, по аккаунтам и IP, активные локауты — таблица уже есть, UI нет.
- Сессии админов + «разлогинить мои остальные сессии».
- **Отдельное управление email-банами**: добавить бан по email без семьи; снять только email-бан
  (сейчас — лишь вместе с family_unban, известное допущение v1).
- Чекер «этот email забанен?»: ввод адреса → sha256 → сравнение со стоп-листом (адрес не сохраняется).
- 2FA (TOTP) для админских аккаунтов — это детские данные; вес — после остального.
- Пагинация банов (Б7).

## 5. Сквозной макет — «ультра-функциональный» каркас

Десять правил, применимых ко всем разделам (это и есть «layout improvements»; косметика не трогается):

1. **Hash-роутинг**: `#/users?q=…&page=2`, `#/family/3`, `#/ticket/15`, `#/user/7/points`.
   Даёт: рабочую кнопку «Назад» (сейчас она выкидывает из панели), закладки, ссылки коллеге-админу,
   несколько вкладок. localStorage-память раздела остаётся как фолбэк.
2. **Карточка пользователя — хаб** (§5.1). Самый дешёвый большой выигрыш: почти все API уже принимают
   `user_id` — это в основном клиентская сборка.
3. **Всё — ссылка.** Любое имя пользователя/семьи/номера тикета в любом разделе ведёт на карточку.
4. **Верхняя панель**: глобальный поиск (Cmd+K: пользователи/семьи/тикеты по одному вводу),
   живые бейджи (тикеты-новые, ошибки почты, ждущие проверки find), кнопка «обновить раздел».
5. **Бейджи в навигации** + лёгкий op `badges` (только счётчики) раз в 60 с. Навигация по клавишам 1–9, `/` — фокус в поиск.
6. **Таблицы**: липкая шапка, сортировка по клику на колонку, строка фильтров, серверная пагинация везде,
   горизонтальный скролл на узких экранах (Б5), кнопка «CSV» на каждой таблице.
7. **Обратная связь действий**: кнопка блокируется на время запроса (Б4); после операции — что изменилось
   («баланс: 120 → 130»), а не голое «Готово».
8. **Секреты — в модалке с «Скопировать»** (Б3), не в тосте.
9. **Единая шкала подтверждений**: обычное действие — confirm с Enter; необратимое — типизированное слово
   (как УДАЛИТЬ у семьи; распространить на жёсткие удаления).
10. **401 → экран входа** (Б2), ошибки — не только тостом, а закрепляемой полосой с «копировать детали»
    (коды ошибок сервера уже отдаются — прецедент find/409).

### 5.1 Карточка пользователя (новый экран `#/user/<id>`)

```
├ Вкладки:
│  Обзор     — счётчики (записи по модулям, картинки, очки, события 7/30д) + последние события
│  Записи    — контент-браузер с предустановленным фильтром владельца (Б8)
│  Очки      — леджер ребёнка + начислить + стрики
│  События   — журнал с фильтром user_id

└ Всё на существующих операциях; новые op: sessions_list/sessions_revoke, auth_log, user_set_email
```

## 6. Что НЕ трогать (осознанные решения, не баги)

- Приватность картинок (содержимое и пути не отдаются) — фундамент, распространить и на новые разделы (Хранилище — только факты).
- Generic-JSON редактор записей — известный долг v1; типизированные формы по admin-контракту — после каркаса §5.
- Админка только на русском — приемлемо (один админ); шаблоны ответов в тикетах при этом нужны на en/ru/lv.
- Email-бан по хэшу не остановит другой адрес — осознанный компромисс, зафиксирован в плане.
- Письмо-приглашение с паролем 1234 текстом — осознанный компромисс (обязательная смена при входе).

## 7. Приоритеты

| Волна | Состав | Почему |
|---|---|---|
| **P0 — баги** | Б1–Б10 (§2) | Б1 ломает письма при росте; Б2–Б5 — ежедневное трение; Б6 — тупик без SQL; Б7 — тихая потеря данных из вида |
| **P1 — каркас** | Hash-роутинг, карточка пользователя, кросс-ссылки, бейджи+оповещение о тикетах (email админу), глобальный поиск, сортировка таблиц | Превращает набор CRUD-экранов в инструмент расследования; оповещение о тикетах — прямой эффект для детей |
| **P2 — надзор** | Разделы Хранилище/Оповещения/Безопасность/Система (§4), экспорт семьи + авто-снимок перед удалением, transfer_owner UI, диагностика видимости плиток, фильтры журнала | Закрывает «слепые зоны» и необратимости |
| **P3 — комфорт** | Типизированные редакторы, рассылки, канонические ответы, ремонт стриков, чат-надзор (после решения политики), 2FA, live-tail журнала | Ценно, но не блокирует ежедневную работу |

> Оценка объёма P0: почти всё — мелкие правки admin.html + 1–2 новых op (mail_recipients, transfer_owner)
> и распространение существующего паттерна пагинации. P1 — в основном клиентская работа поверх готовых API.
