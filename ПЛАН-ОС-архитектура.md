# RobTop как семейная мини-ОС: мастер-план рефакторинга

**Статус:** предложение (анализ + спека + фазовый план). Код не трогаем до твоего «ок».
**Дата:** 2026-06-08 · **Версия приложения на момент анализа:** v2026.06.08.80
**Этот документ вбирает в себя** прежний `ГАЙД-UI-guardrails.md` (становится разделом «Сервис 3: UI-рамка»). После принятия мастер-плана UI-гайд можно пометить как «слит сюда».

---

## TL;DR (для занятых)

RobTop уже фактически операционная система: оболочка (`core/shell.js` + `loader.js`) грузит и монтирует приложения, SDK (`core/sdk.js`) служит единственным «системным вызовом» между приложением и ядром, `data.php` даёт каждому приложению свою «файловую систему», таблица `events` это системный журнал, реестр модулей плюс магазин это менеджер пакетов, аккаунты и роли это пользователи и права. Это хорошая основа.

Чего не хватает для зрелой «семейной ОС»: единых **гардрейлов** (правил, которые приложение не может обойти), вынесенных в ядро **переиспользуемых функций** (сейчас `esc`, иконки, работа с датами и фото-пайплайн скопированы от 7 до 16 раз), и нескольких **системных сервисов**, которые сегодня живут как договорённости, а не как строгий контракт (очки и транзакции, оповещения, обмен сообщениями, права, медиа, жизненный цикл).

План делает три вещи. Первое: формализует **контракт ядра** (одна нерушимая граница: приложение ходит в мир только через `sdk`). Второе: превращает каждую сквозную тему в **строго регламентированный сервис** с гардрейлами и проверкой. Третье: задаёт **безопасный фазовый рефакторинг** (каждая фаза самостоятельна, заканчивается рабочим приложением, своим коммитом и бампом версии; `git lock` не используем никогда).

Главный принцип всего документа: **простейшее решение, удовлетворяющее требованию**. Мы не строим настоящую ОС с ядром и планировщиком. Мы наводим порядок в том, что уже есть, и закрываем дыры, через которые «иногда что-то ломается».

# Часть I. Модель ОС (как смотреть на RobTop)

## 1.1 Соответствие «ОС → RobTop»

| Понятие ОС | Что это в RobTop | Файл |
|---|---|---|
| Ядро (boot, экраны, сессия) | Оболочка | `core/shell.js` |
| Диспетчер процессов (запуск/остановка приложения) | Загрузчик: `register`, `open`, `close`, lifecycle `mount`/`unmount`/`link`/`refresh` | `core/loader.js` |
| Системные вызовы (syscalls) | SDK: единственный мост приложение↔ядро | `core/sdk.js` |
| Приложения (процессы) | Модули: родные `modules/<id>/`, установленные `apps/<id>/` | `modules/*`, `apps/*` |
| Манифест приложения | `module.json` (id, имя, иконка, тема, роли, точка входа) | `modules/<id>/module.json` |
| Файловая система (namespace на приложение) | Универсальный стор `module_data` через `data.php` | `api/data.php`, `sdk.data` |
| Системный журнал (audit log) | Таблица `events`, `sdk.events.track` | `api/data.php`, `api/action.php` |
| Менеджер пакетов / установщик | Реестр `registry.php` + магазин `store/*` (вкл/выкл, порядок, установка бандла) | `api/registry.php`, `api/store/*` |
| Пользователи и права | Аккаунты, роли parent/child, скоуп выбранного ребёнка | `api/_accounts.php`, `sdk.role`, `sdk.can` |
| Уведомления ОС | Центр оповещений + Web Push | `core/notify.js`, `api/notify.php`, `api/_push.php` |
| Системные деньги (внутренняя экономика) | Банк-леджер очков, винстрик, пунктстрик | `sdk.points`, `api/data.php` (коллекция `bank/points`) |
| Демон фоновой синхронизации | Поллинг `sync.php` + хук `refresh()` | `api/sync.php`, `core/loader.js` |
| Локаль / регион | i18n en/ru/lv | `core/i18n.js`, `core/strings.js` |

## 1.2 Нерушимая граница ядра (Гардрейл 0)

Это единственное правило, на котором держится всё остальное. Оно уже записано в шапке `core/sdk.js`, мы лишь возводим его в ранг закона:

> **Приложение взаимодействует с внешним миром ТОЛЬКО через `sdk`.** Оно не трогает DOM за пределами своего `root`, не вызывает `fetch` напрямую, не читает чужой `localStorage`, не лезет в глобальные переменные ядра. Всё (данные, очки, оповещения, медиа, UI, тема, права, время) приходит и уходит через `sdk`.

Почему это и есть «операционная система»: ядро может менять реализацию (перейти с MySQL на другой стор, заменить фото-пайплайн, переписать рамку экрана), не трогая ни одного приложения, пока контракт `sdk` стабилен. И наоборот: ни одно приложение не может сломать другое или само ядро, потому что у него просто нет доступа за границу.

Сегодня граница соблюдается **почти** везде. Исключения, которые план приводит к норме: два модуля держат собственный `api.php` (`chat`, `wishlist`), фото-пайплайн и экранирование строк живут внутри модулей копиями, часть модулей вешает слушатели на `root` и не снимает их при `unmount`.

# Часть II. Системные сервисы (что должно быть строго регламентировано)

## Сервис 1. Очки и транзакции (P0)

**Что есть.** `sdk.points.add/get/summary` поверх банк-леджера (коллекция `bank/points` в универсальном сторе). Каждая транзакция это строка `{n, reason, src, kind, note?}`. Винстрик и пунктстрик не хранятся, а **выводятся из леджера** при каждом чтении (нечему рассинхронизироваться). Это сильное и правильное решение, его не трогаем. Политика тарифов в `ГАЙД-очки.md`.

**Где риск.**
1. **Баланс считается на клиенте.** `bankSummary()` тянет все строки и суммирует `n` в браузере (`core/sdk.js`, функция `bankSummary`). Для детского приложения с магазином (тратами реальных «очков») это значит, что баланс это мнение клиента, а не сервера. Достаточно открыть консоль, чтобы «дорисовать» себе очки.
2. **Типы транзакций (`kind`) не закрыты.** Список видов (`win/loss/task_done/daily_bonus/parent/manual/bonus/spend/...`) живёт в комментариях. Новый модуль может прислать любой `kind`, и он молча запишется.
3. **Начислять и списывать может любой модуль.** `sdk.points.add` доступен всем. Сейчас это терпимо (приложений мало), но для «магазина приложений» это дыра: установленное приложение не должно само себе начислять очки.

**Регламент (контракт сервиса).**
- **Сервер единственный авторитет баланса.** Завести серверный пересчёт: эндпоинт отдаёт `{balance, streak, plusStreak}`, посчитанные в PHP по леджеру (тот же алгоритм, что на клиенте сейчас). Клиент показывает, но не «считает истину». Минимальная правка, движок логики уже есть, переносим суммирование на сервер.
- **`kind` это закрытый перечень.** Объявить enum в одном месте (PHP-константа плюс зеркало в `sdk`), сервер отклоняет неизвестный `kind` кодом `err.bank_kind`. Источник правды по семантике остаётся `ГАЙД-очки.md`.
- **Право начисления через возможности (см. Сервис 6).** Только приложение с capability `points.write` в манифесте может звать `sdk.points.add`. Игры, которые должны давать очки (зубы, угадайка, прогулка, найди предмет), его декларируют; остальные нет. Списание (`spend`) только у `shop`.
- **Идемпотентность награды за задание.** Поток `task_done` уже защищён порядком «сначала очки, потом статус» и серверными условными переходами (409 на гонке). Это фиксируем в каноне как обязательное свойство, чтобы будущие «начисляющие» потоки повторяли паттерн.

**Гардрейл.** Приложение не пишет в леджер напрямую (нет своего `api.php` для очков), не придумывает свои `kind`, не показывает баланс из локального счётчика. Только `sdk.points`.

## Сервис 2. Оповещения (P0)

**Что есть.** Центр оповещений (`core/notify.js`: колокольчик, бейдж, баннеры), серверная отправка `api/notify.php`, тексты `api/_ntf_text.php`, Web Push `api/_push.php`. Контракт `sdk.notify.send(to, type, opts)` уже «fire-and-forget, никогда не reject». Канон `ГАЙД-оповещения.md`. Это уже хорошо регламентировано, доводим до строгого.

**Где риск.**
1. **Можно прислать «сырой текст».** В контракте есть `params.text`-фолбэк. Это удобно на отладке, но это лазейка мимо реестра типов и мимо переводов: появится оповещение без ключа и без 3 языков.
2. **Легко завести тип без перевода.** Текст обязан иметь ключ `ntf.ev.<модуль>.<type>` на en/ru/lv. Сейчас это правило словесное, проверки нет.
3. **Адресация и ссылки свободной формы.** `to` это `parents|child|family`, `link` это `{module,item}` либо `{view,id}`. Без валидации опечатка в `to` или битая ссылка из оповещения тихо «никуда не ведёт».

**Регламент.**
- **Только зарегистрированные типы.** Каждый тип объявляется один раз (ключ в `core/notify.js` плюс серверный шаблон), с обязательными переводами на 3 языка. Отправка незарегистрированного типа отклоняется (код `err.ntf_type`). `params.text`-фолбэк оставить только для демо-режима.
- **Реестр оповещений как часть манифеста.** Модуль перечисляет свои типы оповещений в `module.json` (или в `messages`), ядро при сборке проверяет, что для каждого есть 3 перевода.
- **Валидация адреса и ссылки.** `to` из закрытого перечня; `link.module` обязан существовать в реестре; иначе оповещение шлётся без ссылки, но не «битым».
- **Один шлёт, многие не дублируют.** Правило «движок шлёт сам, модули не дублируют» (как у заданий: `task_*` шлёт сам `sdk.tasks`) распространить на все системные потоки. Модуль шлёт оповещение только про свою доменную суть.

**Гардрейл.** Никакого «сырого» текста в проде, никаких незарегистрированных типов, у каждого типа 3 перевода, `link` указывает на существующий модуль.

## Сервис 3. UI-рамка, оконный менеджер (P0), вобрано из ГАЙД-UI-guardrails

Это бывший `ГАЙД-UI-guardrails.md` целиком. Здесь сжатая суть; полная аргументация с доказательствами остаётся в том файле до момента слияния.

**Проблема.** Каждый модуль рисует свою шапку с нуля (16 копий одного компонента, классы `.bk-header/.ch-head/.md-header/...`), сам угадывает резерв снизу под HUD (числа 140/150/нет вразнобой), а магические `bottom` (70/86/96/104/120) раскиданы по экрану. Корень бед: анимация `.view.active{transform}` создаёт containing block и ломает `position:fixed` внутри модулей (из-за этого Чат и родительский таббар пришлось выселять в `<body>`), плюс везде `100vh` вместо `100dvh`, отчего низ «дышит» на мобильных.

**Решение (guardrails).**
- **Токены высот и safe-area** в `core/ui.css :root` (один источник истины: `--chrome-top`, `--hdr-h`, `--kidbar-h`, `--tabbar-h`, производные `--*-total`, полки `--float-*`). Все смещения считаются от токенов, никаких чисел по месту.
- **Единый фрейм** `sdk.ui.frame({title, back, actions, bar})`: ядро строит шапку (‹ назад · заголовок · действия) и прокручиваемое тело, модуль наполняет только тело. Фрейм фиксированный, `100dvh`, на flexbox, без `transform` на всю секцию, поэтому ловушка containing block исчезает.
- **Реестр иконок в ядре** (`BACK/PLUS/STATS/PARENT/CAM/...`): один источник вместо дублей SVG в каждом модуле.
- **Детский нижний бар** `.rt-bar` вместо декоративного HUD: «🏠 Домой · ⭐ очки · ⚙». Полноэкранная сцена (Змейка) может его скрыть (`sdk.ui.bar({hidden:true})`).
- **`fixed` внутри модуля запрещён.** Плавающие элементы (FAB, корзина, тосты) рисует ядро в `<body>` через `sdk`.

**Статус.** `sdk.ui.frame` уже реализован, на нём `mood` и `bank` (пилот, фаза 2). Токены частично введены (фаза 0, по памяти проекта). Остальные ~14 модулей ещё на старых `*-header` и старом `sdk.ui.hud`. Фазы миграции встроены в общий план (Часть V).

**Гардрейл.** Новый или изменённый модуль обязан использовать `sdk.ui.frame`, не объявлять свою шапку, не использовать `position:fixed`, не писать магические отступы (только токены).

## Сервис 4. Обмен сообщениями / Чат (P1)

**Что есть.** Чат это привилегированный модуль (`modules/chat/`) со своим `api.php`, со слоем `#chApp` в `<body>` (обход containing block), с галочками доставки и прочтения, отметками `seen_at` в `sync.php`, и правилом «родитель видит всю переписку семьи». На каждое сообщение шлётся оповещение.

**Почему это сервис, а не «просто модуль».** Это единственный канал общения в семье. У него системные свойства: родительский надзор (модерация и видимость), доставка и прочтение, интеграция с центром оповещений и с живой синхронизацией. Если завтра появится второе приложение, которому нужно «написать в семейный чат» (например, «магазин» шлёт ребёнку сообщение «родитель одобрил покупку»), оно должно делать это через сервис, а не лезть в чужой `api.php`.

**Где риск.** Контракт чата нигде не описан как сервис: правила видимости, модерации, лимиты, allowlist загрузок, гонка ключей доставки живут в коде одного модуля. Слой `#chApp` и `<body>`-таббар это следствие UI-проблемы из Сервиса 3.

**Регламент (простейший достаточный).**
- **Описать чат как привилегированный системный сервис** с явным контрактом в каноне: модель сообщения, видимость (родитель видит всё, дети только свою переписку), доставка/прочтение, лимиты и allowlist вложений, оповещение на каждое сообщение. Это документ, не переписывание.
- **Тонкий `sdk.messaging` (по желанию, P2).** Если понадобится, дать узкий вызов «послать системное сообщение в семейный чат от имени системы» для других приложений, чтобы они не трогали `chat/api.php`.
- **Вернуть чат в общую рамку** на фазе 5 UI-рамки (когда фрейм фиксированный без `transform`, костыль `#chApp` может стать не нужен). Делать только если смоук на iOS Safari чистый, иначе оставить как есть.

**Гардрейл.** Чужие приложения не обращаются к `chat/api.php` напрямую; общение с семейным чатом только через сервис. Правила видимости и модерации описаны в каноне и не дублируются в других модулях.

## Сервис 5. Данные и «файловая система» (P1)

**Что есть.** `sdk.data.list/get/create/update/move/favorite/remove/restore` поверх одной таблицы `module_data` через `api/data.php`. Namespace по модулю и коллекции, мягкое удаление (`deletedAt`), полный паритет с демо-режимом (localStorage). Скоуп выбранного родителем ребёнка централизован (`parentChild` в `sdk`, проверка прав на сервере). Это сильный, правильный «системный диск».

**Где риск.**
1. **Схема данных это «чёрный ящик» JSON.** В `data` лежит произвольный объект. Удобно, но нет ни валидации, ни описания формы, и родительский дашборд / админка вынуждены угадывать поля.
2. **Два модуля в обход держат свой `api.php`** (`chat`, `wishlist`). Для них есть причины (чат сложный, у виш-листа шаринг), но это исключение должно быть осознанным и редким.

**Регламент.**
- **Лёгкая декларация формы в манифесте.** Модуль в `module.json` может (не обязан) описать форму своих коллекций простым списком полей и типов. Это не строгая валидация на старте, а подсказка для дашборда/админки и основа для будущей проверки. Простейшее решение: текстовое описание, без схем-валидаторов.
- **Свой `api.php` это привилегия.** Зафиксировать правило: модуль использует `sdk.data`; собственный серверный код это исключение, требующее обоснования (записать в каноне, кому и почему разрешено). Цель не запретить, а сделать осознанным.

**Гардрейл.** По умолчанию данные только через `sdk.data`. Скоуп ребёнка не трогаем руками (его ставит ядро). Новый `api.php` у модуля только с явным обоснованием в документации.

## Сервис 6. Идентичность, роли и права (P1)

**Что есть.** `sdk.role` (`child|parent`), `sdk.can(action)` (роли на действие из `meta.roles`), `sdk.scopeChild()` (в чьём скоупе работает родитель), серверные гейты (`rt_require_parent`, `rt_can_manage_child` и т. п.). Роль даёт сессия аккаунта.

**Где риск.** `can()` задаётся точечно через `meta.roles[action]` с дефолтами в коде SDK. Нет единой модели «что приложению вообще позволено делать в системе» (а это и есть суть ОС: приложение объявляет свои потребности, система их выдаёт или нет).

**Регламент (модель возможностей, capabilities).**
- **Манифест объявляет возможности.** В `module.json` поле `capabilities`: например `["points.write","media.upload","notify.send","family.read"]`. Ядро при создании `sdk` отдаёт модулю только разрешённые вызовы (или они становятся no-op с понятной ошибкой).
- **Роли остаются для UI-действий** (`read/edit/parent`), возможности для системных ресурсов. Две оси: «кто ты» (роль) и «что приложению дано» (capability).
- **Простейшее внедрение.** На старте capability это просто документированный список плюс проверка в `sdk` на самые чувствительные вызовы (`points.write`, `spend`). Полную «песочницу» не строим.

**Гардрейл.** Ребёнок не выполняет родительские операции (сервер проверяет, не только UI). Приложение не зовёт системный ресурс, который не объявило в манифесте.

## Сервис 7. Медиа и фото (P0 по приватности, P1 по дублям)

**Что есть.** `sdk.media.upload(dataUrl, kind)` грузит на сервер, в БД только путь (`api/upload.php`, хранилище за абстракцией `_storage.php`, на будущее S3/R2). Но **сам пайплайн выбора и ресайза фото скопирован в 7 модулях** (`chat, find, mood, rating, shop, walk, wishlist`): один и тот же `FileReader` → `Image` → `canvas` → `toDataURL(0.82)` → `upload`.

**Где риск.**
1. **7 копий одного кода.** Поменяем качество, максимальный размер или формат, и правим 7 файлов.
2. **Приватность не enforced (детское приложение, это важно).** Папка `uploads/` отдаётся веб-сервером как статика без проверки прав. Правило «фото видят только ребёнок и его прямой родитель, временный родитель никогда» задокументировано, но не действует на уровне отдачи файла (разрыв №8 в `АУДИТ-приложения.md`).
3. **Детская безопасность съёмки.** Для «Найди предмет» правило «только живая камера, галерея и интернет запрещены». Эта политика тоже живёт в коде модуля.

**Регламент.**
- **Единый `sdk.media.pick({onPhoto, source})`.** Один вызов делает выбор → ресайз → загрузку → возвращает путь. Источник (`camera`/`gallery`) и лимиты задаёт ядро по политике, а не модуль. 7 копий схлопываются в один сервис.
- **Отдающий PHP-эндпоинт с проверкой прав** (`rt_can_view_images`) плюс `deny` на прямой доступ к папке. Это закрывает приватность по-настоящему. P0, потому что это про детские фото.
- **Политика съёмки в ядре.** «Только камера» это параметр `source:"camera"` сервиса, а не самописный код модуля.

**Гардрейл.** Модуль не реализует свой ресайз/загрузку, только `sdk.media.pick`. Доступ к чужим фото невозможен в обход проверки прав.

## Сервис 9. Жизненный цикл и события (P1)

**Что есть.** Контракт `register({id, mount, unmount, link?, refresh?, messages?})`. `mount(root, sdk)` строит экран, `unmount()` убирает, `link(link, sdk)` открывает конкретный предмет из оповещения, `refresh()` подхватывает живое обновление. `sdk.events.track(type, payload)` пишет в системный журнал; `loader` автоматически логирует `opened_module`.

**Где риск.**
1. **Утечки слушателей.** Часть модулей вешает обработчик на `root` в `mount` и не снимает его в `unmount` (память проекта отмечает накопление root-листенеров у старых модулей). При повторном открытии слушатели множатся.
2. **`refresh()` внедрён не везде** (примерно 8 из 16). Модуль с чужими/общими данными без `refresh()` показывает устаревшее после живого апдейта.
3. **Чистка таймеров на совести модуля.** `mood` чистит `setInterval` в `unmount` правильно, но это снова копипаст-дисциплина, а не гарантия.

**Регламент.**
- **Авто-снимаемые слушатели.** Дать `sdk.on(el, event, fn)` и `sdk.cleanup(fn)`: ядро запоминает и само снимает всё при `unmount`. Тогда утечка невозможна по построению.
- **`refresh()` обязателен для модулей с общими данными** (чат, копилка, виш-лист, задания). Зафиксировать в контракте; проверять линтером (Часть III).
- **Стандартная телеметрия.** Перечень «осмысленных» доменных событий (политика `isDomain` родительского дашборда: без login/viewed/opened-дублей) описать в каноне, чтобы новые модули логировали единообразно.

**Гардрейл.** `unmount` обязан вернуть систему в исходное (таймеры, слушатели, плавающие элементы). Слушатели вешать через `sdk.on`. Модуль с чужими данными обязан иметь `refresh()`.

## Сервис 11. Ошибки и устойчивость (P1)

**Что есть.** `loader` оборачивает `mount` в `try/catch` и показывает тост при сбое загрузки/монтирования. Бэкенд отдаёт коды ошибок `err.<code>` (правило проекта), фронт переводит их через i18n.

**Где риск.** Ошибки уже **после** `mount` (в обработчиках, в загрузке данных) не ловятся единообразно: каждый модуль сам решает, показать ли тост. Нет «границы ошибки» приложения, нет телеметрии ошибок.

**Регламент (простейший).**
- **Единый код ошибок на фронте.** Уже есть `err.*`; свести их в один справочник и гарантировать перевод на 3 языка (как у оповещений).
- **Граница ошибки приложения.** Тонкая обёртка: непойманная ошибка в коллбэке модуля показывает стандартный детский тост («Что-то сломалось, попробуй ещё раз») и пишет событие `error` в журнал, вместо «белого экрана».
- **Сетевые сбои единообразно.** `sdk.api`/`sdk.data` уже не reject там, где модули зовут fire-and-forget; описать паттерн в каноне, чтобы новый код его повторял.

**Гардрейл.** Приложение не показывает пользователю технический текст ошибки; все ошибки через коды `err.*` с переводом. Сбой одного приложения не роняет оболочку.

## Сервис 13. Версия и релиз (P1)

**Что есть.** Маркер версии в `index.html` (`#appver` + `window.RT_VER`), cache-busting ассетов по версии, индикатор слева внизу. Правило: бамп при каждом изменении. По памяти проекта версия повторяется примерно в 11 местах плюс changelog.

**Где риск.** 11 мест это 11 шансов забыть и получить рассинхрон версии с кэшем. Чисто механическая хрупкость.

**Регламент.**
- **Один источник версии.** Свести к единственному `window.RT_VER` (или одной строке в `config`/манифесте), которым питаются все остальные точки. Где сейчас версия зашита в разметку, подставлять из единого источника при загрузке.
- **Чек-лист релиза в каноне** (что бампится, что в changelog) и проверка линтером, что версия согласована.

**Гардрейл.** Версия живёт в одном месте. Бамп плюс запись в changelog плюс запись в журнал `КОНТЕКСТ.md` обязательны на каждое изменение (правило проекта, фиксируем).

# Часть III. Гардрейлы (конституция приложения)

Это короткий список правил, которые приложение **не может обойти**. Каждое правило это «закон», нарушение которого ловится глазами на ревью или автоматом (см. линтер ниже). Гардрейлы это и есть «чтобы ничего не ломалось».

| № | Правило | Источник правды |
|---|---|---|
| G0 | Граница ядра: приложение ходит в мир только через `sdk`; не трогает DOM вне `root`, не делает `fetch` напрямую, не лезет в глобали ядра | Часть I.2 |
| G1 | UI только через `sdk.ui.frame`; нет своей шапки, нет `position:fixed`, нет магических отступов (только токены) | Сервис 3 |
| G2 | i18n: ноль хардкода пользовательских строк; каждый текст на en/ru/lv; бэкенд отдаёт `err.<code>`, не фразы | Сервис 8 |
| G3 | Очки только через `sdk.points`; `kind` из закрытого перечня; баланс с сервера; начислять может только модуль с capability `points.write` | Сервис 1 |
| G4 | Оповещения только зарегистрированных типов с ключом `ntf.ev.<мод>.<type>` на 3 языках; никакого сырого текста в проде | Сервис 2 |
| G5 | Данные только через `sdk.data`; свой `api.php` это привилегия с обоснованием в каноне | Сервис 5 |
| G6 | `unmount` возвращает систему в исходное (таймеры, слушатели, плавающие элементы); слушатели только через `sdk.on`; модуль с чужими данными имеет `refresh()` | Сервис 9 |
| G7 | Медиа только через `sdk.media.pick`; свой ресайз/загрузка запрещены; доступ к фото только через проверку прав | Сервис 7 |
| G8 | Версия из одного источника; бамп + changelog + запись в `КОНТЕКСТ.md` на каждое изменение | Сервис 13 |
| G9 | Права: ребёнок не делает родительских операций (проверяет сервер); приложение не зовёт системный ресурс, не объявленный в манифесте | Сервис 6 |
| G10 | Детская безопасность: политика съёмки (камера/галерея) и приватность фото задаёт ядро, не модуль | Сервис 7 |

## 3.1 «Definition of done» для любого модуля

Чек-лист, который проходит каждый новый или изменённый модуль перед коммитом:

1. Использует `sdk.ui.frame`, не рисует свою шапку, не использует `fixed`.
2. Ни одной хардкод-строки: всё через `sdk.t`, переводы на en/ru/lv лежат в `messages`.
3. Очки только через `sdk.points` с разрешённым `kind`; capability объявлена в манифесте.
4. Оповещения только зарегистрированных типов с 3 переводами.
5. Данные через `sdk.data` (или обоснованный `api.php`, записанный в каноне).
6. `unmount` снимает все таймеры и слушатели; слушатели заведены через `sdk.on`; есть `refresh()`, если данные общие.
7. Фото только через `sdk.media.pick`.
8. Версия сбамплена, changelog и журнал `КОНТЕКСТ.md` обновлены.
9. Смоук пройден (Часть V.10).

## 3.2 Автопроверка: линтер гардрейлов (`tools/check.js`)

Чтобы гардрейлы не держались только на дисциплине, ввести лёгкий скрипт-проверку (Node, без зависимостей), который гоняется руками перед коммитом и в CI. Он не понимает семантику, он ловит грубые нарушения по тексту:

- хардкод-строки в `module.js` (кириллица/латиница в кавычках вне `messages` и ключей) → подозрение на G2;
- `position:fixed` в `modules/*/module.css` → нарушение G1;
- `fetch(`, `XMLHttpRequest`, `localStorage` напрямую в `module.js` → подозрение на G0;
- объявление своих `*-header` в css или своего `BACK_IC/PLUS_IC/...` в js → остатки до миграции на Сервис 3;
- `addEventListener` на `root`/`document` без парного снятия и без `sdk.on` → подозрение на G6;
- рассинхрон версии между точками (G8);
- тип оповещения без 3 переводов; `kind` очков вне перечня (по простому грепу).

Линтер это «простейшее решение»: он не гарантирует корректность, но превращает молчаливые нарушения в видимые предупреждения. Это и есть практичный гардрейл для проекта одного разработчика с растянутыми во времени правками.

# Часть IV. Переиспользуемые функции (что вынести в ядро)

Прямое следствие анализа дублирования. Каждая строка это «было N копий → стало 1 в ядре».

| Что | Сейчас | Куда вынести | Подпись (примерно) |
|---|---|---|---|
| Экранирование HTML `esc` | 16 копий в модулях | `core/util.js` (или в `sdk.util`) | `esc(s)`, `escAttr(s)` |
| Реестр SVG-иконок (`BACK/PLUS/STATS/PARENT/CAM/...`) | объявлены в каждом модуле | `core/icons.js`, отдаются через `frame` и `sdk.ui.icon` | `icon(name)` |
| Фото-пайплайн (выбор + ресайз + загрузка) | 7 копий | `sdk.media.pick` | `pick({source, onPhoto})` |
| Авто-снимаемые слушатели | копипаст `addEventListener` + ручная чистка | `sdk.on`, `sdk.cleanup` | `on(el, ev, fn)` |
| Даты и окна времени (`pad2/dayKey/inWindow`) | ~10 вариантов | `sdk.time` | `dayKey(d)`, `inWindow(from,to)` |
| Относительное/человеческое время | 10 разных форматов | `sdk.fmt` (поверх i18n) | `fmt.relative(ts)`, `fmt.day(ts)` |
| Пустое состояние / карточка списка / форма-шторка | свои варианты у всех | `sdk.ui` примитивы | `ui.empty(text)`, `ui.card(...)`, `ui.formSheet(...)` |
| HUD-счётчики | заменяются детским баром | `sdk.ui.bar` | `bar({points, hidden})` |

Принцип: выносим только то, что **реально** дублируется (есть счётчик копий), и кладём за `sdk`, а не в глобальную область. Никаких «фреймворков на будущее», которые никто не использует.

# Приложение А. Контракт модуля (канон)

Единая точка истины по тому, что модуль обязан и что ему дано. Это снимок текущего `sdk` плюс предлагаемые добавления (помечены `(план)`).

**Регистрация.**
```js
RobTop.register({
  id: "mood",

  messages: { en:{...}, ru:{...}, lv:{...} }
});
```

**Манифест `module.json`** (предлагаемые поля плана помечены):
```jsonc
{
  "id": "mood",
```

## Б.1 Ф4: приватность фото — отдающий эндпоинт `api/image.php`

Проблема (разрыв №8 в `АУДИТ-приложения.md`): папка `uploads/` отдаётся веб-сервером как статика, без проверки прав. Любой, кто знает или подберёт путь, видит детское фото. Правило «фото видят только ребёнок и его прямой родитель, временный родитель никогда» задокументировано, но не действует.

Решение (простейшее): фото отдаёт PHP-эндпоинт с проверкой прав, а прямой доступ к `uploads/` закрыт.

Шаг 1. Новый файл `app/api/image.php` (логика; функцию `rt_user_id`/`rt_can_view_images` берём из `_bootstrap.php`/`_accounts.php`):
```php
<?php require __DIR__.'/_bootstrap.php';
// путь вида users/<ownerId>/<kind>/<file>; никаких .. и абсолютных путей
$rel = isset($_GET['p']) ? (string)$_GET['p'] : '';
if($rel==='' || strpos($rel,'..')!==false || $rel[0]==='/'){ http_response_code(400); exit; }
if(!preg_match('#^users/(\d+)/#', $rel, $m)){ http_response_code(400); exit; }
$ownerId = (int)$m[1];
// rt_can_view_images($ownerId): true если зритель = владелец, ИЛИ его primary-родитель
// (provisional/временный — НЕ видит); реализация рядом с rt_can_read в _accounts.php
if(!function_exists('rt_can_view_images') || !rt_can_view_images($ownerId)){ http_response_code(403); exit; }
$base = realpath(__DIR__.'/../uploads');
$full = realpath($base.'/'.$rel);
if($full===false || strpos($full,$base)!==0 || !is_file($full)){ http_response_code(404); exit; }
$mime = function_exists('mime_content_type') ? mime_content_type($full) : 'image/jpeg';
header('Content-Type: '.$mime);
header('Cache-Control: private, max-age=86400');
readfile($full); 
```

Шаг 2. Хелпер прав в `_accounts.php` (рядом с `rt_can_read`):
```php
function rt_can_view_images($ownerId){
  $uid = rt_user_id();
  if($uid && (int)$uid === (int)$ownerId) return true;        // сам ребёнок
  return rt_can_read($ownerId) && rt_is_primary_guardian($uid, $ownerId); // прямой родитель, НЕ provisional
}
```
(если `rt_is_primary_guardian` нет — добавить: SELECT по `guardianships` где type='primary' и status='active'.)

Шаг 3. Закрыть прямой доступ к `uploads/` в `app/.htaccess` (или в `uploads/.htaccess`):
```apache
RewriteRule ^uploads/ - [F]
```
(SPA-правило отдаёт `api/`, `media/` напрямую; для `uploads/` теперь deny.)

Шаг 4. Переключить ссылки на фото. Сейчас в БД лежит относительный путь (напр. `users/1/mood/abc.jpg`). Клиент строит `<img src="uploads/...">` или `background-image:url('uploads/...')`. Вариант минимального риска: оставить в БД тот же путь, а ОТДАЧУ перевести на `api/image.php?p=<path>`. Точка склейки одна — функция, которая из пути фото делает URL. Сейчас её нет (модули вставляют путь напрямую), поэтому: либо ввести хелпер `sdk.media.url(path)` (возвращает `api/image.php?p=`+encodeURIComponent(path)) и заменить им вставки `uploads/...` в модулях, либо сделать `upload.php` хранящим уже-готовый `api/image.php?p=...` (хуже: завязка на эндпоинт в данных). Рекомендуется `sdk.media.url`.

Тест на сервере: залогиниться ребёнком — своё фото видно; залогиниться его родителем — видно; чужим аккаунтом или без сессии открыть `api/image.php?p=users/1/mood/...` напрямую — 403; открыть `…/uploads/users/1/...` напрямую — 403 (deny). Проверить `php -l image.php`.

Риск отката: если что-то не так, вернуть `.htaccess` (убрать deny) и ссылки на `uploads/` — фото снова отдаются статикой (как сейчас).

## Б.2 Ф7: серверный авторитет баланса + закрытый перечень `kind` + валидация оповещений

Сейчас баланс считает КЛИЕНТ (`bankSummary` суммирует леджер в браузере). Для детской экономики с магазином это значит, что баланс это мнение клиента. Цель: сервер считает баланс тем же алгоритмом и отдаёт его; клиент показывает.

Делать на сервере, не вслепую в боевом коде. Рекомендуемый безопасный порядок:

1. Новый эндпоинт-чтение `api/balance.php` (или `op:"balance"` в `data.php`): по леджеру `module='bank', collection='points'` пользователя считает `balance = SUM(n)`, и винстрик/пунктстрик тем же алгоритмом, что в `core/sdk.js` (`bankStreakFrom`/`bankPlusStreakFrom`). Отдаёт `{balance, streak, plusStreak}`. Ничего не пишет.
2. Закрытый перечень `kind` на сервере: при записи в леджер (`data.php` op `create` для `bank/points`) проверять `data.kind` против белого списка `win|loss|task_done|daily_bonus|parent|manual|bonus|task_fail|spend|spend_refund|streak_bonus`. Неизвестный → `400 {error:"bank_kind"}`. Семантика — `ГАЙД-очки.md`.
3. Клиент: `sdk.points.summary()` начинает брать число с сервера (`balance.php`), а вывод винстрика/пунктстрика из леджера оставить как фолбэк, пока сверяешь. Переключение мягкое: сравнить серверный и клиентский баланс на нескольких аккаунтах, затем сделать сервер источником истины.
4. Валидация оповещений в `api/notify.php`: тип принимать только из зарегистрированных (есть ключ `ntf.ev.<mod>.<type>`); `params.text`-фолбэк разрешить только при demo. `to` из перечня `parents|child|family`. Неизвестный тип → `400 {error:"ntf_type"}`.
5. Справочник `err.*`: свести коды ошибок (`bank_kind`, `ntf_type`, `image_denied`, существующие) в один список, гарантировать перевод на en/ru/lv в `core/strings.js`.

Тест: `php -l` всех правленых файлов; на сервере сверить `balance.php` с клиентским подсчётом на 2-3 аккаунтах; отправка незарегистрированного типа оповещения даёт 400; запись неизвестного `kind` отклоняется. Это критично (деньги и приватность), поэтому строго на сервере, не в песочнице.
