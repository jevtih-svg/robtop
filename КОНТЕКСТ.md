# RobTop — детальный контекст проекта

> Живой документ. Обновляется при каждом значимом изменении. Последнее обновление: **2026-06-05**.
> Это главный источник правды по проекту. Перед началом любой новой сессии читать этот файл целиком.

---

## 1. Что это такое

**RobTop** — личное детское приложение для одного ребёнка (**Артём**). Это **одно единое приложение** с множеством мини-функций («микроприложений»), а не набор отдельных приложений.

- Тип: детское приложение.
- Язык интерфейса: **многоязычный** — English (по умолчанию), Русский, Latviešu. Архитектура i18n рассчитана на лёгкое добавление языков (движок `core/i18n.js` + словари `core/strings.js` и `messages` модулей; выбор языка в ⚙ Настройках). Подробности — в журнале изменений (запись «многоязычность: en/ru/lv»). Раньше было «только русский».
- Хостинг: **apps.tilley.live/robtop** (Hostinger).
- Стек: **PHP + MySQL** (так как хостинг Hostinger).
- Главный экран: **плитки** (tiles), каждая плитка = отдельная мини-функция.
- Пользуется приложением только Артём (личное приложение) + родитель в режиме просмотра.

## 2. Архитектура (ключевой принцип)

Приложение — это **оболочка (shell) + набор модулей (микроприложения)**.

Требования к архитектуре:
- Модули должны **легко добавляться и легко убираться**. Добавление новой плитки = регистрация нового модуля, без переписывания ядра.
- Модули могут быть очень разными по функциям (таймер, игра, список, камера, ИИ-генератор и т.д.).
- Приложение **собирает всю статистику в базу данных** (MySQL), чтобы потом построить экран статистики для родителя.
- Архитектура должна быть **максимально простой и функциональной**.

Предлагаемая модель (для PHP+MySQL версии):
- **Реестр модулей**: каждый модуль описывается мета-данными (id, название, иконка, цвет/тема, права ролей) и точкой входа. Главный экран строит плитки из реестра.
- **Единая таблица событий статистики** (`events`): любое действие в любом модуле пишет событие `{user, module, type, payload(JSON), created_at}`. Это даёт гибкость — родительский экран читает события и агрегирует.
- **Таблицы данных по модулям** (например, `wishlist_items`) — конкретные данные функции.
- **Роли** определяются по введённому PIN (см. раздел 4).

Реализация (первый релиз, 2026-06-04): на сервере это PHP + MySQL. Таблица `events` — и есть «единая таблица событий» для всех модулей. Реестр модулей `MODULES` живёт в клиенте. На клиенте сохранён демо-режим (localStorage) для работы без сервера. См. §3 и §9.

## 3. Технологический стек и окружение

| Слой | Решение |
|---|---|
| Хостинг | Hostinger |
| URL | apps.tilley.live/robtop |
| Бэкенд | PHP |
| База данных | MySQL |
| Фронтенд | HTML/CSS/JS (на старте — статический прототип) |
| Локальная папка проекта | `/Cowork/30 Projects/RobTop/Приложение RobTop` |

**Первый релиз (2026-06-04): реализован полный бэкенд и деплой.**
- Клиент: `app/index.html` (SPA), общается с API через `fetch`; при открытии локально (file://) или без сервера — демо-режим на localStorage.
- API (PHP, PDO): `api/state.php` (GET: пользователь + желания с историей), `api/action.php` (POST: все действия + запись событий), `api/stats.php` (GET: агрегаты для дашбордов). Конфиг — `config.php`, только на сервере, не в Git.
- БД (MySQL, utf8mb4): таблицы `users`, `wishlist_items` (мягкое удаление `deleted_at`, `photo` — путь к файлу), `events` (общий аналитический журнал: type, from/to status, meta JSON, индексы), `uploaded_files` (реестр медиа, сразу с `user_id`). Схема — `schema.sql`.
- Медиа: загруженные фото хранятся на диске сервера в `uploads/users/<id>/<вид>/` через слой хранения `api/_storage.php` (драйвер local, на будущее S3/R2/Spaces); в БД — только относительный путь. Папки вне Git и переживают деплой. Endpoint — `api/upload.php`.
- Графика приложения (иконка, логотипы, иллюстрации): в `app/media/` — тоже **вне Git**, заливается по FTP, в коде путь `media/...`. Решение заказчика (2026-06-05): **репозиторий = только код**, все медиа (и графика приложения, и пользовательские файлы) под gitignore. Иконка «на домашний экран»: `media/icon/` (SVG + PNG) + `manifest.webmanifest`.
- Пользователи: **временный однопользовательский режим** — все данные принадлежат Артёму (id 1). Текущий пользователь определяется одной функцией `rt_user_id()` (константа `RT_DEFAULT_USER_ID` в `_bootstrap.php`); при добавлении входа по PIN меняется только она, структура БД уже готова (везде есть `user_id`).
- Деплой: GitHub → Hostinger через **GitHub Actions (FTPS)** при push в `main` или кнопкой «Run workflow». Боевой `config.php` не перезаписывается. Альтернатива — встроенный Git в hPanel с кнопкой Deploy. Полный гайд — `РЕЛИЗ.md`.

## 4. Роли и вход (PIN)

Два пользователя, **никакой регистрации**, всё максимально просто.

- Вход по **PIN-коду**. Какой PIN ввёл — такая роль открылась.
  - **PIN Артёма** → детская часть (полный доступ, редактирование).
  - **PIN родителя** → родительская часть (только просмотр статистики).
- Принцип ролей действует для **каждой** функции:
  - **Роль на редактирование** (Артём): добавляет/меняет данные.
  - **Роль на чтение** (Родитель): видит, что записал ребёнок, как оценивал, что выигрывал, сколько пунктов и т.д. Без редактирования.

Родительский экран = read-only статистика по всем функциям. Делается **позже**.

> Решение по прототипу v1: **PIN пока не делаем**, экран блокировки добавим позже. Сейчас прототип сразу открывает детскую функцию.

## 6. Виш-лист — подробная спецификация (текущий фокус)

Первая разрабатываемая функция. Ассоциация: **«вишлист» = вишня** → дизайн в стиле **вишнёвого дерева**.

**Данные одного желания:**
- Фото (должно красиво выглядеть на плитке/карточке).
- Название.
- Ссылка (необязательная).
- Статус: в каком разделе находится.

**Три раздела (должны визуально отличаться друг от друга):**
1. **Хочу** — активные желания.
2. **Думаю** — то, в чём не уверен (черновик/«передумал, но не убираю»).
3. **Купил** — отмечено как купленное.

**Действия:**
- Поставить галочку «куплено» → желание уходит в **Купил**.
- Кнопка **«Передумал»** → желание уходит в **Думаю**.
- Из **Думаю** можно вернуть в **Хочу**.
- Добавление нового желания: фото + название + (необязательно) ссылка.

**Дизайн:** стиль вишнёвого дерева, узоры-вишенки, на плитке/в шапке — изображение вишнёвого дерева. Современно, красиво, под телефон (вертикально).

**Решения по прототипу v1:**
- Только телефон (вертикально), 1 колонка карточек.
- Хранение — localStorage (имитация БД), плюс лог событий для будущей статистики.
- Визуальная тема: неоновый киберпанк (см. §8); каждый раздел — свой неон-акцент (Хочу розовый, Думаю фиолетовый, Купил мятный). Вишнёвая идентичность сохранена иконкой-вишенкой.

**Возможности Виш-листа (v2, 2026-06-04):**
- **Статистика действий по каждому желанию.** У желания есть история (`history`): добавлено, передумал, куплено, снова хочу, изменено. На карточке — чипы-счётчики (передумал ×N, куплено ×N, за N дней). Полная «История желания» (таймлайн + счётчики) в детальной карточке. Это прямой источник для родительского дашборда.
- **Детальная карточка.** Тап по карточке открывает лист: большое фото, заметка, ссылка, история, действия, изменить/удалить.
- **Двойное подтверждение покупки.** Кнопка «Куплено» открывает праздничный лист «Точно купили?»; подтвердить нужно двумя нажатиями (вторая кнопка «Точно? Нажми ещё раз»). Защита от случайной отметки.
- **Празднование.** При покупке — полноэкранное конфетти (отдельный canvas), мягкий звук (WebAudio-арпеджио) и вибрация; желание переезжает в «Купил» с блеском.
- **Undo.** Удаление и смену раздела можно отменить из тоста («Отменить», 6 секунд).
- **Избранное «Очень хочу» (★).** Любимые желания всплывают наверх в разделе «Хочу».
- **Заметка «почему хочу»** (необязательно) — в форме добавления/редактирования и в деталях.
- **Экран статистики желаний** (кнопка-график в шапке): всего, исполнено, всего покупок, раз передумал, любимых, среднее время до покупки, самое непостоянное желание. Превью будущего родительского дашборда.
- Миграция данных: `normalize()` дописывает `history`/`favorite`/`note` к старым записям, `version = 2`.

## 7. Копилка и Магазин (отложено намеренно)

Заказчик решил: **копилку и награды делаем позже**, в самом конце, когда будет больше функций и придумаем лучшую механику. Механика пока непонятна и обсуждается отдельно.

Сырые данные, которые уже названы (сохранить, не терять):
- Механика очков: **победа +10 пунктов**, **проигрыш −5 пунктов**. Очки начисляются за выполненные квесты/челленджи в функциях.
- Копилка — большая плитка в самом низу, «самое главное».
- Магазин — отдельная вкладка, тратишь пункты (и «лайки»):
  - Наушники — 20 000 000 пунктов + 100 лайков
  - Мяч — 100 000 пунктов
  - Деньги (1 евро) — 20 000 пунктов
  - iPad — 100 000 000 пунктов + 200 000 лайков
  - Телефон — 10 000 000 пунктов + 1 000 000 лайков

## 8. Дизайн-система

Общий стиль (с 2026-06-04): **неоновый киберпанк «печатная плата»** по референсам заказчика (скриншоты игры-змейки). Тёмный фон, неоновые дорожки платы, светящиеся элементы, плавающие орбы. Единый визуальный язык для всего приложения; каждый модуль получает свой неоновый акцент.

Палитра (неон на тёмном):

| Назначение | Цвет |
|---|---|
| Фон (база) | `#05050f` + радиальные свечения purple/cyan/magenta |
| Cyan | `#19e3ff` |
| Magenta / Pink | `#ff2bd6` / `#ff3db0` |
| Purple | `#a64bff` |
| Blue | `#3b6bff` |
| Green / Mint | `#2bf0c0` / `#38e8a0` |
| Gold | `#ffd23b` |
| Red | `#ff4d6d` |
| Текст | `#eaf2ff`, приглушённый `#93a6cf` |

Акценты разделов Виш-листа: **Хочу** — неон-розовый `#ff3db0`, **Думаю** — фиолетовый `#a64bff`, **Купил** — зелёный/мятный `#2bf0c0`. Виш-лист сохраняет «вишнёвую» идентичность через розовый акцент и иконку-вишенку (важно для Артёма).

Анимированный фон (canvas): статичные неоновые дорожки платы (генерируются процедурно) + множество плавающих светящихся орбов разных цветов и размеров (как еда и большие шары в игре), аддитивное свечение (`globalCompositeOperation = lighter`), плавный дрейф с зацикливанием по краям. Производительность: спрайты свечения кэшируются, орбов ~40–78, `devicePixelRatio` ≤ 2, учитывается `prefers-reduced-motion`. Guard: если 2D-canvas недоступен — фон тихо отключается.

Игровой HUD внизу экрана (как в референсе): крупные числа Russo One + цветные подписи. Главный экран: «приложений» / «желаний». Виш-лист: «хочу» / «желаний» / «куплено».

Шрифты (кириллица): заголовки и числа — **Russo One** (техно), интерфейс/текст — **Rubik**. Фолбэк на системные.

UX для ребёнка: крупные кнопки (тап-таргет ≥ 44px), высокий контраст, подтверждение перед удалением, простые русские подписи, приятные анимации.
