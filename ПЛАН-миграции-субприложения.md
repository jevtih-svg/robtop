# RobTop — миграция на архитектуру субприложений («магазин приложений»)
## Исполняемый промпт для агента

> Это **готовый промпт**. Скопируй его целиком в новую сессию агента, работающего в репозитории RobTop. Промпт самодостаточен: содержит роль, контекст, целевую архитектуру, контракты и пошаговый план миграции с критериями приёмки. Выполнять по фазам, **не пропуская проверки**.

---

## 0. Роль и метод

Ты — бережный картограф живой кодовой базы RobTop. Действуешь строго по `AGENTS.md`: сначала карта топологии, потом изменения; картируй обе стороны каждого моста; оставайся в рамках фазы; тестируй швы (фронт↔бэк, PHP↔MySQL); на развилках задавай high-signal вопросы; после каждого шага — самопроверка. Перед стартом прочитай целиком `КОНТЕКСТ.md` (источник правды, BRAIN) и `AGENTS.md`. После значимых изменений обновляй `КОНТЕКСТ.md`, `РЕЛИЗ.md`, журнал изменений. Всё в приложении — на русском.

## 1. Зачем эта миграция

Сейчас «оболочка + модули» — только декларация. Массив `MODULES` в `app/index.html` — это лишь метаданные плиток. Единственный рабочий модуль (Виш-лист) **вшит** в SPA: его view, CSS и ~600 строк JS лежат инлайном; `switchView()` переключает только `home`↔`wishlist`; `openModule()` хардкодит `id==="wishlist"`. Бэкенд такой же: `action.php` — один switch под Виш-лист, `state.php`/`stats.php` целиком завязаны на него. Общими являются только таблицы `events`, `uploaded_files` и колонка `module`. Добавление второго модуля сегодня требует правок монолита в 4 местах. Это не масштабируется до «магазина приложений».

## 2. Принятые решения заказчика (зафиксировано)

1. **Изоляция** — папки-модули: каждое субприложение живёт в своей папке с манифестом и собственными JS/CSS.
2. **Данные** — гибрид: универсальное хранилище по умолчанию (новому приложению **не нужна миграция БД**) + выделенная таблица для «тяжёлых» приложений (Виш-лист сохраняет свою таблицу).
3. **Магазин** — рантайм-установка: установка/включение/выключение приложений во время работы, без передеплоя.
4. **Что делаем** — полный план миграции текущей архитектуры (этот документ).

## 3. Жёсткие правила (инварианты, нарушать нельзя)

- **Аддитивность.** Существующие таблицы (`users`, `wishlist_items`, `events`, `uploaded_files`) и их данные **не меняем и не мигрируем**. Только добавляем новые таблицы.
- **Паритет Виш-листа.** После переноса Виш-лист работает 1:1 (UI, история, статистика, конфетти, undo). Любое расхождение — баг фазы.
- **Граница доверия по папкам.**
  - `app/modules/<id>/` — **родные** модули (в Git, деплоятся пайплайном). Могут иметь серверный `api.php` и `schema.sql`.
  - `app/apps/<id>/` — модули, **установленные в рантайме** (вне Git, переживают деплой). **Только статика**, никакого серверного PHP. В папке — `.htaccess`, запрещающий выполнение `*.php`.
- **Безопасность установки (OWASP).** Установка/удаление/включение модулей — только для админ-роли (PIN родителя или отдельный dev-PIN) **и** при заданном `api_token`. Ребёнок (Артём) ставить приложения не может. Бандл проходит валидацию: whitelist расширений (`js, css, json, svg, png, jpg, jpeg, webp, gif, woff2`), запрет любого `.php/.phtml/.phar/.cgi/.pl/.py/.sh`, лимиты размера, проверка схемы `module.json`, валидный уникальный `id` `^[a-z0-9_-]{2,40}$`.
- **Скоуп на сервере.** Серверные endpoints выводят `user_id` из `rt_user_id()` (никогда из тела запроса), проверяют, что `module` установлен и включён, и что роль пользователя имеет право на операцию.
- **Один пользователь — без регрессий.** Текущий режим (`rt_user_id()` → Артём id 1) сохраняется; вход по PIN добавляется позже одной точкой.
- **Секреты** — только в `config.php` на сервере / в env, не в Git.

## 4. Целевая архитектура

### 4.1 Файловая структура (целевая)

```
app/
  index.html            # тонкий хост: подключает core/*, точка монтирования #root
  core/
    ui.css              # дизайн-токены + общие компоненты (плитки, шторки, hud, toast)
    bg.js               # анимированный фон (вынесен из index.html)
    shell.js            # bootstrap: home, hud, toast/undo, шторки, роутер
    loader.js           # загрузчик модулей: open(id)/close, инъекция css/js, mount/unmount
    sdk.js              # RobTop SDK, выдаётся каждому модулю в mount(root, sdk)
  modules/              # РОДНЫЕ модули (Git). Могут иметь серверный код.
    wishlist/
      module.json       # манифест
      module.js         # RobTop.register({ id, mount, unmount })
      module.css        # стили модуля (скоуп по data-mod="wishlist")
      api.php           # (только родные) серверные операции модуля
      schema.sql        # (только родные) выделенная таблица
      icon.svg
  apps/                 # УСТАНОВЛЕННЫЕ в рантайме модули (вне Git, только статика)
    .htaccess           # deny *.php
  api/
    _bootstrap.php _db.php _storage.php   # общие (без изменений по сути)
    registry.php        # GET: установленные+включённые модули с манифестами
    data.php            # универсальный CRUD по module_data (скоуп user+module+роль)
    action.php          # → ДИСПЕТЧЕР: родной api.php модуля ИЛИ операции data.php
    upload.php          # медиа (без изменений; kind = module id)
    state.php stats.php # совместимость (Виш-лист), потом — поверх диспетчера
    store/
      install.php uninstall.php enable.php reorder.php   # магазин (только админ)
    modules.php         # админ-операции реестра (список/вкл/выкл/порядок)
  config.php            # боевой, только на сервере
  manifest.webmanifest
  media/                # графика приложения (вне Git)
  uploads/              # пользовательские файлы (вне Git)
```

### 4.2 Манифест модуля `module.json`

```json
{
  "id": "mood",
  "name": "Настроение дня",
  "version": "1.0.0",
  "color": "#ffd23b",
  "icon": "icon.svg",
  "wide": false,
  "entry": "module.js",
  "styles": "module.css",
  "roles": { "edit": ["child"], "read": ["parent"] },
  "data": { "store": "generic", "collections": ["entries"] },
  "server": false,
  "permissions": ["camera", "points"],
  "min_shell": "2.0.0"
}
```

Правила: `data.store` = `"generic"` (по умолчанию, через `data.php`/`module_data`) или `"dedicated"` (своя таблица + `api.php`, только родные). `server: true` допустим **только** для родных модулей в `app/modules/`. Для установленных (`app/apps/`) `server` принудительно `false`.

### 4.3 Контракт фронтенда модуля

Каждый `module.js` регистрируется в глобальном реестре оболочки и не трогает DOM вне переданного `root` и не вызывает `fetch` напрямую — только через `sdk`:

```js
RobTop.register({
  id: "mood",
  mount(root, sdk) {
    // отрисовать UI внутрь root; вся работа с данными/медиа/ui — через sdk
  },
  unmount() {
    // снять таймеры/слушатели, очистить root (вызывает оболочка)
  },
  onResume() {}        // опционально: при возврате на модуль
});
```

Жизненный цикл (в `loader.js`): тап по плитке → `loader.open(id)` → берёт манифест из `registry` (кэш) → один раз инъектит `module.css` (`<link>`) и `module.js` (`<script>`) → создаёт контейнер-view с `data-mod=id` → вызывает `mount(view, sdk)`. Кнопка «назад» → `unmount()` + удаление контейнера. Демо/офлайн-режим сохраняется: при `SERVER=false` `sdk.data.*` пишет в localStorage с неймспейсом по модулю (как сейчас в Виш-листе).

### 4.4 SDK (`core/sdk.js`) — поверхность, выдаваемая модулю

```
sdk.id                       // id модуля
sdk.user                     // { id, name, role }
sdk.role                     // 'child' | 'parent'
sdk.can(action)              // 'edit' | 'read' → bool по manifest.roles

sdk.data.list(coll, query?)  // → Promise<items[]>   (по module_data, скоуп модуля)
sdk.data.get(coll, id)
sdk.data.create(coll, data)  // → item
sdk.data.update(coll, id, patch)
sdk.data.move(coll, id, status)
sdk.data.favorite(coll, id, on)
sdk.data.remove(coll, id)    // мягкое удаление
sdk.data.restore(coll, id)

sdk.events.track(type, payload?)        // в общий журнал events
sdk.media.pick()                        // выбор файла → dataUrl
sdk.media.upload(fileOrDataUrl)         // → относительный путь (upload.php, kind=module)

sdk.ui.toast(msg)
sdk.ui.confirm(opts)         // → Promise<bool> (двойное подтверждение и т.п.)
sdk.ui.sheet(node)           // открыть нижнюю шторку (свайп вниз для закрытия)
sdk.ui.undo(label, fn)       // тост «Отменить» на 6 c
sdk.ui.confetti()  sdk.ui.haptics(p)  sdk.ui.chime()

sdk.points.add(n, reason)    // Копилка (пока заглушка, БД позже)
sdk.points.get()
sdk.theme.tokens             // цвета/шрифты дизайн-системы
sdk.storage.local(key)       // localStorage с неймспейсом модуля (офлайн/демо)
```

Это и есть развязка: модули изолированы, не могут сломать друг друга, общаются с системой через стабильный контракт, оболочка владеет жизненным циклом. Контракт — основа «магазина приложений».

### 4.5 Модель данных (новые таблицы, аддитивно)

```sql
-- Реестр установленных модулей (источник главного экрана)
CREATE TABLE IF NOT EXISTS modules (
  id          VARCHAR(40)  NOT NULL,                 -- slug, напр. 'wishlist'
  name        VARCHAR(80)  NOT NULL,
  version     VARCHAR(20)  NOT NULL DEFAULT '1.0.0',
  manifest    JSON         DEFAULT NULL,             -- копия module.json
  source      ENUM('native','installed') NOT NULL DEFAULT 'native',
  trusted     TINYINT(1)   NOT NULL DEFAULT 0,       -- 1 только для родных проверенных
  server      TINYINT(1)   NOT NULL DEFAULT 0,       -- есть ли серверный api.php
  enabled     TINYINT(1)   NOT NULL DEFAULT 0,
  sort_order  INT          NOT NULL DEFAULT 100,
  installed_at DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  deleted_at  DATETIME     DEFAULT NULL,
  PRIMARY KEY (id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- Универсальное хранилище данных модулей (новому приложению миграция не нужна)
CREATE TABLE IF NOT EXISTS module_data (
  id          BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  user_id     INT UNSIGNED NOT NULL,
  module      VARCHAR(40)  NOT NULL,
  collection  VARCHAR(40)  NOT NULL DEFAULT 'default',
  status      VARCHAR(24)  NOT NULL DEFAULT '',
  favorite    TINYINT(1)   NOT NULL DEFAULT 0,
  sort        INT          NOT NULL DEFAULT 0,
  data        JSON         DEFAULT NULL,             -- произвольная нагрузка модуля
  created_at  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  deleted_at  DATETIME     DEFAULT NULL,
  PRIMARY KEY (id),
  KEY idx_scope  (user_id, module, collection, deleted_at),
  KEY idx_status (user_id, module, status),
  CONSTRAINT fk_md_user FOREIGN KEY (user_id) REFERENCES users(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

`events` и `uploaded_files` уже имеют колонку `module` — переиспользуем как есть: любая операция `data.php` дублируется строкой в `events`, поэтому статистика и родительский дашборд работают единообразно для всех модулей.

### 4.6 Бэкенд

- `registry.php` (GET) → массив включённых модулей `{id,name,version,color,icon,wide,source,server,manifest}` из таблицы `modules`, отсортировано по `sort_order`.
- `data.php` (POST) → `{op, module, collection, id?, patch?, status?, query?}`. `op ∈ {list,get,create,update,move,favorite,delete,restore,undo,track}`. Сервер: `user_id = rt_user_id()`; проверка `module` установлен+включён; для записи — `roles.edit` содержит роль; пишет в `module_data`; зеркалит событие в `events` (`module`, `item_id=module_data.id`, `meta=collection`). Возврат `{ok, item|items}`.
- `action.php` → **диспетчер**: если у модуля `server=1` и существует `modules/<id>/api.php` — подключить и делегировать; иначе транслировать в операции `data.php`. Существующие типы Виш-листа продолжают работать через `modules/wishlist/api.php`.
- `store/install.php` (POST, **админ**): принимает бандл (zip или uploaded file), валидирует (см. §3), распаковывает в `app/apps/<id>/`, апсертит строку в `modules` (`source='installed', trusted=0, server=0, enabled=0`), логирует событие, возвращает манифест.
- `store/enable.php` / `reorder.php` / `uninstall.php` и `modules.php` (**админ**): вкл/выкл, порядок, удаление (файлы из `apps/<id>/` + строка; данные и события **сохраняются** для статистики, очистка — по явному запросу).

### 4.7 Магазин и рантайм-установка (поток)

Каталог доступных приложений = (a) локальная папка `app/apps/_catalog/` с бандлами и/или (b) удалённый каталог-URL (на будущее). Поток: Экран «Магазин приложений» (админ) → выбрать/загрузить бандл → `store/install.php` валидирует → распаковка в `apps/<id>/` + строка в `modules` (выключено) → админ включает → `registry.php` отдаёт его → плитка появляется на главном → `loader.open(id)` монтирует. Деплой не требуется. Папка `apps/` исключена из Git и из FTP-деплоя (как `uploads/`), поэтому установленные приложения переживают релизы.

### 4.8 Безопасность — обязательные пункты

1. `apps/.htaccess`: `Options -Indexes` + `<FilesMatch "\.(php|phtml|phar|cgi|pl|py|sh)$"> Require all denied </FilesMatch>` — установленные модули физически не могут исполнять серверный код.
2. Все `store/*` и `modules.php` за `rt_guard()` (api_token) **и** проверкой админ-роли. Дочерняя роль получает 403.
3. Валидация бандла до распаковки: размер (напр. ≤2 МБ бандл, ≤500 КБ файл), whitelist расширений, отсутствие `..` в путях, валидный `id`, схема манифеста.
4. `data.php` namespace’ит данные по `module`; для записи проверяет роль. **Известное ограничение** (задокументировать): в одном origin JS-модуль теоретически может передать чужой `module` в `data.php`. Для личного однопользовательского приложения риск низкий; план усиления — выдавать модулю при `mount` краткоживущий per-module токен и сверять его на сервере. Зафиксировать в `КОНТЕКСТ.md`.
5. Никакого выполнения PHP из `apps/`; родной серверный код — только из `modules/` (Git, ревью).

## 6. Финальная проверка (отдельный проход / субагент)

- **Паритет:** прогнать все сценарии Виш-листа против поведения до миграции.
